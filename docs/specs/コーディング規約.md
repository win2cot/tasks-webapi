## 0. 本書の位置付け

本書は **Java / Spring 実装レベルで遵守すべき規約** を定める。アーキテクチャ・API・DB・セキュリティ等の**設計レベル**の規約は別冊 [`設計規約.md`](./設計規約.md) を参照。

本書は scaffold(`xyz.dgz48.tasks.webapi.*`)の実装パターンと既存ツールチェイン(Spotless / NullAway / JSpecify / Error Prone / Lombok / JaCoCo / Testcontainers / Spring Modulith)に整合する。新規ライブラリの導入は ADR を要する。

## 1. フォーマット

### 1.1 自動整形

- フォーマッタは **Google Java Format v1.24.0**(Spotless 経由)を唯一の正とする。
- **手動整形を禁止**。インデント・改行位置・import の並びの議論は規約上行わない。
- コミット前に必ず `./gradlew :webapi:spotlessApply` を実行する。CI は `:webapi:spotlessCheck` で検証する。

```bash
# 整形(コミット前必須)
./gradlew :webapi:spotlessApply

# 検証のみ
./gradlew :webapi:spotlessCheck
```

### 1.2 文字コード・改行

- ファイル文字コードは UTF-8(BOM なし)。
- 改行は LF。`.gitattributes` で強制する。
- ファイル末尾に改行 1 個。

### 1.3 行幅

- Google Java Format に従う(100 桁)。手動で長さを調整しない。

### 1.4 コンパイル警告ゼロポリシー

- `./gradlew :webapi:compileJava` の出力に **警告を出してはならない**。CI はゼロ警告を強制する。
- Error Prone の以下のチェックは `error` レベルに昇格しており、違反するとコンパイルが失敗する。

| チェック | 内容 | 詳細 |
|---|---|---|
| `NullAway` | null 安全違反 | §2 参照 |
| `BadImport` | ネストクラス・静的メンバーを短い名前で直接インポートすることを禁止 | §14 参照 |
| `JavaTimeDefaultTimeZone` | タイムゾーン未指定の `LocalDateTime.now()` / `Instant.now()` 等を禁止 | §12 参照 |

## 2. null 安全

### 2.1 全パッケージ `@NullMarked`

- すべての Java パッケージに `package-info.java` を置き、JSpecify の `@NullMarked` を付与する。
- 既存例(踏襲):

```java
// xyz/dgz48/tasks/webapi/task/package-info.java
@NullMarked
package xyz.dgz48.tasks.webapi.task;

import org.jspecify.annotations.NullMarked;
```

- 新規パッケージ追加時は `package-info.java` の作成を**忘れない**。Code Review で必ず確認する。

### 2.2 `@Nullable` の使い方

- フィールド・パラメータ・戻り値は**デフォルトで非 null**(NullMarked による)。`null` を許容する場合のみ `org.jspecify.annotations.@Nullable` を**明示**する。
- 「呼び出し側が null を渡すかもしれない」という防御的 `@Nullable` は付けない(NullAway に頼る)。

### 2.3 JPA エンティティの遅延初期化

- JPA は protected な no-arg コンストラクタからフィールドを反射で埋めるため、コンパイル時には全フィールド未初期化と判定される。対応として、エンティティクラスに **`@SuppressWarnings("NullAway.Init")`** を**クラス単位**で付与する。
- 付与時には「JPA requires no-args constructor; fields initialized via JPA」など**意図コメント**を添える(既存 `Task.java` / `User.java` の形を踏襲)。

### 2.4 テストの null 安全

- テストパッケージは NullAway を**無効化済み**(`compileTestJava` で `disable('NullAway')`)。とはいえ、本番コードの API の null 性は `@Nullable` を信頼してテストを書く。

## 3. Lombok 使用ルール

許容するアノテーションは以下のみ。**それ以外の Lombok アノテーション(例: `@Data` / 無条件の `@AllArgsConstructor` / `@Builder` を JPA Entity に適用 等)は禁止**。

| アノテーション | 用途 | 備考 |
|---|---|---|
| `@Getter` | フィールドへの getter | 設定不変オブジェクトを表現するのに使う |
| `@RequiredArgsConstructor` | DI 用コンストラクタ | Service / Controller でのコンストラクタインジェクション |
| `@NoArgsConstructor(access = AccessLevel.PROTECTED)` | JPA Entity の no-arg コンストラクタ | アクセス修飾子は必ず PROTECTED |
| `@EqualsAndHashCode(onlyExplicitlyIncluded = true)` + `@EqualsAndHashCode.Include` | 特定フィールドのみで等価判定 | Lombok 1.18 以降は `of = "..."` ではなくこの新スタイルを使う(下記サンプル参照)。JPA Entity への適用は避け、`TasksPrincipal` 等の値オブジェクトに限定 |
| `@Slf4j` | logger 自動生成 | ロガー変数は `log` で統一 |

### 3.1 `@EqualsAndHashCode` の推奨形

Lombok 1.18.x 以降、`of` / `exclude` パラメータの利用は非推奨となり、`onlyExplicitlyIncluded` と `@EqualsAndHashCode.Include` の組み合わせが推奨形である。`TasksPrincipal` のような値オブジェクトでも以下の形に揃える。

```java
@EqualsAndHashCode(onlyExplicitlyIncluded = true)
public class TasksPrincipal implements Principal {

  @EqualsAndHashCode.Include
  private final String sub;

  // ... 他フィールドは等価判定に含めない
}
```

- 旧来の `@EqualsAndHashCode(of = "sub")` 形は **書かない**(Lombok の警告対象)。
- 現行 scaffold(`xyz.dgz48.tasks.webapi.security.TasksPrincipal` の `@EqualsAndHashCode(of = "sub")`)は本規約と整合しないため、Sprint 0 整合タスク(Issue #121 配下)で新スタイルへ追従する。

### 3.2 禁止理由(主要なもの)

- `@Data` … `@ToString` による無限再帰、`@EqualsAndHashCode` による JPA 識別子衝突、setter 自動生成によるミューテーション漏れ、を一度に引き起こす。
- 無条件 `@AllArgsConstructor` … JPA Entity に付けると不変条件の維持が破綻する。明示的に public コンストラクタを書く方針(§4 参照)。
- `@Builder`(JPA Entity への適用) … バリデーションを通らないインスタンス生成経路を作る。DTO record では使ってよい。

## 4. JPA エンティティ

### 4.0 ドメイン POJO と JPA Entity の命名規約

設計規約 §1.2.3 が定めるとおり、`domain` 層は POJO のみ・JPA アノテーション禁止であり、JPA Entity は `adapter.persistence` 層に置く。**両者の単純クラス名が衝突するとインポートが破綻し UseCase 層から両方を扱えなくなる**ため、命名規約として以下を強制する。

| 層 | 単純クラス名 | パッケージ例 |
|---|---|---|
| `domain` | `Task`(POJO、@Entity 不可) | `xyz.dgz48.tasks.webapi.task.domain.Task` |
| `adapter.persistence` | **`TaskJpaEntity`**(`@Entity` 必須) | `xyz.dgz48.tasks.webapi.task.adapter.persistence.TaskJpaEntity` |

- **JPA Entity の単純名は必ず `~JpaEntity` で終える**。`Task` のような名前で `@Entity` を付けない。
- domain ↔ persistence の変換は `adapter.persistence` 内の Mapper(または静的ファクトリ)が担う。MapStruct 採否は ADR で決定する(現状未採用)。
- 現行 scaffold(`xyz.dgz48.tasks.webapi.task.Task` に `@Entity`)は feature-by-package 単層構造の暫定形であり、Sprint 0 の整合タスク(Issue #121 配下)で `task.adapter.persistence.TaskJpaEntity` へリネーム・移設する。
- 関連エンティティ(`User` / `Tenant` / `Stakeholder` 等)も同様: `UserJpaEntity` / `TenantJpaEntity` / `TaskStakeholderJpaEntity` のように `~JpaEntity` サフィックス。

### 4.1 正規パターン

新規 JPA Entity は以下の形を踏襲する。

```java
// xyz/dgz48/tasks/webapi/task/adapter/persistence/TaskJpaEntity.java
// (domain 層の Task(POJO)と区別するため JpaEntity サフィックスを必須化。§4.0 参照)
@Entity
@Table(name = "tasks")
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
@SuppressWarnings("NullAway.Init") // JPA requires no-args constructor; fields initialized via JPA
public class TaskJpaEntity {

  @Id
  @GeneratedValue(strategy = GenerationType.IDENTITY)
  private Long id;

  @Enumerated(EnumType.STRING)
  @Column(nullable = false)
  private TaskStatus status;

  @Column(name = "owner_id", nullable = false)
  private Long ownerId;

  @Column(nullable = false, length = 255)
  private String title;

  @Nullable
  @Column(columnDefinition = "LONGTEXT")
  private String body;

  @Column(name = "tenant_id", nullable = false)
  private Long tenantId;

  // ----- 監査列(設計規約 §3.5) -----
  // 業務テーブルは created_at / updated_at / created_by / updated_by を必須とする。
  // 自動投入は ADR-0009 で Spring Data JPA Auditing を採用と確定済み。
  // @CreatedDate / @LastModifiedDate は ClockConfig の DateTimeProvider が自動投入する。
  // @CreatedBy / @LastModifiedBy は JpaAuditorAware (security/adapter/persistence) が自動投入する。
  @Column(name = "created_at", nullable = false, updatable = false)
  private OffsetDateTime createdAt;

  @Column(name = "updated_at", nullable = false)
  private OffsetDateTime updatedAt;

  @Column(name = "created_by", nullable = false, updatable = false)
  private Long createdBy;

  @Column(name = "updated_by", nullable = false)
  private Long updatedBy;

  public TaskJpaEntity(
      Long tenantId, TaskStatus status, Long ownerId, String title, @Nullable String body) {
    this.tenantId = tenantId;
    this.status = status;
    this.ownerId = ownerId;
    this.title = title;
    this.body = body;
    // createdAt / updatedAt / createdBy / updatedBy はコンストラクタでは受け取らず、
    // Spring JPA Auditing (@CreatedDate / @LastModifiedDate) が永続化時に自動投入する(ADR-0009)。
  }
}
```

> 設計規約 §3.3 に従い、業務テーブル系の Entity には `tenant_id` 列を**必須**で持たせる(`users` テーブルのみ既存・例外)。サンプル上も `tenantId` フィールドを明示する。設計規約 §3.5 の監査列 4 つも同様に**必須**であり、サンプル末尾にカラム宣言を含めている。`@CreatedDate` / `@LastModifiedDate` 等は ADR-0009 で Spring Data JPA Auditing を採用と確定済み。`@EnableJpaAuditing` + `ClockConfig`（`shared/infra`）の `DateTimeProvider` Bean が Asia/Tokyo の現在時刻を自動投入する。

### 4.2 ルール

- **setter は作らない**。属性を変えるドメイン操作は**意味のある名前のメソッド**として定義する(例: `task.markAsDone()` / `task.changeOwner(newOwnerId)`)。
- public コンストラクタは**業務的に不変条件を満たす最小セット**を引数に取る。「ID と監査列以外のすべて」を要求するのが目安。
- `@Enumerated(EnumType.STRING)` を必須化(`ORDINAL` は禁止 — DB 値の意味が崩れる)。
- 列名は `@Column(name = "...")` で**snake_case を明示**する。Hibernate の自動変換に依存しない。
- `equals` / `hashCode` は基底クラス(Object)既定を使うか、識別子確定後の比較が必要な場面でのみ実装する。**`@EqualsAndHashCode` を Entity 全体に付けない**(未保存インスタンス同士の比較で問題化する)。

### 4.3 関連

- 関連は**遅延ロード(`FetchType.LAZY`)を既定**とする。N+1 が問題化したら `JOIN FETCH` / `@EntityGraph` で個別解決する。
- 親子の cascade は実テーブル設計と一致させる(設計規約 §3.4)。

## 5. 型選択(record / class)

クリーンアーキ各層で新規型を起こすとき、`record` / `class` / `interface` / `enum` のどれで書くかを横断ルールとして定める。本節は層 × 部品の使い分けを俯瞰する位置付けで、個別ルールの詳細(JPA Entity = §4、例外 = §6、REST DTO = §5.3)は各節を SSoT として参照する。

> 設計規約 §1.2.3 が「`domain` 層は POJO のみ・Spring 非依存・JPA アノテーション禁止」と定めるが、ここでの POJO は record / class の双方を含む(JPA アノテーションが付かない限り、record も POJO として扱う)。

### 5.1 層 × 部品 マトリクス

| 層 | 部品 | 推奨型 | 根拠 |
|---|---|---|---|
| `domain` | 業務エンティティ(`Task` / `User` 等) | **class**(setter なし、behavior method を持つ — §4.2 既存ルール継承) | DDD ドメインモデル。状態変化を意味あるメソッド名で表現する(`task.markAsDone()` 等)。 |
| `domain` | 値オブジェクト(`TenantId` / `EmailAddress` / `Money` 等) | **record** | 不変、`equals` / `hashCode` がフィールド全件由来で十分。 |
| `domain` | ドメインサービス(`TaskAuthorizationDomainService` 等) | **class**(stateless、Spring Bean ではない) | 振る舞いの集合。設計規約 §5.2(認可 SSOT)参照。 |
| `domain` | リポジトリ I/F | **interface** | 実装は `adapter.persistence`。 |
| `usecase` | UseCase 実装 | **class**(クラスは `@Service`、メソッドに `@Transactional` — §11.3) | 依存注入 + トランザクション境界(§11)。 |
| `usecase` | Input / Output Port DTO | **record** | 層境界の immutable データ運搬。`adapter.web.dto` の REST DTO とは別物として `usecase.port.*` 配下に置く。 |
| `usecase` | 外部 Port I/F(`MailSender` 等) | **interface** | 実装は `adapter.external`。 |
| `adapter.web` | Controller | **class**(`@RestController`) | Spring Web 規約。 |
| `adapter.web` | Request / Response DTO | **record**(§5.3 で詳細) | Bean Validation 直付け、Jackson 対応。 |
| `adapter.persistence` | JPA Entity | **class**(`~JpaEntity` サフィックス、§4 既存ルール) | Hibernate `@Entity` は class 必須(JPA 仕様)。§4.0 / §4.1 参照。 |
| `adapter.persistence` | `@Embeddable` 値オブジェクト | **record**(Hibernate 6+)/ 例外的に **class** | 短い不変値は record。複数フィールドで挙動を持たせたい場合のみ class。 |
| `adapter.persistence` | Spring Data Repository | **interface**(`extends JpaRepository`) | 派生クエリ自動生成(§8.1)。 |
| `adapter.external` | 外部 API Client | **class** | DI、リトライ・タイムアウト等の状態を持つ。 |
| `adapter.external` | 外部 API Request / Response DTO | **record** | Jackson 対応、immutable。 |
| `infra` | `@Configuration` クラス | **class** | Spring `@Configuration` はサブクラス化される(CGLIB プロキシ)ため record 不可。 |
| `infra` | `@ConfigurationProperties` | **record** 推奨 | immutable 設定。Spring Boot 3+ では record の場合コンストラクタバインディングが自動適用される(`@ConstructorBinding` 明示は不要)。 |
| `shared` | 例外 | **class**(`extends RuntimeException`) | 継承必須(§6)。 |
| `shared` | 共通 enum(`ErrorCode` / `Visibility` 等) | **enum** | 値の有限集合、`@Enumerated(EnumType.STRING)`(§4.2)で永続化。 |
| `shared` | 汎用値オブジェクト(`ErrorResponse` 等) | **record** | [ADR-0011](../adr/0011-error-response-record.md) で確定済み。GraalVM Native Image([ADR-0008](../adr/0008-graalvm-native-image.md))との相性も良い。 |

### 5.2 判断軸

新規型がマトリクスの行に直接当てはまらない場合は、以下 4 条件で判定する。**4 つすべてに該当**するなら `record`、それ以外は `class`。

1. **値比較で十分か** — `equals` / `hashCode` がフィールド全件由来で意味を持つ(ID 同一性ではなく値同一性で比較したい)。
2. **状態変化が外から見えないか** — 後で変えるなら新インスタンスを返す(immutable)。状態遷移を意味付ける behavior method(`order.cancel()` 等)が要るなら class。
3. **JPA `@Entity` が不要か** — Hibernate `@Entity` は class 必須(JPA 仕様、final 不可・no-args 必須)。Entity なら自動的に class。
4. **Spring サブクラス化(CGLIB プロキシ)が不要か** — `@Configuration` / `@Transactional` クラス / `@Async` クラス等は Spring がサブクラスを生成するため record(final)では動かない。

判断軸の優先順位は固定ではないが、3 と 4 はフレームワーク制約として絶対(該当時は record 不可)。1 と 2 は設計判断。

### 5.3 REST 入出力 DTO

- REST 入出力 DTO は **Java `record`** を用い、`adapter.web.dto` 配下に置く(§5.1 マトリクス 9 行目の具体化)。
- record に Bean Validation アノテーションを直接付与する。

```java
public record TaskCreateRequest(
    @NotBlank @Size(max = 255) String title,
    // body はアプリ層上限として 64 KiB(65535)とする。DDL 側は LONGTEXT(物理上限 4 GiB)で
    // データベース制約を最小化し、API/UI で扱える文字量はこの DTO の @Size で律する。
    // 値変更時は OpenAPI / 設計書 / DDL の 3 点を同時更新する(設計規約 §2.5)。
    @Nullable @Size(max = 65_535) String body,
    @NotNull Visibility visibility) {}
```

- record は不変、Domain との変換は Mapper(または静的ファクトリ)で行う。
- 内部のドメインオブジェクトと record を**直接相互参照させない**(`adapter` 層に閉じ込める)。
- `usecase` 層の Input / Output Port DTO は `usecase.port.*` 配下の別 record として定義し、web DTO ↔ port DTO の変換は Controller(`adapter.web`)で行う(port が web 詳細を漏らさないため)。
- **PATCH 部分更新 DTO** はフィールド型を `org.openapitools.jackson.nullable.JsonNullable<T>` で包み、`undefined` / `null` / 値 の 3 状態を区別する([ADR-0014](../adr/0014-jsonnullable-for-patch-dto.md))。`A-16 PATCH /tasks/{id}/status` / `A-17 PATCH /tasks/{id}/visibility` のような単項目 `required` PATCH は対象外。

## 6. 例外処理

### 6.1 例外階層

- アプリの業務例外は `xyz.dgz48.tasks.webapi.shared.exception.DomainException`(`RuntimeException` のサブクラス)を基底とする。
- 認可違反は HTTP マッピング先ごとに**別クラス**を作る。設計規約 §2.3 のマッピングに従う。

| 例外クラス | HTTP | `ErrorCode` | 用途 |
|---|---|---|---|
| `TaskNotViewableException` | 404 | `E_NOT_FOUND` | GET で参照不可(NIST AC-4) |
| `TaskOwnershipException` | 403 | `E_FORBIDDEN` | 編集系で権限不足 |
| `TenantBoundaryViolationException` | 403 | `E_FORBIDDEN` | テナント越境アクセス |
| `ValidationFailedException` | 400 | `E_VALIDATION` | サーバ側で検出した業務制約違反(Bean Validation で表現できないもの) |
| `ConflictException` | 409 | `E_CONFLICT` | 一意制約違反等 |

### 6.2 マッピング

- `@RestControllerAdvice` で全例外を **§2.4 のエラー応答スキーマ**(`timestamp` / `status` / `error` / `code` / `message` / `path` の 6 フィールド)に変換する。**戻り型は独自 `record ErrorResponse(...)`**([ADR-0011](../adr/0011-error-response-record.md))を採用し、実 class は `xyz.dgz48.tasks.webapi.shared.web.ErrorResponse`(record)/ `xyz.dgz48.tasks.webapi.shared.web.ErrorCode`(enum)/ `xyz.dgz48.tasks.webapi.shared.web.GlobalExceptionHandler`(`@RestControllerAdvice` + `extends ResponseEntityExceptionHandler`)に配置する。`ResponseEntityExceptionHandler` の詰め替え hook(本プロジェクトが採用する Spring Boot 4.0.x / Spring Framework 7.x で公開される `handleExceptionInternal(...)` または `createResponseEntity(...)` のいずれか深い側、ADR-0011 §6 参照)を override して Spring built-in 例外(`MethodArgumentNotValidException` 等)が返す `ProblemDetail` を `ErrorResponse` へ詰め替える。Controller 内で `try-catch` して `ResponseEntity` を組み立てない。
- 例外クラス名と `code` は 1:1 で対応させる(`TaskOwnershipException` → `code: E_FORBIDDEN`)。

### 6.3 例外を投げる場所

設計規約 §5.2 の「業務認可は `domain` 層の `AuthorizationDomainService` が**真偽**を返す純粋関数、`usecase` 層がそれを受けて例外を投げる」方針に揃え、層ごとに以下のとおり分離する。

- **業務不変条件違反は `domain` 層で例外を投げる**。`domain` のエンティティ・値オブジェクト・ドメインサービスが**自分自身の不変条件**を守るために投げる例外。
  - 例: タスクのステータスを `DONE` から `IN_PROGRESS` へ戻そうとする(状態遷移違反) → `TaskStatusTransitionException`
  - 例: 関係者リストに既に存在するユーザーを再追加する(重複違反) → `DuplicateStakeholderException`
- **認可違反は `usecase` 層で例外を投げる**。`domain` 層の `AuthorizationDomainService`(例: `TaskAuthorizationDomainService`)が「**この主体はこの操作を許可されているか**」を真偽で返し、`usecase` がその結果を受けて §2.3 のマッピングに従う例外を throw する。
  - 例: 自分以外の所有者のタスクを編集しようとした → `TaskAuthorizationDomainService#canEdit(...)` が `false` → `usecase` が `TaskOwnershipException`(403)を throw
  - 例: 他人の PRIVATE タスクを参照しようとした → `TaskAuthorizationDomainService#canView(...)` が `false` → `usecase` が `TaskNotViewableException`(404)を throw
- **入力検証失敗は Bean Validation に任せる**(`MethodArgumentNotValidException` → 400)。`usecase` / `domain` で再検証しない。

## 7. ログ

出力スキーマ・相関キー・event タクソノミ・環境別レベルの正本は [ログ設計](./ログ設計.md)(Issue #451 / ADR-0019)。本節はコードの書き方を定める。

- `org.slf4j.Logger` を `@Slf4j` で取得し、変数名 `log` で統一。
- `System.out.println` / `e.printStackTrace()` / `Throwable#getStackTrace` を直接出力する書き方は禁止。
- **MDC 項目**: `requestId`(全リクエスト)/ `userId`(認証後)/ `tenantId`(TenantContext 構築時のみ)/ `apiScope`(プラットフォーム API のみ)/ `batchId`(バッチのみ)。フィルタで自動セットし、業務コードで直接操作しない(設計規約 §4.2、ログ設計 §3.1)。
- フォーマットは SLF4J プレースホルダ(`log.info("Task {} updated by {}", taskId, userId)`)を使う。文字列連結禁止。
- 構造化フィールドの動的付与は SLF4J fluent API(`log.atWarn().addKeyValue("event", "TENANT_CROSSED").log(...)`)を使う。値は **string / number のみ、ネスト禁止**(ログ設計 §3.2)。`event` の値はログ設計 §5 のタクソノミに従う。
- 例外を再ラップする場合、`cause` を保持する(`new XException("msg", e)`)。
- **PII(氏名・メール・電話番号等)はログに出力しない**。マスクして出すことも不可(マスクは漏洩経路とのいたちごっこになる)。参照は内部 ID(`userId` / `taskId` 等)のみ。
- PII の**非意図的な漏洩経路**にも対策する:
  - **例外メッセージ**: 入力値を含む既知の例外(MySQL `Duplicate entry '...'`、Bean Validation の rejected value 等)は、そのままログに出さず sanitize してから記録する(`@RestControllerAdvice` での変換と同じ場所で対処)。
  - **toString 経由**: エンティティ / DTO / record をプレースホルダに直接渡さない(record は全 component が toString に出る)。渡すのは ID。
  - **リクエスト body / query 文字列の生ログ禁止**(検索キーワード等に PII が入り得る)。
  - **フレームワーク設定**: Hibernate のパラメータバインドログ(TRACE)等、値を吐く設定を prod 系環境で有効化しない。

### 7.1 監査ログ detail — 手組み JSON 禁止

`AuditLogPort.record` の `detail` 引数には **`Map` または小型 record 等の構造体をそのまま渡す**。usecase 層での JSON 文字列手組み(`StringBuilder` による `"{\"key\":...}"` 等)は禁止する。シリアライズは `AuditLogPersistenceAdapter` 内の `ObjectMapper` に一元化する(設計規約 §1.2.3 遵守)。

- **diff 系**(`TASK_UPDATED` / `TENANT_UPDATED`): `List<AuditFieldChange>` に詰め替えて渡す。`AuditFieldChange` は `@JsonProperty("old")` / `@JsonProperty("new")` で JSON キー名を確定する。
- **シンプル系**: `Map.of("taskId", id)` など。`Map.of` は `null` 値を弾くため、`null` 値を含む場合は `LinkedHashMap` を使う。
- **空 detail**: `Map.of()` を渡す(または `null`、どちらも `"{}"` として格納される)。

**理由**: 手組み JSON は `LocalDate` 等の非プリミティブ型を引用符なし出力する欠陥を起こしやすく(Issue #702 / PR #703)、`ObjectMapper` に集約することで型安全性・保守性・テスタビリティが向上する。

| 種類 | スタイル | 例 |
|---|---|---|
| パッケージ | 小文字(単語連結) | `xyz.dgz48.tasks.webapi.task` |
| クラス | UpperCamelCase | `TaskService`, `TaskRepository` |
| JPA Entity クラス | **`~JpaEntity` で終わる**(domain POJO との衝突回避、§4.0 参照) | `TaskJpaEntity`, `UserJpaEntity` |
| 例外クラス | `*Exception` で終わる | `TaskNotViewableException` |
| インタフェース | 接頭辞 `I` を付けない | `TaskRepository`(✕ `ITaskRepository`) |
| メソッド・変数 | lowerCamelCase | `findByTenantIdAndId` |
| 定数 | UPPER_SNAKE_CASE | `DEFAULT_PAGE_SIZE` |
| Java enum 値 | UPPER_SNAKE_CASE | `NOT_STARTED`, `IN_PROGRESS` |

| Flyway ファイル | `V{semver}_{seq}__{snake_case}.sql` | `V1.3.0_02__add_task_stakeholders.sql` |
| feature パッケージ | 単数形・名詞 | `task`(✕ `tasks`) |

### 8.1 Repository メソッド命名

- Spring Data JPA の派生クエリは `findBy<Property>...` の規約に従う。
- **SELECT 系派生クエリ**(`findBy...` / `existsBy...` / `countBy...`)は、Hibernate Filter が `tenant_id` を自動付与する([ADR-0010](../adr/0010-tenant-isolation-hibernate-filter.md) / 設計規約 §3.3)ため、`findById(Long id)` / `findByStatus(...)` 等の**単純命名で構わない**。`findByTenantIdAndId` のような明示絞り込みは不要(書いても害はないが冗長)。
- **`@Modifying` UPDATE / DELETE および native query**(`@Query(nativeQuery = true)`)は Hibernate Filter が適用されないため、引き続き `tenant_id` を**明示的に絞り込む**(設計規約 §3.3):
  - 派生クエリ: `deleteByTenantIdAndId(Long tenantId, Long id)` 等。
  - JPQL: `@Modifying @Query("UPDATE ... WHERE tenant_id = :tenantId AND id = :id ...")`。
  - native query: `WHERE tenant_id = :tenantId AND ...` を SQL に明示。
- **クエリ実装の優先順位**(設計規約 §3.3): 次の順で選択し、上位で表現できる場合は下位を使わない:
  1. **メソッド名導出クエリ**(`findBy...` / `countBy...` / `existsBy...` 等)— 起動時クエリ検証あり・文字列保守不要・Hibernate Filter が効く
  2. **明示クエリ(`@Query`, JPQL)**— 導出で表現できない場合のみ(スカラ射影 / GROUP BY / 単一文 bulk DML 等)
  3. **native query**— (1) モジュール境界を越えて他モジュールのテーブルを参照する集計、(2) DB 固有関数・JPQL の表現力を超える複雑クエリ ―― に限り、`// native 理由: …` をコードに必ず付与する
- **クエリ射影の既定方針**: 既定はエンティティ取得 + adapter でのマッピング。専用射影(scalar / DTO)は (a) エンティティが持たない他テーブル列が必要、(b) 高頻度・横長テーブルで over-fetch が実測で効く、のいずれかに限る。同一テーブルの列 subset 違いでメソッドを増やさない。

## 11. トランザクション

`@Transactional` は **`usecase` 層のメソッドに付与** する。Controller / Repository / Domain には付けない(設計規約 §1.2.3)。クラスレベル付与は禁止し、必ずメソッドレベルで明示する(§11.3)。

### 11.1 `readOnly` の明示

`@Transactional` を付与する場合、`readOnly` 属性を **常に明示** する(`true` / `false` どちらでも、暗黙の Spring デフォルトに依存しない)。

```java
// OK: 読み取り専用 usecase
@Transactional(readOnly = true)
public TaskView getTask(Long taskId) { ... }

// OK: 書き込み usecase(false を明示)
@Transactional(readOnly = false)
public Task createTask(TaskCreateCommand cmd) { ... }

// NG: readOnly 省略(Spring デフォルト = false に依存)
@Transactional
public Task createTask(TaskCreateCommand cmd) { ... }
```

**理由**: Spring の素のデフォルトは `readOnly = false` だが、規約レベルでは「default は何か」の議論を排除し、**1 行見ればその usecase が読み取りか書き込みか判別できる** ことをレビュー効率の観点で優先する。

### 11.2 Propagation と例外ケース

**read-then-write は単一 usecase 内で完結** する(read 用 usecase の中から別の write 用 usecase を呼び出して 1 トランザクションにしようとしない、別 `@Transactional` の service / usecase を呼び出さない)。ネストした transaction の振る舞いを暗黙の Propagation に依存させないため。同一 usecase 内で read と write を順に行うのは通常の書き込み usecase であり、`@Transactional(readOnly = false)` を付与すれば OK。

ネストや伝播属性(`Propagation.REQUIRES_NEW` 等)を使う場合は、**メソッドの Javadoc に理由を明示** する。

```java
/**
 * 監査ログは元 transaction の rollback と独立して必ず記録するため
 * REQUIRES_NEW で別 transaction にする。
 */
@Transactional(propagation = Propagation.REQUIRES_NEW, readOnly = false)
public void recordAuditEntry(AuditEntry entry) { ... }
```

**Exception rollback rules**:

- デフォルトでは **unchecked exception(`RuntimeException` / `Error`)のみ rollback トリガー**(Spring の挙動)。
- `rollbackFor = ...` / `noRollbackFor = ...` を使う場合は **Javadoc で理由を明示** する。

```java
/**
 * Mail 送信失敗(checked IOException)時もタスク作成は rollback したいため
 * rollbackFor を明示。
 */
@Transactional(readOnly = false, rollbackFor = { IOException.class })
public Task createTaskAndNotify(TaskCreateCommand cmd) throws IOException { ... }
```

**JPA flush の挙動**: `readOnly = true` を付与すると、Spring の `HibernateJpaDialect#beginTransaction` が Hibernate session の flush mode を **`FlushMode.MANUAL`** に切り替え、commit 時の自動 flush(dirty checking 経由の暗黙 UPDATE)を抑止する(これが Spring の "read-only optimization" の主機構)。ただし以下の経路では `readOnly = true` でも DB 書き込みが走る:

- `entityManager.flush()` の **明示呼び出し**
- JPQL `UPDATE` / `DELETE`(`query.executeUpdate()`)
- Native SQL の `UPDATE` / `DELETE` / `INSERT`
- 2nd-level cache の更新(`@Cache` 設定の Entity)

つまり「不注意による書き込み」は防げるが「**意図的な書き込み**」までは bypass されないため、`domain` 層オブジェクトの mutation や上記 4 経路の書き込み API は read-only usecase 内で **書かない**。書き込みが必要なら `@Transactional(readOnly = false)` の usecase に分離する(§11.2 冒頭の「read-then-write は単一 usecase 内で完結」と整合)。

### 11.3 クラスレベル付与の禁止

`@Transactional` を **クラスレベルに付与しない**。メソッドレベルでのみ付与する。

```java
// NG: クラスレベル付与
@Service
@Transactional  // ← 禁止
public class TaskUseCase { ... }

// OK: 各メソッドに明示
@Service
public class TaskUseCase {
  @Transactional(readOnly = true)
  public TaskView getTask(Long id) { ... }

  @Transactional(readOnly = false)
  public Task createTask(TaskCreateCommand cmd) { ... }
}
```

**理由**: クラスレベル付与だと「このクラスの全 public method が `@Transactional` 適用」になり、メソッド単位の `readOnly` 値が一望できなくなる。レビュー・grep いずれの面でも不便。

## 12. 日時・タイムゾーン

- DB 列型は `DATETIME`(MySQL)。Java では原則 **`java.time.OffsetDateTime`** または **`java.time.Instant`** を使う。
- 「日付」の意味で十分なら `LocalDate`。`LocalDateTime` は JDBC マッピング上必要な箇所だけに留める(タイムゾーン情報を失うため)。
- CI 環境は `TZ=Asia/Tokyo` 固定(`cicd.yml`)。テストで `LocalDateTime.now()` を直接使うとマシン依存で失敗するため、`Clock` を DI して固定化する。
- **タイムゾーン未指定の `LocalDateTime.now()` は禁止**(`JavaTimeDefaultTimeZone` Error Prone チェックがビルドエラーにする)。`ZoneId.of("Asia/Tokyo")` を**常に明示**する。JPA Entity のタイムスタンプは ADR-0009 で Spring Data JPA Auditing に確定済み。`@PrePersist`/`@PreUpdate` は使わず、`@CreatedDate`/`@LastModifiedDate` + `ClockConfig` パターンを用いる:

```java
@EntityListeners(AuditingEntityListener.class)
public class SomeJpaEntity {

  @CreatedDate
  @Column(name = "created_at", nullable = false, updatable = false)
  private OffsetDateTime createdAt;

  @LastModifiedDate
  @Column(name = "updated_at", nullable = false)
  private OffsetDateTime updatedAt;
}
```

`ClockConfig`（`shared/infra`）が `Clock.system(ZoneId.of("Asia/Tokyo"))` と `clockDateTimeProvider` Bean を定義。テスト時は `FixedClockConfiguration` で差し替え可能（`@ConditionalOnMissingBean` で制御）。

### 12.1 JDBC ↔ Java 日時型のマッピング設定(重要)

MySQL の `DATETIME` 列はタイムゾーン情報を持たないため、何も設定せず `OffsetDateTime` を読み書きすると、JDBC ドライバが JVM デフォルト TZ で暗黙変換し**意図しないオフセット差**が発生する。これを防ぐため、Spring Boot の JDBC URL に以下を明示する。

```yaml
# application.yml(本番・テスト共通)
spring:
  datasource:
    url: jdbc:mysql://${DB_HOST}:3306/${DB_NAME}?connectionTimeZone=SERVER&forceConnectionTimeZoneToSession=true
```

- `connectionTimeZone=SERVER`: JDBC 接続時に MySQL サーバ TZ をクライアント側 TZ として採用(MySQL 接続側で `Asia/Tokyo` を返す前提)。
- `forceConnectionTimeZoneToSession=true`: セッション TZ を上記接続 TZ に強制し、`NOW()` 等の DB 関数も同じ TZ で評価。
- MySQL Connector/J 8.0.23 以降のオプション名で記述。古い `serverTimezone` パラメータは使わない。
- RDS のパラメータグループで `time_zone = Asia/Tokyo` を設定し、JVM・CI・DB 全てが Asia/Tokyo に揃っていることを前提とする。
- Testcontainers ベースのテストでも本設定を再現する(`MySQLContainer#withUrlParam` 等で同じパラメータを付与)。
- **API レスポンス上のタイムスタンプは JST(オフセット `+09:00`)で返す**(設計規約 §2.4)。Jackson は `OffsetDateTime` を `ZoneOffset.of("+09:00")` または `ZoneId.of("Asia/Tokyo")` に揃えてシリアライズし、`Z` サフィックス(UTC 表記)は出さない。`application.yml` に `spring.jackson.time-zone: Asia/Tokyo` および `spring.jackson.serialization.write-dates-as-timestamps: false` を Sprint 0 で設定する(Issue #142 / #121 配下)。
- **JVM のデフォルト TZ は `Asia/Tokyo` 固定**: ECS Task 環境変数 `TZ=Asia/Tokyo` と JVM オプション `-Duser.timezone=Asia/Tokyo` を併用して二重強制する(`docs/architecture/infrastructure-plan.md` §3.4 参照)。OS シンボリックリンク差や base image の差異に依存させない。

> **現行 scaffold への反映状況**: 2026-05-14 時点で `src/main/resources/application.yml` には `spring.datasource.url` 自体が未設定(`spring-boot-testcontainers` 経由で Testcontainers が JDBC URL を生成している)。本規約に沿った URL パラメータの本番 / ステージング向け反映は **Sprint 0 整合タスクで対応**(Sprint 0 トラッキング Issue #121 配下で起票)。それまでは Testcontainers のローカル実行が前提。

## 14. import

- Google Java Format に従い、ワイルドカード import を**禁止**(`import java.util.*;` 等)。
- static import は AssertJ / Hamcrest の matcher 等、慣用的なものに限定する。
- **ネストクラス・静的フィールド・静的メソッドを直接インポートしてはならない**(`BadImport` Error Prone チェックがビルドエラーにする)。短い名前の衝突リスクを避けるため、外側クラスを import して完全修飾で参照する。

```java
// NG: ネストクラスを直接 import
import org.springframework.modulith.ApplicationModule.Type;
@ApplicationModule(type = Type.OPEN) // Type だけ見ても何のことか分からない

// OK: 外側クラスを import して修飾
import org.springframework.modulith.ApplicationModule;
@ApplicationModule(type = ApplicationModule.Type.OPEN)
```

### 16.1 依存追加時の Native 対応確認

依存ライブラリを追加する PR では、追加するライブラリの Native Image 対応状況を以下の順で確認する。GraalVM Native Image で最も多い native build 破壊の原因は **Reflection を使う依存の追加**であるため、PR マージ前に必ず確認すること。

1. **GraalVM Reachability Metadata Repository** に登録されているか確認する([oracle/graalvm-reachability-metadata](https://github.com/oracle/graalvm-reachability-metadata))。登録済みであれば GraalVM Build Tools が自動的に metadata を取得し、手動 hints は不要。

3. **CI の `Native Build` ワークフロー**(`.github/workflows/native-build.yml`、#490)で `nativeCompile` チェックが緑になることを確認する。PR の `Native Build` チェックが赤になった場合は §20.1 の手動 hints 対応を検討する。
4. **非対応ライブラリ**は採用しないか代替を検討し、不可避な場合は `META-INF/native-image/` に hints を追加して対応する。

## 20. Native Image 実装制約

tasks-webapi は **GraalVM Native Image** でビルド・配信する(ADR-0008)。コンテナイメージは `bootBuildImage`(Cloud Native Buildpacks)で生成し、Dockerfile は作成しない(ADR-0018)。GraalVM baseline は **Java 25**(ADR-0021)。Native Image は Ahead-of-Time(AOT)コンパイルを前提とするため、JVM の動的機能(Reflection・動的クラスロード・Spring プロファイル)に制約が課される。本セクションの規約は Sprint 0 App 着手時点から適用する。

### 20.1 Reflection 禁止 / 明示登録

Native Image は Reflection を静的解析できないため、**動的 Reflection は使用しない**か、build-time hint で明示登録する。

**禁止**:

```java
// NG: 実行時に Class を動的解決しているため Native build が失敗する
Class<?> cls = Class.forName("xyz.dgz48.tasks.webapi.task.domain.Task");
Method m = cls.getMethod("getTitle");
```

**許可(Spring 管理 Bean)**: `@Component` / `@Entity` / `@RestController` などの Spring 管理クラスは Spring Boot AOT processor が自動で hints を生成するため追加対応不要。

**明示登録が必要な場合**:

```java
// OK: Jackson でシリアライズする DTO や Spring 管理外のクラスに付与
@RegisterReflectionForBinding(TaskResponse.class)
@Configuration
public class TaskNativeHints {}
```

手動 hints が必要な場合は `src/main/resources/META-INF/native-image/xyz.dgz48.tasks.webapi/reflect-config.json` に記述し、レビューチェックリストで確認する。

### 20.2 Spring プロファイル不使用

`@Profile` アノテーションは **禁止**。Native Image の AOT コンパイルは profile 分岐を build-time に解決するため、実行時の profile 切り替えが機能しない。

**禁止**:

```java
// NG: @Profile は Native Image で問題を起こす
@Profile("dev")
@Bean
public MockMailSender mockMailSender() { ... }
```

**代替**: 環境差は `@Value("${ENV_VAR}")` + ECS Task Definition / Docker Compose の環境変数注入で吸収する。

```java
// OK: 環境変数で制御する
@Bean
public MailSender mailSender(@Value("${MAIL_ENABLED:true}") boolean enabled) {
    return enabled ? new SmtpMailSender() : new NoOpMailSender();
}
```

ローカル開発時は Docker Compose の `environment` セクション、または `.env.local` で同名変数を設定する(`docs/dev/local-setup.md` 参照)。

### 20.3 動的クラスロード禁止

`Class.forName(...)` / `java.lang.invoke.MethodHandles.lookup().findClass(...)` など、文字列でクラスを実行時に解決するコードは **使用禁止**。

```java
// NG
Class<?> cls = Class.forName(className);

// OK: 型安全な注入を使う
@Autowired
private TaskRepository taskRepository;
```

必要な場合は Spring Boot AOT / `@RegisterReflectionForBinding` で対応できないか先に検討し、それでも不可能な場合のみ `reflect-config.json` に追記する。

### 20.4 Native build 設定方針(確定済み)

`webapi/build.gradle.kts` では `org.graalvm.buildtools.native` プラグインが既に適用されており、以下の設定が確定済みである(#490 / #478 / ADR-0018 / ADR-0021)。

**`graalvmNative` ブロック:**

```groovy
graalvmNative {
    // Native test support は使用しない — JVM テストで全シナリオ担保(ADR-0008)。
    testSupport = false
    binaries {
        main {
            imageName = 'tasks-webapi'
            // CI 専用メモリ削減フラグ。-PciNativeBuild=true で有効化(native-build.yml)。
            // PR チェック専用で bootBuildImage には影響しない。
            if (project.hasProperty('ciNativeBuild')) {
                buildArgs.addAll(['-Ob', '-J-Xmx12g', '--parallelism=2'])
            }
        }
    }
    toolchainDetection = false
}
```

- `testSupport = false`: Native テストは使用しない。JVM テストで全シナリオを担保する(ADR-0008)。
- `toolchainDetection = false`: GraalVM ツールチェインの自動検出を無効化し、CI では `graalvm/setup-graalvm@v1` で明示セットアップする。
- `-PciNativeBuild=true` フラグ: CI(native-build.yml)専用の最適化フラグ。`-Ob`(最適化レベル削減)、`-J-Xmx12g`(16GB GHA 標準ランナー向けヒープ上限)、`--parallelism=2`(並列コンパイルスレッド数制限)。PR チェック専用であり、`bootBuildImage`(#481)には影響しない。

**`bootBuildImage` ブロック:**

```groovy
bootBuildImage {
    imageName = "tasks-webapi:${project.version}"
    // Paketo Spring Boot buildpack は Spring Boot >= 4.0 で GraalVM >= 25 を要求する(ADR-0021)。
    // BP_JVM_VERSION=25 を明示することで BellSoft Liberica NIK 25 を使用させる。
    environment = [
        'BP_JVM_VERSION': '25',
        // Password$ConsoleHolder は Java 25 で ProxyingConsole を heap に格納しようとして失敗する。
        // run-time 初期化に延期することで heap-object 競合を回避する。
        'BP_NATIVE_IMAGE_BUILD_ARGUMENTS': '--initialize-at-run-time=sun.security.util.Password$ConsoleHolder'
    ]
}
```

- `BP_JVM_VERSION=25`: GraalVM baseline が Java 25 に引き上げられているため必須(ADR-0021)。
- `BP_NATIVE_IMAGE_BUILD_ARGUMENTS`: Java 25 特有の `Password$ConsoleHolder` heap-object 競合を回避するための run-time 初期化指定。

**`processAot` 環境変数フォールバック:**

```groovy
tasks.named('processAot') {
    environment 'OIDC_ISSUER_URI',     aotEnv('OIDC_ISSUER_URI',     'http://localhost:18080/realms/tasks')
    environment 'DB_HOST',             aotEnv('DB_HOST',             'localhost')
    environment 'DATASOURCE_USERNAME', aotEnv('DATASOURCE_USERNAME', 'tasks_webapi')
    environment 'DATASOURCE_PASSWORD', aotEnv('DATASOURCE_PASSWORD', 'tasks_webapi')
}
```

`processAot` は build 時に本番 Spring コンテキストを起動するため(Flyway / Hibernate validate を含む)、OIDC・DB の env var が必要になる。Docker Compose を起動済みで `.env.local` を source していない場合でも `bootBuildImage` / `nativeCompile` が動くよう、ローカル開発用デフォルトへフォールバックする仕組みが組み込まれている。`processTestAot` も同様のフォールバックを持つ。

### 20.5 CI での Native build 検証

CI での Native build 検証は以下の役割分担で確定している(ADR-0018 §4 / #490):

| 役割 | workflow / task | 内容 |
|---|---|---|
| PR 検証 | `Native Build`(`.github/workflows/native-build.yml`、#490) | `./gradlew :webapi:nativeCompile -PciNativeBuild=true` を実行し native build 破壊を PR で検知する。デプロイ可能なコンテナイメージは生成しない |
| tag 駆動デプロイ | `bootBuildImage`(#481、ADR-0018) | フルコンテナイメージを生成して ECR へ push する。Dockerfile は使用しない |

- **changes ゲート**: `dorny/paths-filter@v3` で `webapi/**` `*.gradle*` `settings.gradle*` `gradle/**` `gradlew*` `native-build.yml` を監視し、関係のない PR では `nativeCompile` job をスキップする。
- **concurrency**: `${{ github.workflow }}-${{ github.ref }}` + `cancel-in-progress: true` で同一ブランチの重複実行をキャンセルする。
- **`Native Build` チェックが赤になったら、まず hints(§20.1)を疑う。**

GHA での GraalVM 25 セットアップ:

```yaml
- uses: graalvm/setup-graalvm@v1
  with:
    java-version: '25'
    distribution: 'graalvm-community'
    cache: 'gradle'
```

### 20.6 ローカル開発戦略

- **通常の開発・デバッグ**: `./gradlew :webapi:bootRun`(JVM 起動)で行う。Native build をローカルで毎回実行する必要はない。
- **Native build のローカル実行は任意**: GraalVM 25 Community ツールチェインをローカルにインストールすれば `./gradlew :webapi:nativeCompile` で試せるが、強制しない。ローカルでは `-PciNativeBuild=true` を**付けない**(CI 専用の GHA ランナー向けメモリ制限フラグであるため)。
- **Native 動作検証**: CI の `Native Build` ワークフロー(`.github/workflows/native-build.yml`)で確認する。ローカルで `nativeCompile` が通らない場合でも、CI で確認してから PR をマージする。
