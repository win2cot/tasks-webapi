## 2. null 安全

### 2.1 全パッケージ `@NullMarked`

- すべての Java パッケージに `package-info.java` を置き、JSpecify の `@NullMarked` を付与する。
- 既存例(踏襲):

```java
// xyz/dgz48/tasks/webapi/task/package-info.java
@NullMarked
package xyz.dgz48.tasks.webapi.task;

import org.jspecify.annotations.NullMarked;
```

- 新規パッケージ追加時は `package-info.java` の作成を**忘れない**。Code Review で必ず確認する。

### 2.2 `@Nullable` の使い方

- フィールド・パラメータ・戻り値は**デフォルトで非 null**(NullMarked による)。`null` を許容する場合のみ `org.jspecify.annotations.@Nullable` を**明示**する。
- 「呼び出し側が null を渡すかもしれない」という防御的 `@Nullable` は付けない(NullAway に頼る)。

### 2.3 JPA エンティティの遅延初期化

- JPA は protected な no-arg コンストラクタからフィールドを反射で埋めるため、コンパイル時には全フィールド未初期化と判定される。対応として、エンティティクラスに **`@SuppressWarnings("NullAway.Init")`** を**クラス単位**で付与する。
- 付与時には「JPA requires no-args constructor; fields initialized via JPA」など**意図コメント**を添える(既存 `Task.java` / `User.java` の形を踏襲)。

### 2.4 テストの null 安全

- テストパッケージは NullAway を**無効化済み**(`compileTestJava` で `disable('NullAway')`)。とはいえ、本番コードの API の null 性は `@Nullable` を信頼してテストを書く。

## 3. Lombok 使用ルール

許容するアノテーションは以下のみ。**それ以外の Lombok アノテーション(例: `@Data` / 無条件の `@AllArgsConstructor` / `@Builder` を JPA Entity に適用 等)は禁止**。

| アノテーション | 用途 | 備考 |
|---|---|---|
| `@Getter` | フィールドへの getter | 設定不変オブジェクトを表現するのに使う |
| `@RequiredArgsConstructor` | DI 用コンストラクタ | Service / Controller でのコンストラクタインジェクション |
| `@NoArgsConstructor(access = AccessLevel.PROTECTED)` | JPA Entity の no-arg コンストラクタ | アクセス修飾子は必ず PROTECTED |
| `@EqualsAndHashCode(onlyExplicitlyIncluded = true)` + `@EqualsAndHashCode.Include` | 特定フィールドのみで等価判定 | Lombok 1.18 以降は `of = "..."` ではなくこの新スタイルを使う(下記サンプル参照)。JPA Entity への適用は避け、`TasksPrincipal` 等の値オブジェクトに限定 |
| `@Slf4j` | logger 自動生成 | ロガー変数は `log` で統一 |

### 3.1 `@EqualsAndHashCode` の推奨形

Lombok 1.18.x 以降、`of` / `exclude` パラメータの利用は非推奨となり、`onlyExplicitlyIncluded` と `@EqualsAndHashCode.Include` の組み合わせが推奨形である。`TasksPrincipal` のような値オブジェクトでも以下の形に揃える。

```java
@EqualsAndHashCode(onlyExplicitlyIncluded = true)
public class TasksPrincipal implements Principal {

  @EqualsAndHashCode.Include
  private final String sub;

  // ... 他フィールドは等価判定に含めない
}
```

- 旧来の `@EqualsAndHashCode(of = "sub")` 形は **書かない**(Lombok の警告対象)。
- 現行 scaffold(`xyz.dgz48.tasks.webapi.security.TasksPrincipal` の `@EqualsAndHashCode(of = "sub")`)は本規約と整合しないため、Sprint 0 整合タスク(Issue #121 配下)で新スタイルへ追従する。

### 3.2 禁止理由(主要なもの)

- `@Data` … `@ToString` による無限再帰、`@EqualsAndHashCode` による JPA 識別子衝突、setter 自動生成によるミューテーション漏れ、を一度に引き起こす。
- 無条件 `@AllArgsConstructor` … JPA Entity に付けると不変条件の維持が破綻する。明示的に public コンストラクタを書く方針(§4 参照)。
- `@Builder`(JPA Entity への適用) … バリデーションを通らないインスタンス生成経路を作る。DTO record では使ってよい。

## 4. JPA エンティティ

### 4.0 ドメイン POJO と JPA Entity の命名規約

設計規約 §1.2.3 が定めるとおり、`domain` 層は POJO のみ・JPA アノテーション禁止であり、JPA Entity は `adapter.persistence` 層に置く。**両者の単純クラス名が衝突するとインポートが破綻し UseCase 層から両方を扱えなくなる**ため、命名規約として以下を強制する。

| 層 | 単純クラス名 | パッケージ例 |
|---|---|---|
| `domain` | `Task`(POJO、@Entity 不可) | `xyz.dgz48.tasks.webapi.task.domain.Task` |
| `adapter.persistence` | **`TaskJpaEntity`**(`@Entity` 必須) | `xyz.dgz48.tasks.webapi.task.adapter.persistence.TaskJpaEntity` |

- **JPA Entity の単純名は必ず `~JpaEntity` で終える**。`Task` のような名前で `@Entity` を付けない。
- domain ↔ persistence の変換は `adapter.persistence` 内の Mapper(または静的ファクトリ)が担う。MapStruct 採否は ADR で決定する(現状未採用)。
- 現行 scaffold(`xyz.dgz48.tasks.webapi.task.Task` に `@Entity`)は feature-by-package 単層構造の暫定形であり、Sprint 0 の整合タスク(Issue #121 配下)で `task.adapter.persistence.TaskJpaEntity` へリネーム・移設する。
- 関連エンティティ(`User` / `Tenant` / `Stakeholder` 等)も同様: `UserJpaEntity` / `TenantJpaEntity` / `TaskStakeholderJpaEntity` のように `~JpaEntity` サフィックス。

### 4.1 正規パターン

新規 JPA Entity は以下の形を踏襲する。

> **注**: 下記サンプルの `TaskStatus` は基本設計書 §4.2.4 / OpenAPI と整合する 4 値(`NOT_STARTED` / `IN_PROGRESS` / `DONE` / `ON_HOLD`)を前提に書いている。現行 scaffold(`xyz.dgz48.tasks.webapi.task.TaskStatus`)は暫定の 2 値(`INCOMPLETE` / `COMPLETE`)であり、Sprint 0 の scaffold ↔ 設計書整合タスク(参考:プロジェクト Issue #84 系)で 4 値へ追従する。サンプルをコピーする際は追従後を前提にしている点に注意。

```java
// xyz/dgz48/tasks/webapi/task/adapter/persistence/TaskJpaEntity.java
// (domain 層の Task(POJO)と区別するため JpaEntity サフィックスを必須化。§4.0 参照)
@Entity
@Table(name = "tasks")
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
@SuppressWarnings("NullAway.Init") // JPA requires no-args constructor; fields initialized via JPA
public class TaskJpaEntity {

  @Id
  @GeneratedValue(strategy = GenerationType.IDENTITY)
  private Long id;

  @Enumerated(EnumType.STRING)
  @Column(nullable = false)
  private TaskStatus status;

  @Column(name = "owner_id", nullable = false)
  private Long ownerId;

  @Column(nullable = false, length = 255)
  private String title;

  @Nullable
  @Column(columnDefinition = "LONGTEXT")
  private String body;

  @Column(name = "tenant_id", nullable = false)
  private Long tenantId;

  // ----- 監査列(設計規約 §3.5) -----
  // 業務テーブルは created_at / updated_at / created_by / updated_by を必須とする。
  // 自動投入の実装方針(Spring Data JPA Auditing の @CreatedDate / @LastModifiedDate /
  // @CreatedBy / @LastModifiedBy + AuditorAware<Long> を採るか、手動セットか)は
  // Sprint 0 の ADR で確定する。下記はカラム宣言のみのサンプル。
  @Column(name = "created_at", nullable = false, updatable = false)
  private OffsetDateTime createdAt;

  @Column(name = "updated_at", nullable = false)
  private OffsetDateTime updatedAt;

  @Column(name = "created_by", nullable = false, updatable = false)
  private Long createdBy;

  @Column(name = "updated_by", nullable = false)
  private Long updatedBy;

  public TaskJpaEntity(
      Long tenantId, TaskStatus status, Long ownerId, String title, @Nullable String body) {
    this.tenantId = tenantId;
    this.status = status;
    this.ownerId = ownerId;
    this.title = title;
    this.body = body;
    // createdAt / updatedAt / createdBy / updatedBy はコンストラクタでは受け取らず、
    // Auditing 機構が永続化時に自動投入する想定(実装方針は Sprint 0 で確定)。
  }
}
```

> 設計規約 §3.3 に従い、業務テーブル系の Entity には `tenant_id` 列を**必須**で持たせる(`users` テーブルのみ既存・例外)。サンプル上も `tenantId` フィールドを明示する。設計規約 §3.5 の監査列 4 つも同様に**必須**であり、サンプル末尾にカラム宣言を含めている。`@CreatedDate` / `@LastModifiedDate` 等の Spring Data JPA Auditing アノテーションを使うか手動セットかは Sprint 0 の ADR で確定する。

### 4.2 ルール

- **setter は作らない**。属性を変えるドメイン操作は**意味のある名前のメソッド**として定義する(例: `task.markAsDone()` / `task.changeOwner(newOwnerId)`)。
- public コンストラクタは**業務的に不変条件を満たす最小セット**を引数に取る。「ID と監査列以外のすべて」を要求するのが目安。
- `@Enumerated(EnumType.STRING)` を必須化(`ORDINAL` は禁止 — DB 値の意味が崩れる)。
- 列名は `@Column(name = "...")` で**snake_case を明示**する。Hibernate の自動変換に依存しない。
- `equals` / `hashCode` は基底クラス(Object)既定を使うか、識別子確定後の比較が必要な場面でのみ実装する。**`@EqualsAndHashCode` を Entity 全体に付けない**(未保存インスタンス同士の比較で問題化する)。

### 4.3 関連

- 関連は**遅延ロード(`FetchType.LAZY`)を既定**とする。N+1 が問題化したら `JOIN FETCH` / `@EntityGraph` で個別解決する。
- 親子の cascade は実テーブル設計と一致させる(設計規約 §3.4)。

## 5. DTO / record

- REST 入出力 DTO は **Java `record`** を用い、`adapter.web.dto` 配下に置く。
- record に Bean Validation アノテーションを直接付与する。

```java
public record TaskCreateRequest(
    @NotBlank @Size(max = 255) String title,
    // body はアプリ層上限として 64 KiB(65535)とする。DDL 側は LONGTEXT(物理上限 4 GiB)で
    // データベース制約を最小化し、API/UI で扱える文字量はこの DTO の @Size で律する。
    // 値変更時は OpenAPI / 設計書 / DDL の 3 点を同時更新する(設計規約 §2.5)。
    @Nullable @Size(max = 65_535) String body,
    @NotNull Visibility visibility) {}
```

- record は不変、Domain との変換は Mapper(または静的ファクトリ)で行う。
- 内部のドメインオブジェクトと record を**直接相互参照させない**(`adapter` 層に閉じ込める)。

## 6. 例外処理

### 6.1 例外階層

- アプリの業務例外は `xyz.dgz48.tasks.webapi.shared.exception.DomainException`(`RuntimeException` のサブクラス)を基底とする。
- 認可違反は HTTP マッピング先ごとに**別クラス**を作る。設計規約 §2.3 のマッピングに従う。

| 例外クラス | HTTP | 用途 |
|---|---|---|
| `TaskNotViewableException` | 404 | GET で参照不可(NIST AC-4) |
| `TaskOwnershipException` | 403 | 編集系で権限不足 |
| `TenantBoundaryViolationException` | 403 | テナント越境アクセス |
| `ValidationFailedException` | 400 | サーバ側で検出した業務制約違反(Bean Validation で表現できないもの) |
| `ConflictException` | 409 | 一意制約違反等 |

### 6.2 マッピング

- `@RestControllerAdvice` で全例外を **§2.4 のエラー応答スキーマ**(`timestamp` / `status` / `error` / `code` / `message` / `path` の 6 フィールド)に変換する。**具体的な戻り型(Spring の `ProblemDetail` を拡張 / 独自 record / その他)は ADR で確定**(想定 ADR-0005)。Controller 内で `try-catch` して `ResponseEntity` を組み立てない。
- 例外クラス名と `code` は 1:1 で対応させる(`TaskOwnershipException` → `code: E_FORBIDDEN`)。

### 6.3 例外を投げる場所

設計規約 §5.2 の「業務認可は `domain` 層の `AuthorizationDomainService` が**真偽**を返す純粋関数、`usecase` 層がそれを受けて例外を投げる」方針に揃え、層ごとに以下のとおり分離する。

- **業務不変条件違反は `domain` 層で例外を投げる**。`domain` のエンティティ・値オブジェクト・ドメインサービスが**自分自身の不変条件**を守るために投げる例外。
  - 例: タスクのステータスを `DONE` から `IN_PROGRESS` へ戻そうとする(状態遷移違反) → `TaskStatusTransitionException`
  - 例: 関係者リストに既に存在するユーザーを再追加する(重複違反) → `DuplicateStakeholderException`
- **認可違反は `usecase` 層で例外を投げる**。`domain` 層の `AuthorizationDomainService`(例: `TaskAuthorizationDomainService`)が「**この主体はこの操作を許可されているか**」を真偽で返し、`usecase` がその結果を受けて §2.3 のマッピングに従う例外を throw する。
  - 例: 自分以外の所有者のタスクを編集しようとした → `TaskAuthorizationDomainService#canEdit(...)` が `false` → `usecase` が `TaskOwnershipException`(403)を throw
  - 例: 他人の PRIVATE タスクを参照しようとした → `TaskAuthorizationDomainService#canView(...)` が `false` → `usecase` が `TaskNotViewableException`(404)を throw
- **入力検証失敗は Bean Validation に任せる**(`MethodArgumentNotValidException` → 400)。`usecase` / `domain` で再検証しない。

## 7. ログ

- `org.slf4j.Logger` を `@Slf4j` で取得し、変数名 `log` で統一。
- `System.out.println` / `e.printStackTrace()` / `Throwable#getStackTrace` を直接出力する書き方は禁止。
- **MDC 必須項目**: `tenantId` / `userId` / `requestId`(設計規約 §4.2)。
- フォーマットは SLF4J プレースホルダ(`log.info("Task {} updated by {}", taskId, userId)`)を使う。文字列連結禁止。
- 例外を再ラップする場合、`cause` を保持する(`new XException("msg", e)`)。
- **PII(氏名・メール・電話番号)を直接出さない**。マスク化(`example@***.com`)するか、`userId`(内部 ID)で代替する。

| パッケージ | 小文字(単語連結) | `xyz.dgz48.tasks.webapi.task` |

## 9. テスト規約

### 9.1 種別と命名

| 種別 | ファイル名 | アノテーション |
|---|---|---|
| 単体テスト(POJO / UseCase) | `<対象クラス>Test` | プレーン JUnit 5 |
| Slice テスト(Web 層) | `<Controller>WebMvcTest` | `@WebMvcTest` |
| Slice テスト(JPA 層) | `<Repository>JpaTest` | `@DataJpaTest` + Testcontainers |
| 結合テスト | `<シナリオ>IT` | `@SpringBootTest` + Testcontainers |
| モジュール検証 | `ModularityTests` | Spring Modulith `ApplicationModules#verify()` |

### 9.2 DB を使うテスト

- インメモリ H2 / HSQLDB は**禁止**(MySQL 8.x と方言差・列型差が出る)。
- **Testcontainers + MySQL 8.4** を `@ServiceConnection` で接続する。既存 `TestcontainersConfiguration.java` を踏襲する。

### 9.3 セキュリティテスト

- `@WithMockUser` ではなく、`TasksPrincipal` を投入できる `@WithMockJwt` 等のユーティリティを使う(Sprint 0 トラッキング Issue #121 配下で起票・実装する)。`TasksAuthenticationToken` を経由する経路を必ずテストする。

### 9.4 クロステナント漏洩テスト

- 業務 feature を実装する PR では、テナント A のユーザーがテナント B のリソースに対して以下を検証する**結合テストを必ず追加**する(設計規約 §2.3 / §3.3 / §6):
  - **参照系(GET)**: **404 Not Found** が返ること(リソースの存在自体を漏らさない、NIST AC-4)。
  - **更新系(PUT / PATCH / DELETE)**: **403 Forbidden** が返ること(テナント越境拒否)。
  - レスポンス本文の `code` が `E_NOT_FOUND` / `E_FORBIDDEN` であること(§2.4 のエラー応答スキーマ)。

### 9.5 カバレッジ

- JaCoCo 命令網羅 **80%** が CI で enforce 済み(全体 + 変更ファイル)。
- 単に通すためのテスト・トートロジカルなアサーションは書かない。レビューで指摘される。

### 9.6 アサーションスタイル

- AssertJ(`assertThat`)を優先する。`org.junit.jupiter.api.Assertions.assertEquals` は補助的に使ってよい。

## 10. 依存性注入

- **コンストラクタインジェクション**を原則とする(`@RequiredArgsConstructor` + `final` フィールド)。フィールドインジェクション(`@Autowired` を直接フィールドに)は禁止。
- セッターインジェクションは原則禁止(循環依存の温床)。

```java
@Service
@RequiredArgsConstructor
public class TaskUseCase {
  private final TaskRepository taskRepository;
  private final AuditLogger auditLogger;
  // ...
}
```

- Spring の `Configuration` ノートをライブラリ的に隠したい場合のみ、`@Bean` メソッドで明示する。

## 11. トランザクション

- `@Transactional` は **`usecase` 層のみ**に付与する。Controller / Repository に付けない(設計規約 §1.2.3)。
- 読み取りのみのユースケースには `@Transactional(readOnly = true)` を付ける。
- ネストや伝播属性(`Propagation.REQUIRES_NEW` 等)を使う場合は、**メソッドの Javadoc に理由を残す**。

## 12. 日時・タイムゾーン

- DB 列型は `DATETIME`(MySQL)。Java では原則 **`java.time.OffsetDateTime`** または **`java.time.Instant`** を使う。
- 「日付」の意味で十分なら `LocalDate`。`LocalDateTime` は JDBC マッピング上必要な箇所だけに留める(タイムゾーン情報を失うため)。
- CI 環境は `TZ=Asia/Tokyo` 固定(`cicd.yml`)。テストで `LocalDateTime.now()` を直接使うとマシン依存で失敗するため、`Clock` を DI して固定化する。

### 12.1 JDBC ↔ Java 日時型のマッピング設定(重要)

MySQL の `DATETIME` 列はタイムゾーン情報を持たないため、何も設定せず `OffsetDateTime` を読み書きすると、JDBC ドライバが JVM デフォルト TZ で暗黙変換し**意図しないオフセット差**が発生する。これを防ぐため、Spring Boot の JDBC URL に以下を明示する。

```yaml
# application.yml(本番・テスト共通)
spring:
  datasource:
    url: jdbc:mysql://${DB_HOST}:3306/${DB_NAME}?connectionTimeZone=SERVER&forceConnectionTimeZoneToSession=true
```

- `connectionTimeZone=SERVER`: JDBC 接続時に MySQL サーバ TZ をクライアント側 TZ として採用(MySQL 接続側で `Asia/Tokyo` を返す前提)。
- `forceConnectionTimeZoneToSession=true`: セッション TZ を上記接続 TZ に強制し、`NOW()` 等の DB 関数も同じ TZ で評価。
- MySQL Connector/J 8.0.23 以降のオプション名で記述。古い `serverTimezone` パラメータは使わない。
- RDS のパラメータグループで `time_zone = Asia/Tokyo` を設定し、JVM・CI・DB 全てが Asia/Tokyo に揃っていることを前提とする。
- Testcontainers ベースのテストでも本設定を再現する(`MySQLContainer#withUrlParam` 等で同じパラメータを付与)。
- **API レスポンス上のタイムスタンプは UTC(`Z` サフィックス)で返す**(設計規約 §2.4)。サーバ TZ が `Asia/Tokyo` でも、Jackson のシリアライズ時に `Instant` または `ZoneOffset.UTC` に正規化された `OffsetDateTime` を使い、`+09:00` 表記は出さない。`application.yml` の `spring.jackson.time-zone: UTC` および `spring.jackson.serialization.write-dates-as-timestamps: false` を Sprint 0 で設定する(同 Issue #121 配下)。

> **現行 scaffold への反映状況**: 2026-05-14 時点で `src/main/resources/application.yml` には `spring.datasource.url` 自体が未設定(`spring-boot-testcontainers` 経由で Testcontainers が JDBC URL を生成している)。本規約に沿った URL パラメータの本番 / ステージング向け反映は **Sprint 0 整合タスクで対応**(Sprint 0 トラッキング Issue #121 配下で起票)。それまでは Testcontainers のローカル実行が前提。

## 13. 文字列リテラルと国際化

- ログメッセージは英語可。**ユーザー向けエラーメッセージは日本語**(ErrorResponse の `message`)。
- 表示用の文字列は基本的にフロントエンドの i18n リソースで管理する。バックエンドが日本語文字列を直接返すのは ErrorResponse 等の最小限に留める。

## 14. import

- Google Java Format に従い、ワイルドカード import を**禁止**(`import java.util.*;` 等)。
- static import は AssertJ / Hamcrest の matcher 等、慣用的なものに限定する。

---

## 15. コメント

- 「なぜそう書いたか」を書く。「何をしているか」はコードから読めるので不要。
- 公開 API の Java クラス・メソッドには Javadoc を書く。`@param` / `@return` / `@throws` を簡潔に。
- `TODO` / `FIXME` を残す場合は `// TODO(#123): ...` のように **Issue 番号** を付ける。番号なし TODO は禁止。

---

## 16. ビルド・依存追加

- 新しいライブラリの追加は **ADR** を起票してから `build.gradle` を編集する(設計規約 §8)。
- Spring Boot 経由で取れる依存は **バージョン指定なし**(BOM に任せる)。
- Renovate による自動更新を尊重する。手動でメジャー版を上げる場合は ADR を残す。

## 17. PR とコミット

- 1 PR の目安は**400 行以下**、最大 1000 行(開発計画書 §5.3)。それを超える場合は分割。
- コミットメッセージは1行目 50 字以内、必要なら本文に詳細を書く(Conventional Commits の `<type>: <subject>` を**推奨**するが強制はしない)。
- PR を上げる前に必ず `./gradlew check` をローカルで通す。

## 18. レビューチェックリスト

- [ ] JPA Entity が §4 のパターンに沿っているか(setter 無、cons. 引数最小セット、`tenant_id` フィールドあり)

- [ ] 認可違反が発生する経路に audit_logs 記録があるか
