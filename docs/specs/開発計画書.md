本書は、タスク管理システムの開発プロジェクトにおける計画(体制・スケジュール・成果物・品質・リスク等)を定め、関係者間で合意を得ることを目的とする。

## 2. プロジェクト概要

### 2.1 プロジェクト名

タスク管理システム開発プロジェクト(コードネーム: TaskFlow)

### 2.2 開発の目的

チーム/小規模ビジネスにおけるタスク管理を一元化し、抜け漏れ削減・進捗可視化・ナレッジ蓄積を実現する。マルチテナント型SaaSとして提供することで、複数の契約企業へ同時提供可能とする。

### 2.3 達成基準(完了条件)

- 要件定義書記載の全機能要件(F-01〜F-19、カテゴリ・チーム機能を除く)が実装されていること
- Keycloak Realm が本番環境で稼働し、既存usersとの突合が動作すること
- 非機能要件(性能・セキュリティ・可用性)を満たすこと
- クロステナント漏洩テスト・タスク認可テストで全件パスしていること
- ユーザー受入テスト(UAT)で重大欠陥0件、軽微欠陥10件以下
- 運用ドキュメント・利用マニュアルが揃っていること(GitHub Wikiにて公開)
- 本番環境でパイロットテナント3社が安定稼働できること

## 5. 開発手法

### 5.1 開発プロセス

- 基本はウォーターフォール、実装フェーズはスクラム的(2週間スプリント)に運営
- 毎日15分のデイリースタンドアップを実施
- スプリントレビュー・レトロスペクティブを各スプリント末に実施

### 5.1.1 ビルド・依存管理

- Gradle 8.x (Kotlin DSL) を採用、settings.gradle.kts でマルチモジュール構成も視野
- 依存性ロックファイル(gradle.lockfile)をコミットし再現性を確保
- CI で gradle dependencyCheckAnalyze による脆弱性スキャンを実施

### 5.2 ブランチ戦略

GitHub Flow を採用する。

- main: 常に本番デプロイ可能な状態を維持
- feature/*: 機能単位のブランチ。main から分岐し、PR経由でmainへマージ
- hotfix/*: 本番障害対応用。リリース後すぐにmainへマージ
- リリース時にタグ(v1.0.0等)を付与

### 5.3 コードレビュー

- 全PRに対し1名以上のレビュアーが必須(テックリードか他メンバー)
- CIが全てグリーンであること
- コードレビュー観点: 可読性 / 設計遵守 / テストの妥当性 / セキュリティ
- PR1件は理想 ≦400行、最大 ≦1000行を目安

## 9. 課題・変更管理

### 9.1 課題管理

- 全課題は GitHub Issues + GitHub Projects (V2) で管理
- ステータス: Open / In Progress / In Review / Closed
- 優先度: P0(緊急) / P1(高) / P2(中) / P3(低)
- 毎週月曜の定例で課題ステータスをレビュー

### 9.2 変更管理

- 要件変更はPM経由で「変更要求書」を提出
- PMは影響度(工数・スケジュール・コスト)を評価
- 週次変更管理委員会(PM・TL・利用部門代表)で承認/却下
- 承認された変更は要件定義書・設計書に反映、関係者へ通知

## 11. 環境・インフラ

### 11.1 環境一覧

| 環境 | 用途 | 主な構成 |
|---|---|---|
| 開発環境(local) | 個人開発用 | Docker Compose (Spring Boot 4 + MySQL 8 + Keycloak) |

### 11.2 CI/CD

- Pull Request 作成時: ビルド + 単体テスト + 静的解析 + Trivy脆弱性スキャン + クロステナントテスト
- main マージ時: ECRへイメージpush、ECSステージングへBlue/Greenデプロイ
- タグ付与時: 本番環境への手動承認デプロイ(Blue/Greenデプロイ)
- 各ジョブで Gradle のキャッシュを活用しビルド時間を短縮
- IaCはTerraform、リポジトリと連携
