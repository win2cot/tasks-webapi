## 2. プロジェクト概要

### 2.1 プロジェクト名

タスク管理システム開発プロジェクト(コードネーム: TaskFlow)

### 2.2 開発の目的

チーム/小規模ビジネスにおけるタスク管理を一元化し、抜け漏れ削減・進捗可視化・ナレッジ蓄積を実現する。マルチテナント型SaaSとして提供することで、複数の契約企業へ同時提供可能とする。

### 2.3 達成基準(完了条件)

- 要件定義書記載の全機能要件(F-01〜F-19、カテゴリ・チーム機能を除く)が実装されていること
- Keycloak Realm が本番環境で稼働し、既存usersとの突合が動作すること
- 非機能要件(性能・セキュリティ・可用性)を満たすこと
- クロステナント漏洩テスト・タスク認可テストで全件パスしていること
- ユーザー受入テスト(UAT)で重大欠陥0件、軽微欠陥10件以下
- 運用ドキュメント・利用マニュアルが揃っていること(GitHub Wikiにて公開)
- 本番環境でパイロットテナント3社が安定稼働できること

## 5. 開発手法

### 5.1 開発プロセス

- 基本はウォーターフォール、実装フェーズはスクラム的(2週間スプリント)に運営
- 毎日15分のデイリースタンドアップを実施
- スプリントレビュー・レトロスペクティブを各スプリント末に実施

### 5.1.1 ビルド・依存管理

- Gradle 8.x (Kotlin DSL) を採用、settings.gradle.kts でマルチモジュール構成も視野
- 依存性ロックファイル(gradle.lockfile)をコミットし再現性を確保
- CI で gradle dependencyCheckAnalyze による脆弱性スキャンを実施

### 5.2 ブランチ戦略

GitHub Flow を採用する。

- main: 常に本番デプロイ可能な状態を維持
- feature/*: 機能単位のブランチ。main から分岐し、PR経由でmainへマージ
- hotfix/*: 本番障害対応用。リリース後すぐにmainへマージ
- リリース時にタグ(v1.0.0等)を付与

### 5.3 コードレビュー

- 全PRに対し1名以上のレビュアーが必須(テックリードか他メンバー)
- CIが全てグリーンであること
- コードレビュー観点: 可読性 / 設計遵守 / テストの妥当性 / セキュリティ
- PR1件は理想 ≦400行、最大 ≦1000行を目安

## 9. 課題・変更管理

### 9.1 課題管理

- 全課題は GitHub Issues + GitHub Projects (V2) で管理
- ステータス: Open / In Progress / In Review / Closed
- 優先度: P0(緊急) / P1(高) / P2(中) / P3(低)
- 毎週月曜の定例で課題ステータスをレビュー

### 9.2 変更管理

- 要件変更はPM経由で「変更要求書」を提出
- PMは影響度(工数・スケジュール・コスト)を評価
- 週次変更管理委員会(PM・TL・利用部門代表)で承認/却下
- 承認された変更は要件定義書・設計書に反映、関係者へ通知

## 11. 環境・インフラ

### 11.1 環境一覧(v1.3 で 5 環境に拡張)

| 環境名 | 用途 | 主な構成 | MVP リリース時 |
|---|---|---|---|
| local | 個人開発用 | Docker Compose(MySQL 8 + Keycloak Custom)+ tasks-webapi 直起動 + web/ 静的配信 | **稼働** |
| gha | CI 環境 | GitHub Actions / Gradle / OIDC AWS 認証 / Terraform plan&apply | **稼働** |
| dev | 開発・統合テスト用 AWS 環境 | ALB(single)+ ECS Fargate(tasks-webapi×1 / Keycloak Custom×1)+ RDS MySQL(db.t4g.micro、Single AZ)+ Parameter Store + S3/CloudFront + Route53 + ACM + ECR | **稼働** |
| stg | ステージング・UAT 用 AWS 環境 | 本番同構成(縮小サイズ)、Keycloak Realm 別 | **構築せず**(MVP 後の Post-Sprint-0 で構築) |
| prd | 本番・実利用 | AWS ALB + ECS Fargate(tasks-webapi×2 / Keycloak Custom×2)+ RDS MySQL Multi-AZ + S3/CloudFront | **構築せず**(MVP リリース時に構築) |

#### 環境階層の方針(v1.3 で追記)

- **コンテナイメージは全環境共通**(環境別ファイル同梱せず、Native Image 移行を見据え Spring プロファイルは不使用)
- **環境差は環境変数注入で吸収**(ECS Task Definition / Docker Compose で `DATASOURCE_URL` `OIDC_ISSUER_URI` 等)
- **シークレットは Parameter Store SecureString** で管理(`/tasks/<env>/...` パス階層、Secrets Manager は不採用 - コスト面)
- **ドメイン体系**: `tasks.dgz48.xyz` 系(`tasks-dev` / `api-dev.tasks` / `auth-dev.tasks`、stg/prd は環境 suffix 変更)
- **RDS 認証(v1.3)**: tasks-webapi は **IAM 認証**(AWSAuthenticationPlugin、パスワード不使用)/ Keycloak は **パスワード認証**(Parameter Store)/ master は DBA 業務用(Parameter Store、アプリ非使用)

詳細は `docs/architecture/infrastructure-plan.md` v5 §3 を参照。

### 11.2 CI/CD(v1.3 で tag 駆動+OIDC+Parameter Store に更新)

- **Pull Request 作成時(claude-impl/claude-review 自動応答含む)**: ビルド + 単体テスト + 静的解析 + Trivy 脆弱性スキャン + クロステナントテスト
- **GitHub Tag 作成時に環境別デプロイ自動起動**(SemVer 2.0 pre-release 整合)
  - `vX.Y.Z-dev` → dev 環境への自動 deploy(ECR push + ECS update-service、または S3 sync + CloudFront invalidation、または `terraform apply`)
  - `vX.Y.Z-stg` → stg 環境(MVP 後)
  - `vX.Y.Z`(suffix 無)→ prd 環境(MVP 後、手動承認ステップ付き Blue/Green)
- **AWS 認証は OIDC 経由 GitHub Actions**(IAM Role を `assume`、手元 `aws configure` は使わない、ロングリブの IAM User 不発行)
- **シークレット注入**: ECS Task Definition の `secrets` フィールドから Parameter Store SecureString を参照(平文 secrets が Git / Terraform state に出ない)
- **IaC**: Terraform(`infra/` subdir、state は S3 + DynamoDB)、`infra/v*-*` tag で `terraform apply` 自動実行
- **マルチコンポーネント**: tasks-webapi / keycloak custom / web / infra の 4 コンポーネントを単一リポ(`tasks-webapi` monorepo)で管理、Renovate `packageRules` で多 path 対応
- **ビルドキャッシュ**: 各ジョブで Gradle / npm キャッシュを活用しビルド時間を短縮

詳細は `docs/architecture/infrastructure-plan.md` v5 §3 を参照。
