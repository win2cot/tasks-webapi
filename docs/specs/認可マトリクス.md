## 1. 概要

本書は tasks-webapi における認可の全組み合わせを表形式で記述する。**設計規約 §5.2** および **ADR-0005** に基づき、タスク系 API の認可は以下の 3 次元で評価する。

1. **アクター**: SaaS Admin / Tenant Admin / Member / 未認証
2. **テナントスコープ**: 自テナント(メンバー) / 他テナント(非メンバー) / テナント未指定(X-Tenant-Id なし)
3. **操作 × タスク属性**: GET(visibility 別) / PATCH status(所有者・担当者)

ADR-0005 の決定により、タスク認可は **所有者・担当者・関係者の 3 役割のみ** で評価する。Tenant Admin の業務タスク特権は撤廃済み。

---

## 2. 認可レイヤーの責務分担

認可は 5 層で分担する。各層が重複なく責務を持ち、Controller 内での業務認可 if 分岐は禁止(設計規約 §5.2)。

| 層 | クラス / 機構 | 担当内容 | 違反時レスポンス |
|---|---|---|---|
| **粗粒度認可** | `@PreAuthorize` on Controller | ロールの有無・テナントロール所持確認 | 403 |
| **テナント所属確認** | `TenantContextFilter` | X-Tenant-Id の user_tenants 照合、ROLE 付与、TenantContext 設定 | 403 |
| **テナント分離** | Hibernate Filter (`tenantFilter`) | 全 SELECT に `tenant_id = :tenantId` を自動付与 (ADR-0010) | — (404 に見える) |
| **業務認可** | `TaskAuthorizationDomainService` (domain 層) | visibility × 所有者・担当者・関係者の細粒度判定 | `TaskNotViewable` / `TaskOwnership` |
| **例外変換** | `TaskExceptionHandler` (@RestControllerAdvice) | 業務例外 → HTTP ステータス変換 | 404 / 403 |

### GET 系の 404 翻訳ポイント

GET 系でリソースが参照不可な場合、**UseCase 層が `TaskNotViewableException` を投げ、TaskExceptionHandler が 404 に変換する**。Controller では例外を catch しない(設計規約 §4.1)。これにより NIST AC-4(情報フロー保護)に従い、存在自体を漏らさない。

## 3. ロール定義

| ロール | 付与条件 | Spring Security GrantedAuthority |
|---|---|---|
| SaaS Admin | Keycloak realm role `APP_ADMIN` を所持 | `ROLE_APP_ADMIN` |
| Tenant Admin | `user_tenants.role = TENANT_ADMIN` かつ X-Tenant-Id が一致するテナントの ACTIVE メンバー | `ROLE_TENANT_ADMIN` |
| Member | `user_tenants.role = MEMBER` かつ X-Tenant-Id が一致するテナントの ACTIVE メンバー | `ROLE_MEMBER` |
| 未認証 | JWT なし / 無効 | — |

**重要**: SaaS Admin は `user_tenants` に登録されない(設計規約 §5.2)。したがって、SaaS Admin がテナントスコープの API を呼ぶには X-Tenant-Id が必要だが、TenantContextFilter が user_tenants に存在しないと判定して 403 を返す。SaaS Admin にタスク系 API へのアクセス経路は存在しない。

## 4. 認可マトリクス — タスク参照 (GET /api/tasks/{id})

### 4.1 アクター × テナントスコープ

| アクター | X-Tenant-Id | 結果 | 判定層 |
|---|---|---|---|
| **未認証** | 任意 | **401** | SecurityConfig (.authenticated()) |
| **SaaS Admin** (ROLE_APP_ADMIN のみ) | なし | **403** | @PreAuthorize (TENANT_ADMIN/MEMBER なし) |
| **SaaS Admin** | あり(任意) | **403** | TenantContextFilter (user_tenants に不在) |
| **Tenant Admin / Member** | なし | **403** | @PreAuthorize (テナントロール未付与) |
| **Tenant Admin / Member** | 所属していないテナント ID | **403** | TenantContextFilter (非メンバー) |
| **Tenant Admin / Member** | 自テナント ID | → 4.2 へ | TenantContextFilter 通過 |

### 4.2 自テナント × タスク visibility × 役割関係

前提: アクターは自テナントのメンバー(Tenant Admin または Member)

| タスク visibility | 操作者とタスクの関係 | 結果 | 判定層 |
|---|---|---|---|
| **TENANT** | テナントメンバー全員 | **200** | TaskAuthorizationDomainService |
| **STAKEHOLDERS** | 所有者 | **200** | TaskAuthorizationDomainService |
| **STAKEHOLDERS** | 担当者 | **200** | TaskAuthorizationDomainService |
| **STAKEHOLDERS** | task_stakeholders 登録済み | **200** | TaskAuthorizationDomainService |
| **STAKEHOLDERS** | 上記以外のテナントメンバー | **404** | TaskAuthorizationDomainService → TaskExceptionHandler |
| **PRIVATE** | 所有者 | **200** | TaskAuthorizationDomainService |
| **PRIVATE** | 担当者 | **200** | TaskAuthorizationDomainService |
| **PRIVATE** | 上記以外(Tenant Admin を含む) | **404** | TaskAuthorizationDomainService → TaskExceptionHandler |

### 4.3 クロステナントアクセス

| シナリオ | 結果 | 判定層 |
|---|---|---|
| テナント A のメンバーが X-Tenant-Id: A でテナント B のタスク ID を指定 | **404** | Hibernate Filter(tenant_id = A でフィルタ → 見つからない) → TaskExceptionHandler |

## 5. 認可マトリクス — ステータス変更 (PATCH /api/tasks/{id}/status)

### 5.1 アクター × テナントスコープ

| アクター | X-Tenant-Id | 結果 | 判定層 |
|---|---|---|---|
| **未認証** | 任意 | **401** | SecurityConfig (.authenticated()) |
| **SaaS Admin** | なし | **403** | @PreAuthorize |
| **SaaS Admin** | あり(任意) | **403** | TenantContextFilter |
| **Tenant Admin / Member** | なし | **403** | @PreAuthorize |
| **Tenant Admin / Member** | 所属していないテナント ID | **403** | TenantContextFilter |
| **Tenant Admin / Member** | 自テナント ID | → 5.2 へ | TenantContextFilter 通過 |

### 5.2 自テナント × 役割関係

前提: アクターは自テナントのメンバー(Tenant Admin または Member)

| 操作者とタスクの関係 | タスクが参照可能か | 結果 | 判定層 |
|---|---|---|---|
| 所有者 | ○(任意 visibility) | **200** | TaskAuthorizationDomainService |
| 担当者 | ○(任意 visibility) | **200** | TaskAuthorizationDomainService |
| 上記以外 | ×(参照不可) | **404** | TaskAuthorizationDomainService → TaskExceptionHandler |
| 上記以外 | ○(参照可能だが権限不足) | **403** | TaskAuthorizationDomainService → TaskExceptionHandler |

> **注意**: ステータス変更は所有者・担当者のみ可(ADR-0005 §3.1)。Tenant Admin であっても所有者・担当者でなければ変更不可。参照不可なら 404 を返し、参照可能で変更権限がなければ 403 を返す。

### 5.3 クロステナントアクセス

| シナリオ | 結果 | 判定層 |
|---|---|---|
| テナント A のメンバーが X-Tenant-Id: A でテナント B のタスク ID を指定 | **404** | Hibernate Filter → TaskExceptionHandler |

## 6. クロステナント分離の保証経路

クロステナントアクセスは 2 層で多重防御する。

```
[リクエスト]
  │
  ▼
TenantContextFilter
  ├─ X-Tenant-Id のテナントに非メンバー → 403 で即時遮断
  └─ メンバー → TenantContext.set(tenantId) → フィルタ有効化
  │
  ▼
TenantAwareJpaTransactionManager
  └─ トランザクション開始時に Hibernate Session へ tenantFilter を enable
  │
  ▼
taskRepository.findById(taskId)
  └─ SQL: SELECT ... WHERE tenant_id = :tenantId AND id = :taskId
     → 他テナントのタスクは行ごと見えない → 空 Optional → TaskNotFoundException → 404
```

どちらの層が遮断しても最終応答は 403 または 404 であり、他テナントのタスクの存在は漏洩しない。
