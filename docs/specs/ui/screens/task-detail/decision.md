## 1. 採用案: **案 B 2 カラム(Two Column / Issue 風)**

ファイル: [`option-b-two-column.html`](./option-b-two-column.html)

### 採用の理由

1. **「腰を据えて読む」動線と PC 効率を両立する**
   - 左カラム(主情報:タイトル + 説明 + 関係者)を広く取ることで Member が自分の担当タスクの説明を読み下すフローが自然。
   - 同時に右カラム(メタ + 操作:ステータス / 優先度 / 期限 / 担当者 / 公開範囲 / 操作ボタン群)を 1 画面内に固定配置することで、読みながら操作する PC 効率も満たす。
   - 案 A は「読む」に最適だが PC で右半分の余白が無駄になる。案 C は「項目スキャン + 一括編集」で「腰を据えて読む」とは方向が違う。
2. **GitHub Issues / Linear 経験者に馴染む**
   - 開発業務に従事する Member の多くは Issue ツールに慣れている。案 B のレイアウトは違和感なく操作できる。
3. **ドロワー(狭幅)体験差を許容するユーザ判断**
   - ドロワー狭幅では 2 カラムを縦積みにフォールバックする(案 A 相当の体験になる)が、user は「体験差は許容」と回答。
   - 主要動線は単票閲覧(ドロワー + フルページ両方ありうるが、フルページ = PC ブラウザ直 URL アクセス時)で、PC フルページ時の効率最大化を優先する。
4. **S-03 / S-04 の「画面ごとのゴール優先」原則と整合**
   - S-04 ではタスク一覧として案 B 左サイドカレンダー(2 カラム別構造)を採用。S-05 も S-03 の縦積みに無理に揃えず、本画面のゴールに最適な構造を選ぶ判断と一貫している。

### 採用案の主な構成要素(Claude Design 出力から)

- **トップナビ**(S-03 / S-04 継承): ブランド + テナント chip + ロール切替セレクタ + ユーザアバター + **視点切替セレクタ**(所有者 / 担当者 / 関係者 / その他参照可)。
- **左メインサイドナビ**(S-03 / S-04 継承): タスク詳細直接アクセス時は active なし。
- **ドロワー / フルページ表示切替トグル**: モック上で両形態を確認できる UI。本実装では URL ベースで自動判定:
  - 行クリック起点(S-03 / S-04 から)= ドロワー
  - URL 直接アクセス(`/tasks/{id}`、`/tasks/{id}/edit`)= フルページ fallback
- **左カラム(主情報)**:
  - タイトル(編集モードでは text input、参照モードでは大きく表示)
  - 説明(編集モードでは textarea、参照モードでは整形表示。Markdown は将来検討、本 PR では plain text)
  - 関係者リスト(各関係者のアバター + 名前 chip、所有者 ∪ 担当者には削除 × アイコン + 「+ 追加」ボタン表示)
- **右カラム(メタ + 操作)**:
  - ステータス(プルダウン、所有者 ∪ 担当者で活性)
  - 優先度(ラジオ or プルダウン、所有者のみ活性)
  - 期限(date picker、所有者のみ活性)
  - 担当者(プルダウン、所有者のみ活性、テナント内ユーザー)
  - 公開範囲(visibility ラジオ、所有者のみ活性、`PATCH /api/tasks/{id}/visibility` = A-17 経由)
  - メタ情報(作成日 / 更新日 / `completed_at` 完了時のみ)
  - 操作ボタン群:編集ボタン(参照モード時、所有者のみ表示)/ 削除ボタン(所有者のみ表示、確認ダイアログ)
- **行レベル認可フラグ**(S-03 / S-04 継承): `canFields` / `canStatus` / `canStakeholder` / `canVisibility` / `canDelete` 5 フラグで UI を動的判定。操作不可項目は `cell-locked` クラス相当(グレー + 鍵アイコン + tooltip で「所有者のみ」等の理由表示)。
- **ETag / If-Match 楽観ロック**(ADR-0012): GET レスポンスの ETag を保持、PATCH 時に If-Match ヘッダで送信。412 Precondition Failed 時の衝突表示は #154 で具体化。
- **論理削除**: 所有者のみ「削除」ボタン表示、クリックで確認ダイアログ(タスクタイトルを表示して誤クリック防止)→ `DELETE /api/tasks/{id}` = A-15。
- **visibility バッジ**(S-03 / S-04 継承): TENANT(緑)/ STAKEHOLDERS(青)/ PRIVATE(紫)+ アイコン。タイトル横に表示。
- **監査ログ表示は本画面に出さない**: Tenant Admin が `/api/audit-logs` で別途参照する設計(MVP 範囲、本 PR スコープ外)。

### 1.1 ハイファイ化で追加された UI 要素(2026-06-02 第 2 反復)

採用後に Claude Design に作り込みを依頼し、以下を追加実装:

- **ローディングスケルトン**: GET /tasks/{id} 取得中の placeholder 表示
- **保存スピナー**: PATCH 送信中の操作ボタン disabled + spinner
- **トースト通知**: 保存成功 / 失敗のフィードバック(画面右下)
- **入力バリデーション + 文字数カウンタ**: タイトル(100 文字)/ 説明(2000 文字)の即時 feedback
- **関係者ピッカー**: + 追加ボタン展開で検索可能なテナント内ユーザー候補リスト + 複数選択
- **空状態**: 関係者ゼロ件時のメッセージ
- **412 衝突表示**(`conflict-banner`):他者の編集を検知時、ヘッダに警告 + 再読込ボタン + 自分の編集内容を保持する選択肢(ADR-0012 楽観ロック衝突 UI の参考実装)
- **フォーカスリング**: アクセシビリティ強化、Tab 移動時の視認性

## 2. 却下案

### 案 A 縦スクロール型(Vertical Stack)

ファイル: [`option-a-vertical-stack.html`](./option-a-vertical-stack.html)

**Claude Design 出力 注釈から**:

- 強み:1 カラム縦積みで上から「基本情報 → メタ → 関係者 → 監査メタ → 操作」と読み下せる。S-03 ダッシュボードのセクションカード哲学をそのまま継承し、タブレット縦・狭幅でも崩れない。実装が最も素直。
- 弱み:PC の横幅を使い切れず、メタ項目を見るのにスクロールが要る。説明とメタを同時に見比べづらく、情報密度は 3 案中もっとも低い。
- 想定読者:1 件のタスクを腰を据えて読む Member。タブレット併用や通知メールリンクからの単票閲覧が多いユーザー。

**却下の理由**(本書):

- 「読む動線」では適合するが、「PC 効率も取りたい」「体験差は許容」の追加判断軸で案 B に劣る。
- ドロワー / フルページのシームレス性は最強だが、user は「シームレス性より PC 効率」を選んだ。

**Phase 2 で再評価する価値はある**: タブレット縦運用や狭幅利用が増えた場合の「シンプルビュー」として個人設定で復活する余地あり。

### 案 C メタデータグリッド型(Meta Grid / Notion 風)

ファイル: [`option-c-meta-grid.html`](./option-c-meta-grid.html)

**Claude Design 出力 注釈から**:

- 強み:全メタ項目をラベル + 値のタイルで一画面に敷き詰め、情報密度が 3 案中最大。各タイルにホバーで ✏ が出て認可可能な項目だけ即編集に入れる。Notion / Asana 経験者に直感的で、項目の俯瞰性が高い。
- 弱み:密度が高いぶん視覚的な強弱が付きにくく、「次に何をすべきか」の誘導は弱い。狭幅では 2 → 1 カラムに畳まれ密度の利点が薄れ、ドロワー(狭幅)とフルページで体験差が出やすい。
- 想定読者:項目を素早くスキャン・一括編集したいパワーユーザー、PM。多数のタスクを高速にさばく運用フェーズ向き。

**却下の理由**(本書):

- 「腰を据えて読む」動線(user 確定)に対して案 C は「スキャン + 一括編集」の方向で合致しない。
- 説明の表示優先度がメタ項目と同等になり、「説明を読み下す」体験が弱まる。
- 編集モード切替の明確性が低く、所有者と他ロールでの UI 差分(何が編集可能か)が一見でわかりにくい。

**Phase 2 で復活する余地**: 件数多運用フェーズで「コンパクトビュー」として個人設定で切替可能にする選択肢。Notion 風の一括編集動線が必要な PM/リーダー向け。

## 3. 開いている論点(別 Issue に引き継ぐ)

### #154 デザインシステム素案に引き継ぐ

1. **ドロワー狭幅時の 2 カラム → 1 カラムフォールバック規則**:
   - 折り返しのブレークポイント(例:幅 < 720px で縦積みに切替)。
   - フォールバック時のセクション順序(主情報 → メタ → 操作)。
2. **タイトル + visibility バッジの視覚バランス**:
   - 案 B では大きめタイトル + バッジがヘッダ。バッジサイズ・配置・色強度のトークン化。
3. **編集モード切替の遷移**:
   - 参照モード(`/tasks/{id}`)→ 編集モード(`/tasks/{id}/edit`)への遷移アニメーション(チラつき防止)。
   - 同一ドロワー内で in-place 切替するか、ドロワーを一旦閉じて再描画するか。
4. **ETag / If-Match 楽観ロック衝突表示**(ADR-0012):
   - ハイファイ版で `conflict-banner`(警告 + 再読込 + 自分の編集保持選択肢)を実装済。これをデザインシステムの正式コンポーネントに昇格させる際の規格化。
   - 他者編集内容の差分表示(どのフィールドが変わったか)を出すかの判断。
5. **削除確認ダイアログ**:
   - タスクタイトル表示 + 「削除する」入力で誤クリック防止 / シンプル「はい・いいえ」のどちらにするかの判断。
   - 削除後の遷移先(S-04 タスク一覧 or S-03 ダッシュボードに戻る)。
6. **関係者追加 UI**:
   - 「+ 追加」ボタンクリックでテナント内ユーザープルダウン(検索可能)を展開、複数選択。
7. **権限不可項目の見せ方**:
   - 非表示(削除ボタン、公開範囲セレクタ) vs グレーアウト + tooltip(編集ボタン、ステータスプルダウン)の使い分けルール。
8. **行レベル認可フラグの正規化**:
   - S-03 / S-04 / S-05 共通の `canFields` / `canStatus` / `canStakeholder` / `canVisibility` / `canDelete` 5 フラグを API レスポンスに含めるか、クライアント側で `currentUserId === task.ownerId` 判定するかの方針確定(S-04 と統合議論)。

## 4. 認可ポリシー整合確認

ロール × 画面 × visibility の網羅マトリクスは `docs/specs/ui/access-matrix.md`(#151 成果物、main マージ済み)を SSOT とする。本書は案 B の UI 表現が SSOT と矛盾しないことを以下で局所的に確認する。差分が生じた場合は access-matrix.md と基本設計書 §6.2.1 を優先する。

| 観点 | 仕様 | 案 B での扱い | OK |
|---|---|---|---|
| §6.2.1 参照 (V2) | 参照ルール不通過は 404(`VIEW_DENIED`、NIST AC-4) | URL 直接アクセス時にサーバ側 404 判定、クライアントはエラー画面表示 | ✓ |
| ADR-0005 編集可否 (V3) | 3 役割評価(所有者 = 全項目 / 担当者 = ステータス + 関係者のみ / その他 = 読み取り専用) | `canFields` / `canStatus` 等 5 フラグで「編集ボタン表示 / プルダウン活性」を動的判定、操作不可項目は `cell-locked` + tooltip | ✓ |
| 関係者編集 (V4) | 所有者 ∪ 担当者 | 「+ 追加」ボタン / 各 × アイコンを `canStakeholder` で表示制御 | ✓ |
| 公開範囲変更 | 所有者のみ | visibility ラジオを `canVisibility` で活性化制御 | ✓ |
| 論理削除 | 所有者のみ | 「削除」ボタンを `canDelete` で表示制御 + 確認ダイアログ | ✓ |
| SaaS Admin の扱い | S-05 / S-06 到達不可 | ロール切替セレクタには SaaS Admin 単独オプションを置かない | ✓ |
| Tenant Admin の業務タスク特権 | なし(ADR-0005) | ロール = Tenant Admin でも `canFields` / `canStatus` 等は所有者 / 担当者役割に基づく(`current.userId === task.ownerId` 等で判定) | ✓ |
| visibility バッジの常時表示 | 案 B ヘッダで表示 | タイトル横に常時表示 | ✓ |
| 楽観ロック (ADR-0012) | 412 Precondition Failed の処理 | ハイファイ版で `conflict-banner` を実装(警告 + 再読込 + 自分の編集保持選択肢)。詳細トランジション・他者編集内容の差分表示は #154 で正式化 | ○(参考実装あり、#154 で詳細詰め) |
| 監査ログ | S-05 では表示せず、Tenant Admin が `/api/audit-logs` で参照 | 本画面に履歴セクションを置かない | ✓ |
| 削除導線 | 所有者のみ、確認ダイアログ、論理削除 | ボタン表示制御 + 確認ダイアログ + `DELETE /api/tasks/{id}` 呼び出し | ✓ |
