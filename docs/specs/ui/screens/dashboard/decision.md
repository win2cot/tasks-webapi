# S-03 ダッシュボード 方向性決定(Issue #152)

Version 1.0 / 2026-06-01 / 親 Issue #152 / #149

## 0. 本書の位置付け

- 本書は Issue #152 のサブ画面 **S-03 ダッシュボード** について、Claude Design に SSOT を投入して生成した 2〜3 案の HTML 案を比較し、方向性を 1 案に絞った決定記録。
- 採用案・却下案ともこのディレクトリ配下に HTML をそのまま残し、Sprint 1 以降の画面実装スプリントが本書だけで着手判断できるようにする。
- 本書は実装着手 Issue ではない(ハイファイモック完成ではなく「方向性確定」が目的)。

### 0.1 決定プロセス(2026-06-01)

1. SSOT(`docs/specs/ui/screen-flow.md` v1.1 §5)を Claude Design に投入、3 案(option-a/b/c)を生成。
2. 採用判断の前提として要件定義書から以下を確認:
   - メインユーザ = Member 5〜20 名/テナント(Tenant Admin は 1〜2 名で少数。要件 §2.3)
   - Tenant Admin に業務タスクの特権なし(ADR-0005、要件 §2.5)→ S-03 で俯瞰需要は薄い(俯瞰用は S-15)
   - デバイス = PC + タブレット(スマホは将来検討。要件 §3.6)
   - 1 ユーザ/日 のワークリスト典型件数 = **5〜15 件** を MVP 想定(要件定義書に明記なし、本決定で確定)
3. 個人切替の方針:**MVP は 1 案固定、Phase 2 で別ビュー(ボードビュー / コンパクトビュー等)を追加する余地は残す**。
4. 上記前提下で 3 案を 7 軸(振り返り常時可視 / 緊急対比 / 認知負荷 / 実装コスト / Member 主体ペルソナ整合 / PC+タブレット / F-17 数値の存在感)で比較し、**案 A 集約縦積みを採用**。

## 1. 採用案: **案 A 集約縦積み(Condensed Stack)**

ファイル: [`option-a-condensed-stack.html`](./option-a-condensed-stack.html)

### 採用の理由

1. **要件定義書のペルソナ・デバイス・件数前提との整合性が最も高い**
   - メインユーザ = Member 主体(5〜20 名/テナント)。S-03 は「自分のワークリストを上から消化する個人実務動線」になり、案 A の縦積み構造が自然。
   - Tenant Admin の業務タスク俯瞰需要は ADR-0005 で否定されているため、案 B の 2 カラム対比メリット(マネージャー俯瞰)は S-03 では発動しない。
   - PC + タブレット前提でレスポンシブ崩しが素直(案 B は xl 未満で 2 系統設計、案 A は単一カラム延長)。
   - MVP 件数 5〜15 件想定なら案 A の弱点(縦に長すぎ)は発動しない。
2. **screen-flow.md §5.1 の前提と最も整合**
   - SSOT §5.1 は「S-03 で **大体完結** する状態を作る」「S-04 への遷移を強制せずに当日業務と振り返りをスムーズにする」と明記。
   - 案 A は数値カードと 4 セクションをすべて 1 スクロール内に並べ、画面の上から下にスクロールするだけで「期限超過 → 今日 → これから → 振り返り」の認知フローが時系列で流れる。
3. **振り返り (4) の常時可視性が確保される**
   - 案 A は (4) 今日やったこと も他セクションと同じ重みで縦に並ぶため、業務終盤に振り返りを促す効果が画面構造として機能する(完了化のたびに (4) 先頭にカードが積み上がる視覚効果)。
   - 案 C は (4) を折り畳み既定にしているため、振り返り機能の効果が画面構造として薄まる。
4. **実装コストとレスポンシブ対応のシンプルさ**
   - 縦積み単一カラム構造なので、xl → lg → md → sm のブレークポイント崩しが容易(数値カードのみ折り返し制御で吸収)。Claude Design 出力でも `grid-template-columns` の単純な media query で対応済み。
   - 案 B は xl 以上で 2 カラム、xl 未満で縦積みフォールバックという 2 系統の設計が必要。レビュー・実装・E2E テストいずれもコストが増える。
5. **「期限超過 → 今日 → これから → 振り返り」の認知順序を視覚的に崩さない**
   - 業務遂行者の主流動線(緊急 → 当日 → 先 → 終わったもの)が縦軸の時間順と一致する。
   - 案 B は (3) これから と (4) 振り返り を同じ右カラム下段に置くため、時系列の流れが空間的に逆転して見える違和感がある。

### 採用案の主な構成要素(Claude Design 出力から)

- **トップナビ**: ブランド + テナント chip + ロール切替セレクタ + ユーザアバター。ロール切替で `body.role-admin` クラスが切り替わり、`.admin-only` 要素群(サイドナビの S-15 / S-08 リンクとロール注釈)が一括表示制御される。
- **サイドナビ**: ダッシュボード(active) / タスク一覧 / Tenant Admin 限定の「運営者ダッシュボード(S-15)」「ユーザー管理(S-08)」/ プロフィール / 通知設定。
- **数値カード 5 枚**(`.stat-grid`、3 列 + 1.3fr × 2 のグリッド、md 以下で 2 列、sm で 1 列):
  - 当日の期限件数(オレンジ強調)
  - 期限超過(未完了)件数(赤強調)
  - 本日完了件数(緑強調)
  - 自分の進捗(円グラフ + 凡例 + 完了/進行中/未着手の件数)
  - 優先度別件数(高・中・低のバーチャート)
- **4 セクション**(`#dashboardSections` に動的レンダリング):screen-flow.md §5.1 の条件・ソート順をそのまま視覚化。各セクションヘッダにアイコン色帯(赤/橙/青/緑)・件数・ソート順テキスト。
- **行内編集**: `editable` クラスで点線下線 + hover で primary 色強調。`cell-locked` で操作不可セルをグレー化 + 鍵アイコン + tooltip で理由表示(ADR-0005 由来)。タイトル・期限・担当者・優先度・ステータスがインライン編集、説明はポップオーバー型。
- **visibility バッジ**: TENANT(緑)/ STAKEHOLDERS(青)/ PRIVATE(紫)の 3 色 + アイコン。各行に必ず 1 バッジ + 凡例 popover。
- **+ 新規ボタン → 右ドロワー**(`#newTaskDrawer`、Bootstrap Offcanvas):タイトル / 説明 / 期限 / 担当者 / 優先度 / 公開範囲 / 関係者の 7 フィールド。URL = `/tasks/new` の同期は実装時に History API で別途。
- **詳細ドロワー**(`#detailDrawer`):行クリックで開き、ADR-0005 の 3 役割評価で編集ボタン / 削除ボタン / visibility セレクタ / 関係者追加ボタンの活性度を動的判定。各フィールドに `${lock(canX, "所有者")}` で操作可否ヒントを表示。

### Cowork たたき台との関係

このディレクトリには `cowork-draft-*` prefix で Cowork が事前に作った 3 案 + decision draft が残っている。これらは Claude Design 投入時の評価軸のたたき台として利用したもので、commit はせず参考資料として残置する選択肢もあるが、本 PR では **commit せず削除予定**(`git status` 上は untracked)。詳細は §6 参照。

## 2. 却下案

### 案 B 2 カラム対比(Two-Column)

ファイル: [`option-b-two-column.html`](./option-b-two-column.html)

**Claude Design 出力 注釈から**:
- 強み:左=緊急 / 右=予定 + 振り返り で「今すぐ」と「先の予定」を横並びで対比でき、スクロール量が約半分。緊急タスクが常に左視野に残る。俯瞰性が高い。
- 弱み:横幅を要し、各カラムが狭まるため行内編集の列がやや窮屈。狭い画面では結局 1 カラムに折り返るため案 A に近づく。左右の件数バランスが崩れると一方が間延びする。
- 想定読者:緊急対応と先々の段取りを同時に見たいリードやマネージャー寄りの利用者。

**却下の理由**(本書):
- (3) これから と (4) 振り返り を同じ右カラム上下に置くため、**時系列の流れが空間で逆転**(右上=未来、右下=過去)。
- xl 以上の幅が前提で、xl 未満で 1 カラムに崩れる際にどちらを上に持ってくるかの判断が必要。「マネージャー俯瞰モード」として個人設定で切替可能にすれば残せる選択肢だが、MVP では複雑度を持ち込まない。
- 案 A に比べて実装・テスト・レスポンシブ調整のコストが約 1.5〜2 倍。

**Phase 2 で再評価する価値はある**: ワイドモニタ × マネージャー向けの「俯瞰ビュー」として個人設定で切り替え可能にする案は残せる。

### 案 C アコーディオン折り畳み(Accordion)

ファイル: [`option-c-accordion.html`](./option-c-accordion.html)

**Claude Design 出力 注釈から**:
- 強み:4 セクションを折り畳み可能にし、デフォルトは (1) 期限超過・(2) 今日やること のみ展開。視線が「今日処理すべきこと」に集中し認知負荷が最も低い。件数バッジで畳んだまま残量を把握できる。
- 弱み:(3) これから・(4) 振り返り が初期状態で隠れるため、先の予定や達成感を見るには 1 クリック必要。開閉状態を記憶しないと毎回畳む手間が出る(要設定保持)。
- 想定読者:タスク件数が多く一覧が長くなりがちなチームや、まず緊急分だけに集中したい利用者。

**却下の理由**(本書):
- (4) 振り返り を畳み既定にすると、screen-flow.md §5.1 が掲げる「振り返り機能」の効果が薄れる。
- 完了化のたびに (4) を自動展開するかの決定が必要になり、UX 仕様が肥大化する。
- アコーディオン操作の手間が「1 画面で完結」原則と矛盾(クリックを 1 段増やす)。
- 開閉状態の永続化(localStorage 等)を MVP 範囲外に追加する必要がある。

**Phase 2 で復活する余地**: 件数が極端に増えた場合(例:期限超過 50 件以上)の (1) セクション内の「もっと見る」展開トリガとして類似の UI が必要になる可能性がある。その時に再評価する。

## 3. 開いている論点(別 Issue に引き継ぐ)

### #154 デザインシステム素案に引き継ぐ

1. **行内編集セルの UI 詳細**:
   - クリックでテキスト入力に切り替わるアニメーション(タイトル / 説明セル)。
   - ステータス完了化の楽観的 UI 更新と、(2) → (4) セクション間移動アニメーション(スライドアウト + スライドイン or フェード)。
   - バリデーション失敗時のセル内エラー表示。
   - `cell-locked` セルのツールチップ文言(ADR-0005 の理由を簡潔に表示する規約)。
2. **visibility バッジのアイコン・色トークン**:
   - Claude Design 出力では `vis-TENANT` = 緑系、`vis-STAKEHOLDERS` = 青系、`vis-PRIVATE` = 紫系。デザインシステム素案で正式トークン(`color-vis-tenant` 等)を定義。
3. **数値カードのレスポンシブ規則**:
   - 5 枚並びを md 以下で 2 列、sm で 1 列に折り返す方式が出力に組み込まれている。デザインシステム素案で正式採用するかを判断。
   - 優先度別バーチャートと「自分の進捗」円グラフを 1 カードに統合する案も検討余地あり(数値カードのみ 4 枚にする選択肢)。
4. **アコーディオン挙動の予備設計**:案 C で却下したが、「もっと見る」展開トリガとして似た UI が将来必要になる可能性。基本コンポーネントとして登録する候補。
5. **完了化アニメーションの正式仕様**(セクション間移動)。
6. **行内編集中の「他者編集衝突」表示**(楽観ロック ADR-0012 で 412 が返った時の UI)。
7. **凡例 popover の常設化判断**:Claude Design 出力では公開範囲の凡例を popover で提供。デザインシステム素案で「初回オンボーディング表示」「設定で常時表示」等の方針を確定。
8. **詳細ドロワーのフォーム設計**:Claude Design 出力の `detailBody` 動的レンダリングが参考実装になる(ADR-0005 の `canFields` / `canStatus` / `canStakeholder` / `canVisibility` / `canDelete` 5 フラグでフィールド活性を制御)。

### 別 Issue として独立起票候補(本書では結論を出さない)

- **件数が大量になった場合のセクション内ページング**: (1) 期限超過が 50 件以上等の場合の挙動。「もっと見る」で S-04 にジャンプ or セクション内ページング。
- **モバイル幅(< lg)レイアウトの正式仕様**: 案 A の縦積みは Claude Design 出力で自然に動作する(`@media(max-width:1200px)` で 2 列、`max-width:620px` で 1 列)。Sprint 1 以降の実装時に動作確認の上、正式仕様化。
- **「自分の進捗」円グラフ + 「優先度別」バーチャートの MVP 採否**: 数値カード 5 枚のうち後ろ 2 枚は集計コストが高め。MVP で削減するかは #153 で API スキーマ確定後に判断。
- **完了化時のセクション間移動アニメーション設計**: #154 に最終的に集約するが、UX 仕様としては別 Issue 起票も選択肢。
- **ロール切替セレクタの最終的扱い**: 本案では検証用にトップナビにロール切替セレクタを置いているが、本番 UI ではロールはサーバ側の認可情報で確定するため不要。MVP では削除し、開発環境のみ表示する仕組みを別 Issue で検討。
