# S-04 タスク一覧 方向性決定(Issue #152)

Version 1.0 / 2026-06-02 / 親 Issue #152 / #149

## 0. 本書の位置付け

- 本書は Issue #152 のサブ画面 **S-04 タスク一覧** について、Claude Design に SSOT を投入して生成した 3 案の HTML 案を比較し、方向性を 1 案に絞った決定記録。
- 採用案・却下案ともこのディレクトリ配下に HTML をそのまま残し、Sprint 1 以降の画面実装スプリントが本書だけで着手判断できるようにする。
- 本書は実装着手 Issue ではない(ハイファイモック完成ではなく「方向性確定」が目的)。

### 0.1 決定プロセス(2026-06-02)

1. SSOT(`docs/specs/ui/screen-flow.md` §5.2 / `docs/specs/基本設計書.md` §3.3.1 / `docs/specs/ui/access-matrix.md` / `api/openapi.yaml` v1.5.0 `listTasks`)+ S-03 採用案 HTML(`docs/specs/ui/screens/dashboard/option-a-condensed-stack.html`、デザインシステム継承元)を Claude Design に投入、3 案生成。
2. 採用判断の前提として S-03 議論時の前提を再確認:
   - メインユーザ = Member 主体(変わらず)
   - デバイス = PC + タブレット(変わらず)
   - 件数規模 = **S-04 ではテナント全体のフィルタ通過分なので S-03(個人視点)より多くなる**(本決定で確定)
   - 個人切替 = MVP 1 案固定、Phase 2 で別ビュー追加余地(変わらず)
3. ユーザ議論で以下を確認:
   - 期限切れ常時可視は望ましいが、件数バッジで補えるならタブ切替も完全 NG ではない
   - **日付ナビゲーションで遠い未来 / 月跨ぎへジャンプする頻度は高い**(週・月で見渡したい、計画者動線重視)
   - S-03 との視覚的統一性は「画面ごとのゴールに合わせた構造選択を許容」
   - 件数は S-04 では S-03 より自然に増える
4. 上記前提下で 3 案を 6 軸(日付ナビ高頻度 / 件数多前提 / 期限切れ常時可視 / S-03 構造統一 / タブレット縦対応 / 計画者動線)で比較し、**案 B 左サイドカレンダーを採用**。

## 1. 採用案: **案 B 左サイドカレンダー(Side Calendar)**

ファイル: [`option-b-side-calendar.html`](./option-b-side-calendar.html)

### 採用の理由

1. **日付ナビゲーション高頻度 × 計画者動線にハマる**
   - ユーザ議論で「週・月で見渡したい、計画者動線重視」が確定。案 B の月単位カレンダー + 件数バッジは「いつどれだけのタスクがあるか」を一望でき、遠い未来日や月跨ぎジャンプもクリック一発。
   - 案 A の日付ストリップは ±3 日程度の俯瞰しかなく、案 C はピッカー単独で月跨ぎが面倒。
2. **期限切れの常時可視性が確保される**
   - SSOT(screen-flow.md §5.2 / 基本設計書 §3.3.1)が定める「期限切れセクション常時表示」を案 B はそのまま満たす。
   - 案 C はタブで隠れる(件数バッジで補完可だが、SSOT 仕様の趣旨が薄まる)。
3. **件数多めの前提にスケールする**
   - S-04 はテナント内のフィルタ通過タスク全体を扱うため、件数は S-03(個人視点)より大きい。
   - 案 B は上部固定フィルタバー + 左カレンダー + 縦 2 セクションでフィルタリングと俯瞰が同時にできる。
4. **フィルタの存在を常に視認できる**
   - 案 B / 案 C は上部フィルタバーが常時展開で、絞り込み手段が常に見える。
   - 案 A は折り畳み既定で「絞り込み機能の存在に気づきにくい」(設計判断ポイント注釈)。

### 採用案の主な構成要素(Claude Design 出力から)

- **トップナビ**(S-03 継承): ブランド + テナント chip + ロール切替セレクタ + ユーザアバター。`body.role-admin` クラス連動の `.admin-only` 表示制御で Tenant Admin 限定サイドナビが切り替わる。
- **左メインサイドナビ**(S-03 継承): ダッシュボード / タスク一覧(active) / Tenant Admin 限定の S-15 / S-08 / プロフィール / 通知設定。
- **左サブサイドナビ(カレンダー)**: 月単位、選択日強調、日セル内に件数バッジ(期限切れ件数 + 当日件数の合算 or 別表示は #154 で確定)。月前後ボタン。
- **上部フィルタバー**(常時展開):
  - ステータス(複数選択)/ 所有者 / 担当者 / 優先度 / 公開範囲 / キーワード(`listTasks` parameters と一致)
  - 適用中フィルタ chip 表示(「ステータス=未着手 ×」等で個別解除可)
- **上部ソート選択**: 期限 / 優先度 / 作成日(昇順・降順)(`listTasks` `sort` parameter)
- **期限切れセクション**(常時表示、赤帯):未完了かつ期限 < 表示日のタスクを 50 件単位ページング。
- **選択日セクション**:選択日 = 期限のタスクを 50 件単位ページング。
- **行内編集**:`editable` クラス(点線下線 + hover で primary 強調) / `cell-locked` クラス(グレー + 鍵アイコン + tooltip)で ADR-0005 3 役割評価に基づくセル単位の出し分け。タイトル・期限・担当者・優先度・ステータスがインライン編集、説明はポップオーバー型。
- **visibility バッジ**: S-03 と同じ TENANT(緑)/ STAKEHOLDERS(青)/ PRIVATE(紫)+ アイコン。各行に必ず 1 バッジ + 凡例 popover。
- **+ 新規ボタン → 右ドロワー**(`#newTaskDrawer`、URL = `/tasks/new` 同期、初期値: 期限 = 選択日):S-03 と同じ。
- **詳細ドロワー**(`#detailDrawer`、URL = `/tasks/{id}` 同期):S-03 と同じ。ADR-0005 の `canFields` / `canStatus` / `canStakeholder` / `canVisibility` / `canDelete` 5 フラグでフィールド活性を動的判定。

## 2. 却下案

### 案 A 日付ストリップ(Date Strip)

ファイル: [`option-a-date-strip.html`](./option-a-date-strip.html)

**Claude Design 出力 注釈から**:
- 強み:横方向の日付ストリップで前後数日を件数つきで一望、ワンタップで日移動。フィルタは初期折り畳みで普段は邪魔にならず、画面の主役を常に「期限切れ + 選択日」の 2 テーブルに保てる。S-03 の縦積み哲学をそのまま継承。
- 弱み:日付ストリップは ±3 日程度しか見えず、月をまたぐ俯瞰や遠い未来日への移動はピッカー頼み。フィルタが折り畳まれているため「絞り込み機能の存在」に気づきにくい(chip 表示で補う)。
- 想定読者:当日〜数日のワークリストを上から消化する Member。日次運用で「今日と前後」を素早く行き来したい個人実務者。

**却下の理由**(本書):
- 「日付ナビゲーション高頻度・計画者動線重視」のユーザ要件と合わない(±3 日 chip + ピッカーでは月跨ぎが体感に出る)。
- S-03 視覚統一の利点はあるが、user は「画面ごとのゴールに合わせた構造選択を許容」と回答済み。

**Phase 2 で再評価する価値はある**: 件数が極端に少ない運用 / 日次運用のみで月跨ぎがほぼ発生しない使い方には案 A が最もシンプル。個人設定で「シンプルビュー」として復活する余地あり。

### 案 C タブ式単一テーブル(Tabbed)

ファイル: [`option-c-tabs.html`](./option-c-tabs.html)

**Claude Design 出力 注釈から**:
- 強み:期限超過と選択日をタブで切替、下半分を単一テーブルが使うためスクロール量が最小。フィルタを横一列に平めて常時表示し、画面密度が最も高い。件数が多いテナント・ページング多用前提の運用に強い。
- 弱み:期限超過と選択日を同時に見られず、タブ切替の手間が入る。「期限切れを常に目に入れる」という S-03/S-04 の設計意図はバッジ件数で補う必要がある。
- 想定読者:1 セクションあたりの件数が多く、一度に多くのタスクを走査・処理したいパワーユーザーや、画面高さに制約のあるタブレット利用者。

**却下の理由**(本書):
- SSOT(screen-flow.md §5.2 / 基本設計書 §3.3.1)が定める「期限切れセクション常時表示」が構造的に満たされない(バッジ件数で補完可だが、表示するための 1 クリックが要る)。
- 「日付ナビゲーション高頻度・計画者動線」の要件に対して、ピッカー単独では月跨ぎ俯瞰が弱い。
- 案 B と比べたとき、案 C のメリット(画面密度最大、タブレット縦対応)は MVP 件数規模(テナント 1 日数十件以下想定)では決定的ではない。

**Phase 2 で復活する余地**: 件数が極端に増えた場合(例:1 日 200 件以上)に「コンパクトビュー」として個人設定で切替可能にする選択肢。

## 3. 開いている論点(別 Issue に引き継ぐ)

### #154 デザインシステム素案に引き継ぐ

1. **左サイドカレンダーコンポーネント**(本案で新出):
   - 月単位カレンダーの正式コンポーネント設計、件数バッジ(期限切れ件数 / 当日件数 / 合算のどれを表示するか)。
   - 月前後ナビ、年跨ぎナビ(例:2 年以上先に飛ぶ場合の UI)。
   - 「今日に戻る」ボタンの位置(カレンダーヘッダ or 上部ピッカー)。
2. **タブレット縦レイアウトの正式仕様**:
   - 案 B の弱み(左カレンダーが下に回り、一覧との同時視認が崩れる)を吸収する規則。
   - 折り畳み可能(コンパクトモード時のみ表示 / 常時非表示)等のオプションを検討。
3. **フィルタ chip の常時表示ルール**:
   - 「適用中: ステータス=未着手 ×」等の chip 表示、クリア導線(個別 × / 全クリア / 保存)。
   - フィルタ保存(よく使う絞り込みのプリセット)は Phase 2 候補。
4. **期限切れセクションの視覚的優先度**:
   - 件数が多時の折り畳み(50 件以降を「もっと見る」化、件数バッジを常時表示)。
   - 期限切れの色強度(赤帯)が他要素を圧迫しないバランス。
5. **行内編集セルの UI 詳細**(S-03 と共通):
   - クリック → テキスト入力切替アニメーション、確定キー、バリデーション失敗時のセル内エラー表示、`cell-locked` セルのツールチップ文言。
6. **編集衝突表示**(ADR-0012 楽観ロック、412 Precondition Failed)(S-03 と共通)。
7. **詳細ドロワーのフォーム設計**(S-03 と共通、`canFields` / `canStatus` / `canStakeholder` / `canVisibility` / `canDelete` 5 フラグ実装)。

## 4. 認可ポリシー整合確認

ロール × 画面 × visibility の網羅マトリクスは `docs/specs/ui/access-matrix.md`(#151 成果物、main マージ済み)を SSOT とする。本書は案 B の UI 表現が SSOT と矛盾しないことを以下で局所的に確認する。差分が生じた場合は access-matrix.md と基本設計書 §6.2.1 を優先する。

| 観点 | 仕様 | 案 B での扱い | OK |
|---|---|---|---|
| §6.2.1 認可フィルタ通過 (V1) | リスト表示前に Hibernate Filter + 認可フィルタ適用 | サーバ側適用前提。クライアント側は通過後のデータのみを表示 | ✓ |
| §6.2.3 参照不可 → 404 (V2) | URL 直接アクセス時 | S-04 自身は一覧画面で 404 発生はリスト除外で吸収。詳細ドロワー遷移時に S-05 側で 404 判定 | ✓ |
| ADR-0005 編集可否 (V3) | 3 役割評価(所有者 / 担当者 / その他) | `canFields` / `canStatus` 等 5 フラグで `cell-locked` クラスに反映、ツールチップで理由表示 | ✓ |
| 関係者編集 (V4) | 所有者 ∪ 担当者、ドロワー内のみ | 行内編集対象外。新規 / 編集ドロワーで管理 | ✓ |
| 一覧集計 | 認可通過後のタスクのみ表示 | サーバ側集計前提。Tenant Admin 特権なし(ADR-0005)| ✓ |
| 左カレンダー件数バッジ | 認可通過後のタスクのみカウント | client が `listTasks`(認可済タスクのみ返す)を集計するため認可済件数のみ反映(#356 案 B 確定) | ✓ |
| SaaS Admin の扱い | S-04 到達不可 | ロール切替セレクタには SaaS Admin 単独オプションを置かない(Member / Tenant Admin のみ) | ✓ |
| Tenant Admin の S-15 動線 | サイドナビに S-15 リンク追加のみ、業務タスクへの特権なし | `.admin-only` セクションで `body.role-admin` クラスに連動して表示。行内編集 / ドロワー編集の認可判定は ADR-0005 と同じ | ✓ |
| visibility バッジの常時表示 (V1) | 各行に表示 | `.vis-badge` を `<table.tasks>` の `col-vis` 列に必ず描画 | ✓ |
| 楽観ロック (ADR-0012) | 412 Precondition Failed の処理 | Claude Design 出力ではエラー処理未実装。#154 で実装方針確定 | △(別 Issue) |
