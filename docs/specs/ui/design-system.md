## 1. 本書の位置付け

本書は Bootstrap 5 を基盤に、本プロジェクト固有のデザイン方針(配色 / タイポグラフィ / 主要コンポーネントのバリエーション)を **素案** として文書化したものである。完成版のデザインガイドではなく、「Sprint 1 以降の画面実装が迷わない最低限」を目的とする。

- **デザイントークンの SSOT**: 本書のトークン値は #152 Claude Design 検証で採用した 4 画面案の HTML 内 `:root` CSS 変数を正本とする。すなわち `docs/specs/ui/screens/dashboard/option-a-condensed-stack.html` / `screens/task-list/option-b-side-calendar.html` / `screens/task-detail/option-b-two-column.html` / `screens/tenant-admin/option-b-admin-console.html` の 4 ファイル。値が食い違う場合は採用案 HTML を優先し、本書を追従させる。
- **画面構造の SSOT**: `docs/specs/ui/screen-flow.md`(画面遷移 / サイトマップ)、`docs/specs/ui/screens/*/decision.md`(各画面の方向性決定)。
- **認可とデータの SSOT**: `docs/specs/ui/access-matrix.md`(ロール × 画面 × visibility)、`api/openapi.yaml`(API 契約 / enum)、`docs/specs/基本設計書.md`。
- **本書が扱わないもの**: ハイファイモック(ピクセル単位の仕上げ)、認証 / ログイン画面(Keycloak Hosted UI 前提)、項目別の入力バリデーション仕様、API スキーマ。これらは別 Sprint / 別 Issue の領分とする。

## 2. 設計原則と前提

### 2.1 フロントエンドスタック

- HTML5 / CSS3 / 素の JavaScript / **Bootstrap 5**。React 等の SPA フレームワークは MVP では採用しない。
- 再利用 UI 部品(行内編集セル / ドロワー / ポップオーバー / バッジ等)は **light DOM Custom Elements**(`app-` プレフィックス、Shadow DOM 不使用)で実装する(ADR-0024)。スタイルは部品内に持ち込まず、Bootstrap クラス + 本書のトークン(§2.4)を参照する。静的マークアップは HTML Templates(`<template>`)の雛形 + clone で展開し、データは `textContent` / 属性で反映する(詳細は ADR-0024)。
- アイコンは Bootstrap Icons(`bi-*`)を採用案 HTML が前提にしている。
- Web フォントは `Noto Sans JP` を第一候補とし、未取得時は system-ui 系へフォールバックする。

### 2.2 想定ユーザとデバイス

- メインユーザは Member(5〜20 名 / テナント)。Tenant Admin は少数。SaaS Admin は数名・専任(要件 §2.3)。
- 対応デバイスは **PC + タブレット**。スマートフォン最適化は Phase 2 で検討する。
- 1 画面の典型タスク件数は MVP では数十件規模を想定(#152 決定記録準拠)。

### 2.3 マルチテナント SaaS としての方針

- 将来のテナント別テーマ可変性(ロゴ・アクセントカラー)を見越し、可変箇所と固定箇所を §9 で明示する。
- 業務動線(Member / Tenant Admin)と SaaS Admin 運営動線は **視覚的に別空間** として分離する(§8)。

### 2.4 トークン駆動の徹底

色・余白・角丸・影は直値をコンポーネントに直書きせず、CSS 変数(`--app-*` / `--bs-*`)経由で参照する。これによりテナント別テーマ差し替えと一貫性維持を両立する。

## 3. カラーパレット

### 3.1 基盤色(業務動線)

採用案 HTML の `:root` から確定。

| トークン | 値 | 用途 |
|---|---|---|
| `--app-bg` | `#f4f5f7` | ページ背景(薄いグレー) |
| `--app-surface` | `#ffffff` | カード / テーブル / ドロワー等の面 |
| `--app-border` | `#e5e7eb` | 罫線 / 区切り |
| `--app-ink` | `#1f2733` | 本文テキスト(基準文字色) |
| `--app-muted` | `#6b7280` | 補助テキスト / プレースホルダ |

### 3.2 プライマリ / アクセント

| トークン | 値 | 用途 |
|---|---|---|
| `--app-primary` | `#3b5bdb` | 主要アクション / アクティブ状態 / リンク |
| `--app-primary-d` | `#2f49b0` | プライマリの hover / pressed(濃色) |

Bootstrap の `--bs-primary` 系をこの値で上書きし、`btn-primary` / `text-primary` / `link-primary` 等のユーティリティを自動的に本プロジェクト色へ揃える。セカンダリは Bootstrap 既定のグレー(`--bs-secondary`)を踏襲し、本プロジェクト固有色は設けない。

### 3.3 セマンティックカラー(状態色)

タスクの時間状態を表す色。**意味を担うため固定**とし、テナント別テーマでも変更しない(§9.2)。

| トークン | 値 | 意味 | 主な使用箇所 |
|---|---|---|---|
| `--c-overdue` | `#e03131` | 期限超過(赤) | 期限切れ帯 / バッジ / カレンダー赤チント |
| `--c-today` | `#e8830c` | 今日(オレンジ) | 当日セクション / 今日セル枠 |
| `--c-soon` | `#1c7ed6` | 近接 / 予定(青) | これから(明日以降)/ 情報トースト |
| `--c-done` | `#2f9e44` | 完了(緑) | 完了タスク / 成功トースト |

Bootstrap のセマンティクス対応: `danger`≈`--c-overdue`、`warning`≈`--c-today`、`info`≈`--c-soon`、`success`≈`--c-done`。アラート / トーストはこの対応で Bootstrap コンポーネントに割り当てる。

### 3.4 フォーカスリングと影

| トークン | 値 | 用途 |
|---|---|---|
| `--ring` | `0 0 0 3px rgba(59,91,219,.22)` | キーボードフォーカス時の外周リング |
| `--shadow-sm` | `0 1px 2px rgba(31,39,51,.06)` | カード / 行の控えめな浮き |
| `--shadow-md` | `0 6px 22px rgba(31,39,51,.09)` | ドロワー / ポップオーバー / ドロップダウン |

### 3.5 コントラストの担保

`--app-primary`(`#3b5bdb`)上の白文字、`--app-ink` の白背景文字はいずれも WCAG AA(4.5:1)を満たす。`--app-muted`(`#6b7280`)を本文サイズで多用する場合は背景とのコントラストを確認する(§7.1)。

## 4. タイポグラフィ

### 4.1 フォントファミリ

```css
--bs-body-font-family: "Noto Sans JP", system-ui, -apple-system, "Segoe UI", sans-serif;
```

### 4.2 タイプスケール(素案)

採用案 HTML は情報密度が高く(11〜16px 中心、KPI 数値のみ 23px)、Bootstrap 既定(16px 基準)よりやや小さめである。これを踏まえた階層を素案として定義する。実装時は `rem` で表現し、ルート 16px を基準とする。

| 役割 | サイズ | ウェイト | 備考 |
|---|---|---|---|
| KPI 数値(統計カード) | 1.45rem(約 23px) | 700 | ダッシュボードの数値強調 |
| ページ見出し(h1) | 1.25rem(20px) | 700 | 画面タイトル |
| セクション見出し(h2) | 1.0625rem(17px) | 700 | セクション帯 |
| サブ見出し(h3) | 0.9375rem(15px) | 600 | カード内見出し |
| 本文 | 0.875rem(14px) | 400 | 既定の文字サイズ |
| 補助テキスト | 0.8125rem(13px) | 400 / 500 | メタ情報 / 補足 |
| ラベル / バッジ | 0.6875rem(11px) | 500 / 600 / 700 | バッジ / chip / 表ヘッダ |

行間は本文 1.45 前後、長文の説明領域は 1.9 程度まで広げて可読性を確保する(採用案準拠)。ウェイトは 400 / 500 / 600 / 700 の 4 段階に限定する。

### 4.3 適用方針

見出しの色は `--app-ink`、補助テキストは `--app-muted` を使う。Bootstrap の `fs-*` / `fw-*` / `small` / `text-muted` ユーティリティを基本とし、固有スケールが必要な箇所のみコンポーネント CSS で定義する。

## 5. スペーシングと角丸

### 5.1 スペーシング基準

基準単位は **4px**。Bootstrap 5 のスペーシングスケール(`0`=0 / `1`=.25rem / `2`=.5rem / `3`=1rem / `4`=1.5rem / `5`=3rem)を踏襲し、`m-*` / `p-*` / `gap-*` / `g-*` ユーティリティで表現する。固有値の直書きは避け、近いスケール値へ寄せる。

| 文脈 | 目安 | Bootstrap 対応 |
|---|---|---|
| バッジ / chip の内側 | 3〜4px × 8〜9px | カスタム(`py-1 px-2` 近似) |
| ボタン(sm)内側 | 4〜6px × 10〜12px | `btn-sm` |
| カード内側 | 12〜16px | `p-3` |
| セクション間 | 16〜24px | `mb-3` / `mb-4` |

### 5.2 角丸

| トークン | 値 | 用途 |
|---|---|---|
| 角丸 sm | 6px | バッジ / 入力欄 / 小ボタン |
| 角丸 md | 8px | カード / テーブルセル / カレンダーセル |
| 角丸 lg | 12px | 大きめのパネル / モーダル |
| pill | 999px | ステータスピル / カウントバッジ |
| circle | 50% | アバター |

## 6. 主要コンポーネント方針

すべて Bootstrap 5 のコンポーネントを基礎にし、上記トークンで色と形を上書きする方針。再利用部品の実装単位は light DOM Custom Elements とする(§2.1 / ADR-0024)。

### 6.1 ボタン

- 主要操作は `btn-primary`(`--app-primary` / hover `--app-primary-d`)。副次操作は `btn-outline-primary` / `btn-outline-secondary`。
- 密度を保つため画面内のボタンは原則 `btn-sm` を基本サイズとする。
- 破壊的操作(削除 / テナント停止)は `btn-danger` 系 + 確認モーダル(§6.5)を必須とする。
- アイコン + ラベルの併記を基本とし、アイコン単独ボタンには `aria-label` / `title` を付ける。

### 6.2 フォームと行内編集

- 入力は `form-control` / `form-select`。フォーカス時は `--ring` を適用する。
- **行内編集(一覧 / 詳細)**: 編集可能セルは `.editable`(点線下線 + hover で primary 強調)、編集不可セルは `.cell-locked`(グレー + 鍵アイコン + ツールチップで理由表示)。活性判定は ADR-0005 の 5 フラグ(`canFields` / `canStatus` / `canStakeholder` / `canVisibility` / `canDelete`)に基づく。
- 行内編集の確定は Enter、取消は Esc。バリデーション失敗はセル内にインラインエラー表示(§6.10)。
- 長文の説明はポップオーバー型エディタで編集する(一覧の行高を崩さないため)。

### 6.3 テーブル(一覧)

- タスク一覧は密度重視の表形式。各行に visibility バッジ列を必ず置く(access-matrix の V1)。
- ソートは `listTasks` の `sort`(`dueDate` / `priority` / `createdAt` / `updatedAt` / `title`)に対応する列ヘッダで表現する。
- ページングは 50 件単位(`size` 既定 50、最大 100)。件数が多いセクションは「もっと見る」/ ページャで分割する。

### 6.4 バッジ / ピル

状態を色 + アイコン + ラベルの三重符号化で表す(色覚多様性対応、§7.3)。enum 値は `api/openapi.yaml` を正本とする。

| バッジ | enum 値 | 色 | 例 |
|---|---|---|---|
| ステータス(`.st-badge`) | `NOT_STARTED` / `IN_PROGRESS` / `DONE` / `ON_HOLD` | 既定グレー / `--c-soon` / `--c-done` / `--c-today` | 未着手 / 進行中 / 完了 / 保留 |
| 優先度(`.pri-badge`) | `HIGH` / `MEDIUM` / `LOW` | `--c-overdue` 系 / `--c-today` 系 / グレー | 高 / 中 / 低 |
| 公開範囲(`.vis-badge`) | `TENANT` / `STAKEHOLDERS` / `PRIVATE` | 緑 / 青 / 紫(+ アイコン `globe2` / `people-fill` / `lock-fill`) | テナント / 関係者 / 非公開 |
| カウント(pill) | 数値 | グレー / 赤(超過時) | 件数バッジ(§6.8) |

ステータスバッジは行内でドロップダウン化し、その場で状態変更できる(`updateTaskStatus`、権限がある場合のみ活性)。

### 6.5 モーダル / 確認ダイアログ

- 破壊的操作は Bootstrap モーダルで二段確認する。本文に影響範囲を明記する(例: 「このテナントを停止すると所属ユーザー全員がログインできなくなります」)。
- 確認ボタンは危険操作色、既定フォーカスは取消側に置き誤操作を防ぐ。

### 6.6 トースト

- 操作結果は右上トーストで通知する。成功 = `--c-done`、エラー = `--c-overdue`、情報 = `--c-soon`。
- 自動消滅は成功・情報のみ。エラーは手動クローズを基本とし、再試行導線があれば併記する。

### 6.7 ナビゲーション

- **トップナビ**: ブランド + テナント chip + ロール切替セレクタ + ユーザアバター。`body.role-admin` クラスに連動して `.admin-only` 要素(Tenant Admin 限定リンク等)を一括表示制御する。
- **サイドナビ**: ダッシュボード / タスク一覧 / プロフィール / 通知設定。Tenant Admin のみ S-15(運営者ダッシュボード)/ S-08(ユーザー管理)を `.admin-only` で追加表示する。アクティブ項目は primary で強調する。
- ロール切替セレクタは業務動線(Member / Tenant Admin)のみを対象とし、SaaS Admin 単独オプションは置かない(SaaS Admin は別空間、§8)。

### 6.8 カレンダー(タスク一覧の左サイド)

- 月単位カレンダー。`.cal-cell` は選択日強調(`is-active`)、今日枠(`is-today`、`--c-today`)、過去日で未完了が残るセルの赤チント(`has-overdue`)を持つ。月前後ナビ + 「今日に戻る」を備える。
- 各セルのカウントバッジは pill 形状。表示方針は §6.8.1 で確定する。
- カレンダーのバッジ件数は `listTasks`(認可済タスクのみ返す)を当月分集計したクライアント派生値とする(#356 / field-mapping.md §2.3、専用 API は設けない)。

#### 6.8.1 カウントバッジ表示方針(本書で確定)

task-list 決定記録 §3 が #154 に引き継いだ宿題「カレンダー件数バッジは『期限切れ件数 / 当日件数 / 合算』のどれを表示するか」を、本書で次のとおり確定する。

- **採用: 各セル = その日が期限の未完了タスク件数(当日件数モデル)**。`countForDate(iso)` = `due === iso && status !== DONE` の件数を採用案 HTML どおり踏襲する。
- 過去日で未完了が残るセルは赤チント(`has-overdue`)で識別し、期限超過であることを色で示す。
- 期限超過の **総量** はカレンダーセルに集約せず、常時表示の「期限切れセクション」見出し件数(`.sec-count`)で示す。
- **不採用: 今日セルへの合算**。期限超過タスクを今日セルに足し込むと、元の期限日セルと今日セルで二重カウントされ、「セル = その日が期限のタスク」という意味論が崩れるため。

採用理由: (1) 期限超過の総量はもともと選択日に依存せず期限切れセクションで常時集約済み、(2) 二重カウントを避けられる、(3) カレンダーの意味論が一貫する、(4) 採用案 HTML の実装をそのまま使え追加実装が不要。将来この方針を変える場合は `countForDate` とバッジ描画の 1 箇所変更で済む。

### 6.9 ドロワー(詳細 / 新規)

- タスク詳細 / 新規は右ドロワー(Bootstrap offcanvas)で開く。URL を同期(`/tasks/{id}` / `/tasks/new` / `/tasks/{id}/edit`)し、URL 直接アクセス時はフルページにフォールバックする(task-detail 決定記録)。
- 詳細は 2 カラム(左 = 主情報、右 = メタ + 操作)を基本とし、狭幅では縦積みにフォールバックする。

### 6.10 状態 / エラーのインライン表現

- 行内編集の失敗はセル直下に簡潔なエラーを出す。
- 楽観ロック競合(ADR-0012、412 Precondition Failed)は「他のユーザーが更新しました。再読み込みしてください」系のメッセージ + 再読込導線で扱う。具体 UI は実装 Issue で詰める(§10)。

### 6.11 状態表現(loading / empty / error)

画面・領域の 3 状態を一貫して扱う。具体的な見た目は実装 Issue で詰めるが、方針は本書で固定する。

- **loading(読込中)**: 初回ロードや再取得はスケルトン(テーブル行 / カードのプレースホルダ)で枠を保持し、レイアウトシフトを防ぐ。送信中のボタンは `spinner-border` + disabled とし二重送信を防ぐ。小領域の再取得はスピナー単体でもよい。
- **empty(空)**: データ 0 件は空状態を明示する。アイコン + 簡潔な説明 + 主要アクション導線をセットで出す(例: ダッシュボードの「今日やること」が 0 件なら肯定的な完了メッセージ、タスク一覧の絞り込み 0 件なら「条件に一致するタスクはありません」+ フィルタ解除導線)。空白のまま放置しない。
- **error(失敗)**: 取得失敗は領域 / ページ単位のエラーバナー + 再試行ボタン。操作失敗はトースト(§6.6)。フォーム / 行内編集の項目エラーはインライン(§6.10)。認可・不在は `access-matrix.md` の応答ポリシー(401 / 403 / 404)に従い、404 はリソースの存在を漏らさない文言にする。

## 7. アクセシビリティ方針

### 7.1 コントラスト最低基準

- 本文テキストと背景は WCAG 2.1 AA の **4.5:1** 以上、大きい文字(18pt ≈ 24px 以上、または太字 14pt ≈ 18.5px 以上)と UI 部品 / グラフィック境界は **3:1** 以上を満たす。
- `--app-muted` を本文サイズで使う箇所は背景とのコントラストを実測で確認する。

### 7.2 キーボード操作

- すべての操作はキーボードのみで完結できることを目標とする。フォーカス可能要素には `--ring` による可視フォーカスを必ず与える(`:focus-visible`)。
- Esc でドロワー / モーダル / ドロップダウンを閉じ、Enter で行内編集を確定する。タブ順は視覚順と一致させる。
- ページ先頭にメインコンテンツへのスキップリンクを置く。

### 7.3 色だけに依存しない

- ステータス / 優先度 / 公開範囲はアイコンとラベルを併用し、色のみで意味を判別させない。
- 期限超過は色(赤)に加えセクション見出し / アイコンでも示す。

### 7.4 セマンティクスと支援技術

- Bootstrap コンポーネントの ARIA 属性を尊重する。アイコン単独要素には代替テキスト、動的更新領域(トースト等)には `aria-live` を付ける。

## 8. SaaS Admin コンソールの別空間テーマ

SaaS Admin 運営画面(S-13 / S-14 等)は業務動線と認可境界が異なるため、配色から別空間として分離する(tenant-admin 決定記録、採用案 `option-b-admin-console.html`)。

| トークン | 値 | 用途 |
|---|---|---|
| `--console-band` | `#1b212e` | コンソール chrome(濃色スレート) |
| `--console-band-2` | `#242c3c` | chrome の副帯 |
| `--console-line` | `#333d52` | コンソール内の罫線 |
| `--console-accent` | `#13a394` | 運営アクセント(ティール) |
| `--console-accent-d` | `#0f8678` | アクセント hover |
| `--c-active` | `#2f9e44` | テナント状態 `ACTIVE` |
| `--c-suspended` | `#b8650b` | テナント状態 `SUSPENDED` |

- 濃色 chrome + ティールアクセントで「専任運営用の別空間」を視覚化し、業務動線(明るい primary)と明確に区別する。
- このコンソールテーマは固定で、テナント別テーマ可変の対象外とする(§9.2)。
- テナント状態(`ACTIVE` / `SUSPENDED`)、メンバー状態(`ACTIVE` / `INVITED` / `DISABLED`)はバッジで表し、状態切替は §6.5 の確認モーダルを必須とする。

## 9. テナント別テーマ可変性の指針

### 9.1 可変箇所(Phase 2 で本格対応)

将来のテナント別テーマでは次のみを可変とする。

- **ロゴ / ブランド名**: トップナビのブランド領域。
- **アクセントカラー**: `--app-primary` と派生(`--app-primary-d`、`--ring`)。

実装方針(素案): テナント単位の上記 CSS 変数を `body[data-tenant]` または `:root` スコープへ実行時注入し、コンポーネント側は常に変数参照のみとする(§2.4)。Phase 1 は単一の既定テーマで運用し、可変点だけを設計上確保しておく。

### 9.2 固定箇所(テーマで変更しない)

- **セマンティックカラー**(`--c-overdue` / `--c-today` / `--c-soon` / `--c-done`): 意味を担うため固定。
- **テナント状態色**(`--c-active` / `--c-suspended`)。
- **SaaS Admin コンソールテーマ**(§8): 運営空間であり、特定テナントの色に染めない。
- 基盤色(背景 / 面 / 罫線 / 文字)は原則固定。テナント差し替えはアクセントとロゴに限定し、可読性とコントラスト基準(§7.1)を崩さない。
