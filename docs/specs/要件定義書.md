## 1. はじめに

### 1.1 本書の目的

本書は、マルチテナント型SaaSとして提供するタスク管理システム(以下、本システム)の要件を定義し、関係者間で合意形成を図ることを目的とする。本書は基本設計および以降の開発工程の入力となる。

### 1.2 システム概要

本システムは、複数の契約企業(テナント)が独立してタスク管理を行えるマルチテナントSaaSである。共通のインフラ・コードベース上で論理的にデータを分離し、テナント単位での課金・管理を可能とする。利用者はWebブラウザからアクセスし、Keycloakによるシングルサインオンで認証する。

### 1.3 適用範囲

本書は本システムの全機能および非機能要件を対象とする。Keycloak の構築・運用要件、users テーブル(本システム内 SoT・ADR-0006)との連携要件も含む。外部システム連携(Slack、Google Calendar等)は本フェーズでは対象外とし、将来拡張機能として位置付ける。

## 2. 業務要件

### 2.1 背景・課題

現在、多くの小規模チームでは Excel やチャットツール、付箋等を併用してタスクを管理しており、以下の課題が発生している。

- タスクの抜け漏れ・期限超過の発生
- 担当者・進捗状況の不透明さによる重複作業
- 過去のタスク履歴・対応経緯が散逸し、ナレッジ化されない
- 管理者がチーム全体の負荷状況をリアルタイムに把握できない
- 複数の契約企業へSaaSとして提供したいが、データ分離・課金モデルが未整備

### 2.2 目標

本システムにより、以下の状態を実現することを目標とする。

- テナントごとに独立したタスク管理空間を提供し、データ漏洩リスクをゼロにする
- チーム全員のタスクを一元管理し、抜け漏れをゼロにする
- 担当者・期限・優先度をひと目で把握できる仕組みを提供する
- 管理者がチームの稼働状況をダッシュボードで可視化できる
- Keycloak + 本システム内 users テーブル(SoT・ADR-0006)を活用し、ユーザー登録の二重管理を排除する

### 2.3 想定ユーザー・業務

| 役割 | 想定人数 | 主な業務 |
|---|---|---|
| SaaS運営者(本システム提供者) | 数名 | テナント管理(状態変更・停止)・全体監視。業務データには触れない(プラットフォーム運用 API のみ) |
| テナント管理者(Tenant Admin) | 1〜2名/テナント | ユーザー招待、ロール変更、監査ログ確認、テナント運営者向けダッシュボード参照(タスクへの特別権限はなし、ADR-0005) |
| メンバー(Member) | 5〜20名/テナント | タスク登録・更新、可視性ルールに従うタスク閲覧(§3.4.3)、所有者・担当者として関係者管理(§3.4.5) |

> テナント新規作成は SaaS運営者ではなく**認証済みユーザーが自分自身で行う**(セルフサインアップ、§3.3.1 参照)。SaaS運営者の責務は作成済みテナントの状態管理・全体監視に限定される。契約管理・テナント解約は Phase 2 で扱う。

### 2.4 マルチテナントモデル

- 1ユーザー(users.id)は複数テナントに所属可能(コンサル等の兼務想定)
- テナントごとにロール(Tenant Admin / Member)を保持(参照のみのViewerは設けない)
- データは全業務テーブルに tenant_id 列を保持し、API/SQLレベルで自動的にテナント絞込を行う
- テナント間のデータ参照は不可とする。SaaS運営者(`APP_ADMIN`)もテナント運営 API(テナント管理・プラットフォームメトリクス)以外では業務データに触れない

### 2.5 タスクの可視性・編集権限モデル

- デフォルト方針: テナント内のタスクは全メンバーが参照可能(ナレッジ共有を促進)
- **タスク認可は所有者・担当者・関係者の 3 役割のみで評価する**(ADR-0005)。Tenant Admin の業務タスクに対する特別権限(強制編集・常時参照・特権削除等)はない — Tenant Admin はテナント運営権限(招待・ロール管理・監査ログ参照・テナント運営者ダッシュボード参照)のみを持つ
- タスクの可視性は 3 段階を所有者が設定可能: 全公開(テナント全員) / 関係者限定 / 非公開(所有者・担当者のみ)
- 編集・論理削除・公開範囲変更は所有者のみ(関係者リストの追加・削除は担当者も可)。ステータス変更は所有者・担当者
- 関係者(Stakeholder)は所有者・担当者がテナント内ユーザーから自由に指名し、タスク単位で関係を構築する(指名自体は visibility に関係なく可能、参照権限は visibility = STAKEHOLDERS のときのみ発動)
- 可視性を切替えても監査ログには元タスクIDが残り、不正な参照は検知される

### 3.2 認証・認可機能(Keycloak)

#### 3.2.1 ログイン(F-01)

- Keycloak (OpenID Connect) によるSSO
- 認可コードフロー(PKCE)を使用
- Keycloakから取得したJWTの `sub` クレームを users テーブル(本システム内 SoT・ADR-0006)の `oidc_sub` と突合
- 初回ログイン時、`oidc_sub` が users に存在しない場合は **JIT(Just-In-Time)プロビジョニングで users 行を自動作成**する(セルフサインアップを前提とした仕様。Keycloak の `sub` / `email` / `name` クレームを users に転写)
- ユーザー属性(full_name 等)は users(本システム内 SoT)から取得

#### 3.2.2 認可(F-06)

- 認可は本システム内の `user_tenants(role)` で制御
- Keycloakのrealm rolesは「APP_USER」「APP_ADMIN(SaaS運営)」のみを使用
- テナント単位の細粒度ロールは本システム側で管理

#### 3.2.3 テナント切替(F-03)

- ヘッダにテナント切替プルダウンを表示
- 切替時はJWTを再取得せず、本システムが発行する内部セッションコンテキストを切替
- 選択中のテナントは全API呼出のCookie/ヘッダで送出

### 3.3 テナント・ユーザー管理機能

#### 3.3.1 テナント管理(F-04)

**テナント新規作成(セルフサインアップ、モデル B)**

- 認証済み(Keycloak ログイン済み)のユーザーが自分自身でテナントを新規作成する
- 作成と同時に呼び出しユーザーが初代 Tenant Admin として `user_tenants` に登録される(アトミック)
- 呼び出しユーザーの users 行は初回ログイン時に JIT で自動作成される(§3.2.1 参照)。JIT は JWT 認証フィルタ層で実行されるため、本 API のハンドラ到達時点で users 行は必ず確保されている前提
- レート制限・ドメイン制限・承認制等の濫用対策は **Phase 2 で導入**(本フェーズでは認証済みなら誰でも作成可能)

**テナント状態管理**

- SaaS運営者のみが操作可能
- 状態は `ACTIVE` / `SUSPENDED` の 2 値。`SUSPENDED` のテナントに所属するユーザーは、当該テナントへの**業務 API アクセスが 403 で拒否される**(Keycloak ログイン自体は通り、所属テナント一覧 `/api/auth/me` の取得は可能。SUSPENDED のテナントをアクティブテナントとして選択しての業務操作のみが拒否される)。データは無傷で残る
- 状態切替は監査ログに記録され、SaaS運営者は復活させることができる
- 実装上の認可拒否ポイント(TenantContext 解決位置・Spring Security フィルタの配置・`X-Tenant-Id` ヘッダ不要枠の扱い等)は基本設計書 §6.2 / §6.3 に委任する
- テナント解約(物理削除・データ凍結)は **Phase 2 で扱う**(本フェーズの停止はアクセス拒否のみで、データの物理削除や凍結は行わない)

**プラン変更・契約管理**

- プラン・契約期間・利用枠・課金は **Phase 2 で扱う**(本フェーズでは全テナント無料・利用枠なし)

#### 3.3.2 ユーザー招待(F-05)

- テナント管理者がメールアドレス指定で招待
- 招待されたメールアドレスが Keycloak に未登録の場合、Keycloak 側で先にユーザー登録が必要(本システム外)
- 招待された人物が初めてログインしたタイミングで、本システム側 users 行は JIT で自動作成される(§3.2.1)。`user_tenants` への紐付けは招待トークンの照合により行う
- users(本システム内 SoT)に `oidc_sub` が既に登録済みであれば、`user_tenants` への紐付けのみが発生し即時利用可
- 招待トークンの有効期限、JIT(users 行作成)と `user_tenants` 紐付けのトランザクション境界、招待トークン照合エラー時の挙動等の実装詳細は基本設計書 §3.3 / §6.2 に委任する

#### 3.3.3 ロール管理(F-06)

- Tenant Admin: **テナント運営権限のみ**(ユーザー招待、ロール管理、テナント設定変更、監査ログ参照、テナント運営者向けダッシュボード参照)。**業務タスクへの特別権限は持たない**(ADR-0005)。タスク操作は所有者・担当者・関係者として通常のルール(§2.5)で行う
- Member: タスク CRUD(編集・論理削除は所有者のみ、ステータス変更は所有者・担当者、参照は §3.4.3・§3.4.4 の可視性ルールに従う)
- Tenant Admin / Member の区別は**タスク認可には影響しない**。両者ともテナント内ユーザーとして同等のタスク操作権限を持ち、所有者・担当者・関係者の役割に応じて操作可能

### 3.4 タスク管理機能

#### 3.4.1 タスク登録(F-08)

- タイトル(必須、100文字以内)、説明(任意、2000文字以内)を入力
- 期限日(必須、YYYY-MM-DD)、優先度(必須)、担当者(任意)を設定
- 公開範囲(必須): 全公開 / 関係者限定 / 非公開 のいずれかを選択
- 関係者(任意): **公開範囲に関係なく指名可能**(ADR-0005)。関係者であることが参照権限として機能するのは公開範囲が「関係者限定」のときのみで、全公開時はテナント内全員、非公開時は所有者・担当者のみが参照可。関係者リストへの登録は将来の公開範囲変更に備える「事前準備」としても機能する
- 登録者は自動的に所有者(owner)となる。初期ステータスは「未着手」
- 登録時に自動的にログイン中のテナントの tenant_id を付与

#### 3.4.2 タスク編集(F-09)/ 完了(F-11)/ 削除(F-10)

- ステータスは「未着手」「進行中」「完了」「保留」の4種類
- 編集・論理削除はタスク所有者のみ実行可能(Tenant Admin の特別権限なし、ADR-0005)
- ステータス変更(F-11 完了化を含む)は所有者・担当者が実行可能
- 関係者の追加・削除は所有者・担当者が実行可能(詳細は §3.4.5)
- 物理削除は MVP では提供しない(完全削除はテナント解約 #167 / Phase 2 で扱う)
- 更新時は更新者・更新日時を記録(監査ログ)

#### 3.4.3 タスク一覧表示(F-12)

- デフォルト表示: ログイン中テナント・当日(due_date = TODAY)に期限のタスクを期限昇順で表示
- 日付切替: 画面上部の日付ピッカーで別日のタスクに切替可能
- 期限切れ常時表示: 完了していない期限切れタスク(due_date < TODAY)は表示日付に関わらず常に上段に表示
- 可視性フィルタ: ログイン中ユーザーが参照可能なタスクのみ表示(全公開 ∪ 自分が所有者・担当者・関係者の関係者限定タスク ∪ 自分が所有者・担当者の非公開タスク)。Tenant Admin であってもこのフィルタは同じく適用される(ADR-0005、特別権限なし)
- 1ページ50件のページネーション(期限切れタスクが50件超の場合は別タブ)

#### 3.4.4 公開範囲設定(F-13)

- TENANT(全公開): テナント全メンバーが参照可。デフォルト値
- STAKEHOLDERS(関係者限定): 所有者・担当者・関係者リスト登録者のみ参照可
- PRIVATE(非公開): **所有者・担当者**のみ参照可(ADR-0005、旧仕様「所有者のみ」から拡張)
- 可視性は所有者のみ変更可能。変更時は監査ログに記録

##### visibility 変更時の task_stakeholders レコード挙動

`PATCH /api/tasks/{id}/visibility` で公開範囲を変更した際の `task_stakeholders` レコードの扱いは以下のとおり。

| 変更方向 | task_stakeholders レコード | audit_logs |
|---|---|---|
| → STAKEHOLDERS | `stakeholderUserIds` で指定したユーザーに**置換**。既存レコードは一括削除後に再 INSERT | `VISIBILITY_CHANGED` |
| → TENANT | レコードを**保持**する(将来 STAKEHOLDERS に再昇格した際に流用可) | `VISIBILITY_CHANGED` |
| → PRIVATE | レコードを**CASCADE 削除**する | `VISIBILITY_CHANGED` + `STAKEHOLDER_PURGED`(削除レコード件数を `detail` に記録) |

> **PRIVATE 戻し時に削除する理由**: visibility = PRIVATE では関係者リストが参照権限に影響しない(所有者・担当者のみ参照可)。レコードを残すと「PRIVATE タスクの関係者候補リストが見えてしまう」リスクがあるため、削除して監査ログに記録することで情報フロー保護(NIST AC-4)と監査追跡の両立を図る。

#### 3.4.5 関係者管理(F-14)

- **所有者・担当者**がタスク単位で関係者を追加・削除可能(ADR-0005)
- 関係者はテナント内の Tenant Admin / Member から選択(役割区別なし)
- 関係者の指名は公開範囲に関係なく可能(参照権限が発動するのは公開範囲が STAKEHOLDERS のときのみ、§3.4.1 参照)
- 関係者は期限当日の通知**対象外**(ADR-0005、旧仕様から変更)
- 関係者は参照可能(公開範囲が STAKEHOLDERS のタスクに限る、§3.4.1 参照)、編集は不可(ステータス変更も不可)

### 3.5 期限・優先度機能

#### 3.5.1 期限管理

- 期限日は登録時に必須(YYYY-MM-DD)
- 期限を過ぎ未完了のタスクは赤色強調表示で常時表示
- 期限当日のタスクは黄色強調表示

#### 3.5.2 優先度設定(F-15)

- 高(赤)/中(黄)/低(緑) の3段階
- 一覧でアイコンによる視覚的区別を行う

### 3.6 検索・絞り込み機能(F-16)

- キーワード検索(タイトル・説明を対象、部分一致)
- 所有者・担当者・ステータス・優先度・公開範囲で絞り込み
- 期間指定(期限の from-to)で絞り込み
- 検索条件はURLクエリで保存可能とする

### 3.7 通知機能(F-18)

- 期限当日 09:00 に**所有者・担当者**へメール通知を送信(ADR-0005、関係者は対象外)
- ユーザー設定で通知の有効/無効を切替可能(本フェーズの対象は所有者・担当者向け F-18 期限当日通知のみ)
- 本フェーズでは F-18 期限当日通知のみ実装。アサイン通知・関係者追加通知・ステータス変更通知等は Phase 2 で整理(ADR-0005 §3.4)
- 将来はSlack/Teams連携を想定しI/F設計に拡張余地を残すこと

### 4.2 セキュリティ要件(NIST準拠)

本システムは社内セキュリティ規定が無いため、NIST SP 800-53 Rev.5(セキュリティ統制)、NIST SP 800-63B(認証ガイドライン)、NIST Cybersecurity Framework に準拠する。

| NISTカテゴリ | 本システム適用 |
|---|---|
| AC (Access Control) | RBAC + テナント分離、最小権限の原則 |
| AU (Audit and Accountability) | 監査ログ1年保管、改ざん不可なログ設計 |
| IA (Identification and Authentication) | Keycloak によるOIDC SSO、MFA推奨設定 |
| SC (System and Communications Protection) | TLS 1.3、テナント間データ分離、保存時暗号化 |
| SI (System and Information Integrity) | 脆弱性パッチ管理、入力検証、整合性チェック |
| パスワードポリシー(SP 800-63B) | Keycloak側で実装、漏洩パスワードDB照合、定期変更強制なし |

- 通信は全てHTTPS(TLS 1.3、TLS 1.2は最小要件)で暗号化
- 認証はKeycloakが発行するJWT、有効期限15分、リフレッシュトークン8時間
- CSRF/XSS/SQLインジェクション対策を実装(OWASP Top 10準拠)
- RBAC + テナント分離の二重防衛
- DBは保存時暗号化(AWS RDS MySQLの場合は KMS暗号化を有効化)
- 監査ログ(認証イベント・データ変更)を1年間保管、改ざん検知のためハッシュチェーン化

### 5.3 ID 基盤・users テーブルとの連携(ADR-0006)

- users テーブルは**本システム DB 内に保有する新規テーブル**であり、本システムが SoT として所有する(将来的に既存 ID 基盤へのデータ移行余地を残す)
- Keycloak は Writable User Storage SPI 経由で users を read/write する(Sprint 1 Infra 実装)
- `oidc_sub` 列で Keycloak の `sub` クレームと突合
- テナント所属とロールは別テーブル `user_tenants` で管理
- プロフィール(full_name 等)は users を参照・更新。退会・解約時は論理削除 + 個人情報匿名化(ADR-0006 §3.4)

| カラム名 | 型 | NULL | 説明 |
|---|---|---|---|
| id | BIGINT AUTO_INCREMENT | NO | 主キー(変更不可) |
| oidc_sub | VARCHAR(255) | NO | Keycloak の Subject(UNIQUE)。匿名化時は `__deleted__{id}` に置換 |
| email | VARCHAR(255) | NO | メールアドレス。匿名化時は `__deleted__{id}@deleted.invalid` に置換 |
| full_name | VARCHAR(255) | NO | 氏名。匿名化時は `__deleted__` に置換 |
| full_name_kana | VARCHAR(255) | NO | 氏名カナ。匿名化時は `__deleted__` に置換 |
| department_name | VARCHAR(255) | YES | 部署名。匿名化時は NULL に置換 |
| version | BIGINT | NO DEFAULT 0 | JPA `@Version` 楽観排他用(本システム API と Keycloak Console の 2 経路書き込みに対応) |
| deleted_at | DATETIME | YES | 論理削除日時。NULL = 有効、NOT NULL = 匿名化済み |

## 8. 用語集

| 用語 | 定義 |

| 担当者(Assignee) | タスクを実施する者。任意設定。ステータス変更・関係者リストの追加・削除が可能。所有者と同じく公開範囲が PRIVATE のタスクも参照可。期限当日に通知される |
| 関係者(Stakeholder) | 公開範囲が STAKEHOLDERS のタスクの参照を許可される。所有者・担当者がタスク毎に指名(指名自体は visibility に関係なく可能、参照権限は STAKEHOLDERS 時のみ発動)。期限当日通知の対象外 |

| ステータス | 未着手/進行中/完了/保留 の4種 |
| クリーンアーキテクチャ | 依存性が外側から内側へのみ流れるレイヤード設計 |
| Keycloak | OSSのID/アクセス管理基盤。本プロジェクトで新規構築する |
| OIDC | OpenID Connect |
| JWT | Keycloakが発行する署名付き認証トークン |
