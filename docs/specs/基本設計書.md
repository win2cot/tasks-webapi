## 2. システム構成

### 2.1 アーキテクチャ概要

本システムはクリーンアーキテクチャ(Domain / UseCase / Adapter / Infrastructure の4層)を採用するマルチテナントSaaSである。依存関係は外側から内側にのみ流れ、Domain層はフレームワーク非依存である。フロントエンドはHTML/CSS/JavaScriptによるSPA風の単一ページ構成とし、バックエンドはJava 21 / Spring Boot 4で構築するREST APIを提供する。認証・認可はKeycloak(OIDC)に委譲し、データはMySQL 8.xに永続化する。実行基盤はAmazon ECS on AWS Fargate(EC2は使用しない)。テナント分離は共用DB・共用スキーマ方式とし、全業務テーブルにtenant_id列を持つ。

### 2.1.1 クリーンアーキテクチャ層構成

| 層 | パッケージ例 | 責務 |
|---|---|---|
| Domain | com.example.task.domain | エンティティ、値オブジェクト、ドメインサービス、リポジトリI/F。Spring非依存 |
| UseCase (Application) | com.example.task.usecase | ユースケース実装、入出力Port定義、トランザクション境界 |
| Adapter | com.example.task.adapter.web / .persistence / .external | REST Controller、JPA Repository実装、Keycloak/SES クライアント |
| Infrastructure | com.example.task.infra | Spring Boot設定、DataSource、SecurityConfig、起動用Application |

- 依存方向: Infrastructure → Adapter → UseCase → Domain(逆方向の参照は禁止)
- Domain は POJO のみ。JPA アノテーションは Adapter.persistence の Entity クラスに記述
- DTO ↔ Domain の変換は Adapter 層の Mapper(MapStruct使用)で行う
- テストは Domain / UseCase が単体テストの主対象。Adapter は結合テスト中心

### 3.3 主要画面仕様

#### 3.3.1 タスク一覧画面(S-04)

| 項目 | 仕様 |
|---|---|
| 日付ピッカー | 画面上部。デフォルトはTODAY。前日/翌日ボタン併設。「今日に戻る」ボタン |
| 期限切れセクション | 日付フィルタの上に常時表示。期限切れ未完了タスクのみを赤色強調で一覧表示 |
| 本日(または選択日)セクション | 選択日を期限とするタスクを表示 |
| 上部フィルタ | ステータス・所有者・担当者・優先度・公開範囲・キーワード |
| 並び替え | 期限/優先度/作成日(昇順・降順) |
| 一覧項目 | タイトル / ステータス / 所有者 / 担当者 / 期限 / 優先度 / 公開範囲アイコン |
| 行操作 | クリックで詳細遷移、チェックボックスで完了化(所有者/担当者のみ) |
| 編集ボタン | 所有者にのみ表示(ADR-0005、Tenant Admin の特権なし) |
| 新規追加 | 右上「+ 新規」ボタンで S-06 へ遷移(初期値: 期限=表示中の日付) |
| ページング | 本日セクション・期限切れセクションそれぞれ50件単位 |

#### 3.3.2 タスク登録/編集画面(S-06)

- タイトル(必須、100文字)
- 説明(任意、2000文字、Markdownサポート将来検討)
- 期限日(必須、カレンダーピッカー)
- 担当者(任意、プルダウン、テナント内ユーザーから選択)
- 優先度(必須、ラジオボタン: 高/中/低)
- ステータス(プルダウン: 未着手/進行中/完了/保留)
- 公開範囲(必須、ラジオボタン: 全公開/関係者限定/非公開)
- 関係者リスト(公開範囲に関係なく活性、テナント内ユーザーから複数選択可。参照権限が機能するのは公開範囲が「関係者限定」のときのみだが、指名自体は visibility 非依存に行える(ADR-0005 §3.4))
- 編集画面の操作可否は ADR-0005 の 3 役割評価に従う:
    - 所有者: 全項目編集可(タイトル / 説明 / 期限 / 担当者 / 優先度 / 公開範囲 / 関係者リスト)。対応 API は A-14(`PUT /api/tasks/{id}`) / A-17(`PATCH /api/tasks/{id}/visibility`) / A-19・A-20(関係者の追加・削除)
    - 担当者: ステータス・関係者リストのみ編集可。他項目は読み取り専用。対応 API は A-16(`PATCH /api/tasks/{id}/status`) / A-19・A-20(関係者の追加・削除)
    - その他(参照可能なユーザー): 全項目読み取り専用
- 関係者追加・削除ボタンは**所有者または担当者にのみ表示**(A-19 / A-20 の認可ルールに準拠)。閲覧のみのユーザー(関係者など)には表示しない
- Tenant Admin は本画面で業務タスクへの特別権限を持たない(ADR-0005、Tenant Admin はテナント運営 API のみ操作可)

#### 3.3.3 ダッシュボード画面(S-03)

- 当日の期限タスク件数(大きく表示)
- 期限超過(未完了)タスク件数
- 本日完了したタスク件数
- 自分が所有するタスクの進捗グラフ(完了/進行中/未着手の円グラフ)
- 優先度別の件数バーチャート

> 集計スコープは §6.2.2 のとおり、ログイン中ユーザーが §6.2.1 のタスク参照認可フィルタを通過したタスク集合を対象とする(Tenant Admin であっても通常の参照認可フィルタが適用される。ADR-0005)。

#### 3.3.4 テナント運営者向けダッシュボード(S-15)

| 項目 | 仕様 |
|---|---|
| アクセス権 | `hasRole('TENANT_ADMIN')`。Member は 403。SaaS Admin は業務 API 不可で 403(§6.2.1「SaaS Admin の扱い」参照) |
| 集計対象 | 自テナント内の `visibility ∈ {TENANT, STAKEHOLDERS}` のタスクのみ。`PRIVATE` タスクは件数も含めて集計対象外(NIST AC-4 整合、ADR-0005) |
| 表示項目 | 集計可能タスク総数 / 当日期限件数 / 期限超過件数(未完了かつ `due_date < TODAY`)/ 本日完了件数 / ステータス別件数(未着手・進行中・完了・保留) / 優先度別件数(高・中・低) / 所属メンバー数(`user_tenants.status = ACTIVE` の人数) |
| MVP 対象外 | メンバー別稼働状況・個人特定可能な指標は MVP では実装しない(ADR-0005 §3.5)。時系列グラフは未対応(将来検討) |
| 算出 API | A-28 `GET /api/tenant/dashboard/summary`(§5.1) |

> S-03(個人視点)と S-15(運営者視点)の認可哲学は一貫している: いずれも PRIVATE タスクの**存在**を他者の集計値経由で推定できないようにする(NIST AC-4 情報フロー保護)。

#### 4.2.5 task_stakeholders (タスク関係者)

関係者指名は visibility 非依存に可能(ADR-0005 §3.3)。参照権限として機能するのは `visibility = STAKEHOLDERS` のときのみ。

| カラム名 | 型 | PK | NOT NULL | 備考 |
|---|---|---|---|---|
| task_id | BIGINT | ○ | ○ | FK: tasks.id |
| user_id | BIGINT | ○ | ○ | FK: users.id |
| tenant_id | BIGINT |   | ○ | テナント分離(検索高速化) |
| added_by | BIGINT |   | ○ | 追加した所有者または担当者(監査用、ADR-0005) |
| added_at | DATETIME |   | ○ | 追加日時 |

> **visibility 変更時のレコード挙動**(Issue #92 / 要件定義書 §3.4.4):
>
> | 変更方向 | task_stakeholders レコード | audit_logs |
> |---|---|---|
> | → STAKEHOLDERS | `stakeholderUserIds` で**置換**(既存を一括削除後に再 INSERT) | `VISIBILITY_CHANGED` |
> | → TENANT | **保持**(将来 STAKEHOLDERS 再昇格時に流用) | `VISIBILITY_CHANGED` |
> | → PRIVATE | **CASCADE 削除**(NIST AC-4 情報フロー保護) | `VISIBILITY_CHANGED` + `STAKEHOLDER_PURGED` |
>
> PRIVATE 戻し時に削除する理由: visibility = PRIVATE では関係者リストが参照権限に影響しない(所有者・担当者のみ参照可)。レコードを残すとリスト情報が不必要に残留するため削除し、件数を audit_logs の `detail` に記録して監査追跡を確保する。

### 5.2 認証方式(Keycloak / OIDC)

- 認証はKeycloakのOIDC(認可コードフロー + PKCE)に委譲
- Keycloak が発行するJWTを Spring Security OAuth2 Resource Server で検証
- アクセストークン有効期限: 15分、リフレッシュトークン: 8時間(NIST SP 800-63B AAL2想定)
- API呼出時は HTTPヘッダ Authorization: Bearer {access_token} を付与
- ログイン直後に oidc_sub を usersテーブルと突合し、内部ユーザーIDを解決
- アクティブテナントはCookieまたは X-Tenant-Id ヘッダで送出、毎リクエストで user_tenants の権限を確認
- **`X-Tenant-Id` ヘッダ不要枠**: 以下の API は `X-Tenant-Id` を要求せず、TenantContext を構築しない。Spring Security レベルの認証(JWT)と必要に応じてロール(`APP_ADMIN`)で制御する
  - 認証系: A-01 `GET /api/auth/me` / A-02 `POST /api/auth/tenants/{id}/select` / A-03 `POST /api/auth/logout`
  - セルフサインアップ: A-05 `POST /api/tenants`(認証済みでテナント未所属でも呼び出し可)
  - SaaS Admin 専用テナント運営 API: A-04 `GET /api/tenants` / A-06 `PUT /api/tenants/{id}` / A-25 `GET /api/tenants/{id}` / A-26 `PATCH /api/tenants/{id}/status`
  - SaaS Admin 専用プラットフォーム API: A-27 `GET /api/platform/metrics`
- 上記以外の業務 API(A-07 以降の `/api/users/*`、`/api/tenant/users/*`、`/api/tenant/dashboard/*`、`/api/tasks/*`、`/api/dashboard/*`、`/api/audit-logs`、`/api/users/me/notification-settings`)は `X-Tenant-Id` 必須

#### 5.2.1 認証フロー

[1] ブラウザ ──(GET /)──> AP

[2] AP ──(302 認可エンドポイント)──> Keycloak

[3] Keycloak ── ログイン画面表示・認証 ──> ブラウザ

[4] Keycloak ──(認可コード)──> ブラウザ ──> AP /callback

[5] AP ──(コード+PKCE)──> Keycloak ──(JWT)──> AP

[6] AP: JWT.sub を users.oidc_sub と突合、user_tenants からロール解決

[7] AP ──(セッションCookie)──> ブラウザ

[8] 以降のAPI呼出: Authorization: Bearer ... + X-Tenant-Id: {tenant_id}

### 6.2 認証・認可

- Keycloak経由のOIDC認証、Spring Security OAuth2 Resource Server でJWT検証
- ロールは **2 軸で直交**:
  - **プラットフォーム軸**: Keycloak realm role `APP_ADMIN`(SaaS Admin)。本システム DB には保持しない。テナント運営 API(テナント管理・プラットフォームメトリクス)のみを操作可能。**業務 API には触れない**
  - **テナント軸**: `user_tenants.role`(`TENANT_ADMIN` / `MEMBER`)。テナントスコープで本システム内に保持。Viewer は設けない
- 両軸は包含関係ではなく直交する。あるユーザーが両軸で同時に所属することはあり得る(例: SaaS 運営者が自社テナントの Tenant Admin も兼ねる)。**兼務ユーザーが業務 API を呼び出した場合の認可は、テナント軸の所属(`user_tenants.role`)に基づいて評価される**:
  - 兼務ユーザーが自分が `TENANT_ADMIN` / `MEMBER` として登録されているテナントを `X-Tenant-Id` で指定 → そのテナント軸ロールで業務 API を利用可能(`APP_ADMIN` ロールは業務 API では追加権限を付与しない)
  - 同じ兼務ユーザーがプラットフォーム運用 API(`/api/platform/*`、`/api/tenants` の SaaS Admin 操作)を呼び出す → `APP_ADMIN` ロールで通過
  - `APP_ADMIN` のみを持ち `user_tenants` に行を持たないユーザーが業務 API を呼び出す → **403**(後述 §6.2.1)
- @PreAuthorize でメソッドレベル認可、TenantContextでテナント絞込を強制

> **SaaS Admin の管理方式**: SaaS Admin は Keycloak realm role `APP_ADMIN` で管理し、本システム DB には保持しない。`user_tenants.role` には `TENANT_ADMIN` / `MEMBER` のみが存在し、SaaS Admin に対応する値はない。Spring Security で `hasRole('APP_ADMIN')` を用いてプラットフォーム API アクセスを制御する。

#### ロール表記 設計書表記 ↔ 内部コード値マッピング

| 設計書表記(UI・ドキュメント) | 内部コード値(OpenAPI / DDL / Keycloak) | 保持場所 |
|---|---|---|
| SaaS Admin | `APP_ADMIN` | Keycloak realm role(本システム DB には保持しない) |
| Tenant Admin | `TENANT_ADMIN` | `user_tenants.role` |
| Member | `MEMBER` | `user_tenants.role` |

#### セキュリティ検証の層別責務

JWT 検証・テナント解決・認可ロジックが**どの層のどのコンポーネント**で実行されるかを明示する。実装者が同じ検証を複数層で重複実装することを防ぐ。

| 検証・解決の種類 | 担当コンポーネント | 層 |
|---|---|---|
| JWT 署名検証 | Spring Security フィルタ(OAuth2 Resource Server) | Infrastructure |
| JWT クレーム抽出 | `TasksJwtAuthenticationConverter` | Infrastructure |
| アクティブテナント解決 | `TenantContextFilter`(`X-Tenant-Id` ↔ `user_tenants` 照合) | Infrastructure |
| テナント越境検出 | Hibernate Filter | Infrastructure |
| メソッドロール認可 | `@PreAuthorize` | Adapter(UseCase 入口) |
| タスクごとの認可 | `TaskAuthorizationDomainService` | Domain |

各ステップはリクエスト処理順に上から実行される。JWT 署名検証が失敗した時点でそれ以降のステップには進まない(401 を返却)。テナント解決が失敗した場合(X-Tenant-Id 未指定・非所属テナント)も後続ステップには進まない(403)。

#### 6.2.1 タスクごとの認可ロジック

タスク認可は **所有者(Owner)・担当者(Assignee)・関係者(Stakeholder)の 3 役割のみで評価**する(ADR-0005)。Tenant Admin はテナント運営権限(招待・ロール管理・監査ログ参照・S-15 ダッシュボード参照)のみを持ち、業務タスクへの特別権限は持たない。Tenant Admin であってもタスク操作時は所有者・担当者・関係者のいずれかとして通常ルールに従う。

| 操作 | 認可ルール |
|---|---|
| 参照 | 公開範囲 = `TENANT`: 同テナント全員 / `STAKEHOLDERS`: 所有者 ∪ 担当者 ∪ `task_stakeholders` 登録者 / `PRIVATE`: 所有者 ∪ 担当者 |
| 編集(タイトル / 説明 / 期限 / 担当者 / 優先度 等) | 所有者のみ |
| 論理削除 | 所有者のみ |
| 物理削除 | MVP 不実装(テナント解約 #167 / Phase 2 で扱う) |
| ステータス変更 | 所有者 ∪ 担当者 |
| 公開範囲変更 | 所有者のみ |
| 関係者の追加・削除 | 所有者 ∪ 担当者(指名自体は visibility 非依存に可能。参照権限が機能するのは `visibility = STAKEHOLDERS` のときのみ、ADR-0005 §3.4) |
| 関係者一覧の参照(`GET /api/tasks/{id}/stakeholders`) | タスクを参照可能なユーザー全員(上記「参照」ルール適用後) |

> **注**: ADR-0005 で `PRIVATE` の参照対象に**担当者**を追加(旧仕様の「所有者のみ」から拡張)。所有者・担当者が役割としてタスクに関与している前提に揃え、3 役割評価モデルの一貫性を確保している。

**SaaS Admin の扱い**: SaaS Admin(`APP_ADMIN`)は **業務 API(`/api/tasks` 配下、`/api/dashboard/*`、`/api/tenant/users/*`、`/api/tenant/dashboard/*`、`/api/audit-logs` 等)に対するアクセス権を持たない**。仮にこれらを呼び出した場合、Spring Security レベルで `403 Forbidden` を返却する(`X-Tenant-Id` ヘッダの有無に関わらず)。

**実装 SSOT**: 上記ロジックは `TaskAuthorizationDomainService`(Domain 層、`xyz.dgz48.tasks.webapi.task.domain`)に一元化する。UseCase は同サービスを呼び出し、不可なら `TaskNotViewableException`(404)または `TaskOwnershipException`(403)を throw する。シーケンス図・OpenAPI の認可説明は本節を参照すること。本節と ADR-0005 §3 は内容上等価であり、矛盾が生じた場合は ADR-0005 を SSOT とする。

> **実装注意 — STAKEHOLDERS 認可判定は事前一括フェッチ必須**: `visibility = STAKEHOLDERS` のタスクに対して参照認可判定を行う際、ループ内でタスクごとに `task_stakeholders` をクエリすると N+1 問題が発生する。`ListTasksUseCase` 等でタスク一覧に認可フィルタを適用する前に、必ず `StakeholderRepository.findTaskIdsByUser(userId, tenantId)` を 1 回呼び出して `Set<Long> stakeholderTaskIds` を一括取得し、ループ内では取得済みセットを参照すること(sequence-03 参照)。

#### 6.2.2 ダッシュボード集計の認可スコープ

##### 6.2.2.1 個人視点ダッシュボード(S-03 / `GET /api/dashboard/summary`)

F-17 個人ダッシュボードは、§6.2.1 のタスク参照認可フィルタを適用した後のタスク集合に対して件数集計を行う(Tenant Admin であっても通常の参照認可フィルタが適用される、ADR-0005)。

- `PRIVATE` タスクは **所有者・担当者** のダッシュボードでのみ件数加算される(ADR-0005 で PRIVATE 参照は所有者・担当者に拡張済)
- `STAKEHOLDERS` タスクは「所有者・担当者・関係者リスト登録ユーザー」にのみ件数加算される
- `TENANT` タスクはテナント全員に同一件数で加算される

##### 6.2.2.2 テナント運営者向けダッシュボード(S-15 / `GET /api/tenant/dashboard/summary`)

S-15 は Tenant Admin のテナント運営視点での集計を提供する。認可と集計スコープは以下のとおり(ADR-0005 §3.5):

- 認可: `hasRole('TENANT_ADMIN')`。Member / SaaS Admin は 403(§6.2.1)
- 集計対象: 自テナント内の `visibility ∈ {TENANT, STAKEHOLDERS}` のタスクのみ
- **`PRIVATE` タスクは件数も含めて集計対象外**。Tenant Admin であっても PRIVATE タスクの**存在**を集計値経由で推定できない
- 表示項目: §3.3.4 のとおり(集計可能タスク総数 / 当日期限件数 / 期限超過件数 / 本日完了件数 / ステータス別 / 優先度別 / 所属メンバー数)

##### 6.2.2.3 共通根拠

**NIST AC-4(情報フロー保護)を S-03 と S-15 の両方で一貫させる**。集計件数からも他者の PRIVATE タスクの存在を推定させない。「Tenant Admin がテナント内 PRIVATE タスクの件数は把握できない」というのは ADR-0005 で確定した意図的な設計判断であり、運営者視点としては「テナント全体で共有または関係者限定で動いている業務量」までを把握する建付けに収束させている。

#### 6.2.3 認可違反時の応答ポリシー

認可違反は HTTPステータスを使い分けることで、情報漏洩経路を最小化する。

| 違反種別 | HTTPステータス | 理由 |
|---|---|---|
| 未認証(JWT不正・期限切れ) | **401** Unauthorized | 標準 |
| 参照系(GET)で参照不可 | **404** Not Found | リソースの存在自体を漏らさない(NIST AC-4) |
| 更新系(PUT/PATCH/DELETE)で権限不足 | **403** Forbidden | 編集権限が無いことを明示 |
| テナント越境アクセス | **403** Forbidden | `X-Tenant-Id` と認可コンテキストの不一致は明示的に拒否 |
| ロールベース不可(SaaS Admin専用APIなど) | **403** Forbidden | 認証済だが権限が無い |

**実装方針**: `TaskOwnershipException`(403) と `TaskNotViewableException`(404)を分離し、`@RestControllerAdvice` で対応する HTTPステータスにマッピングする。違反は監査ログに必ず記録する。各違反シナリオで記録する `action` 値を以下に示す。

| 違反シナリオ | HTTP | 記録する action 値 |
|---|---|---|
| タスク参照拒否(可視性フィルタ不通過) | 404 | `VIEW_DENIED` |
| タスク編集拒否(所有者でない) | 403 | `EDIT_DENIED` |
| タスク削除拒否(所有者でない) | 403 | `DELETE_DENIED` |
| ステータス変更拒否(所有者・担当者でない) | 403 | `STATUS_CHANGE_DENIED` |
| 公開範囲変更拒否(所有者でない) | 403 | `VISIBILITY_CHANGE_DENIED` |
| 関係者追加・削除拒否(所有者・担当者でない) | 403 | `STAKEHOLDER_EDIT_DENIED` |
| テナント越境アクセス試行 | 403 | `TENANT_CROSSED` |
| ロール権限不足(SaaS Admin 専用 API 等) | 403 | `ROLE_BASED_DENIED` |

- 機密情報は AWS Systems Manager Parameter Store SecureString で管理(詳細は[設計規約 §5.4](./設計規約.md#54-通信データ保護) 参照、Secrets Manager は本プロジェクトでは未採用)

### 6.6 CSRF/CORS

- API はステートレスのためCSRFトークンは無効化、JWT検証で代替
- CORSは許可Origin をホワイトリスト管理

### 6.7 監査ログの改ざん検知

- audit_logs.hash_chain は前レコードのid+detail+created_atのSHA-256
- バッチで日次整合性チェックを実行、不整合時はアラート

## 7. 例外・ログ設計

### 7.1 例外設計

- 業務例外(BusinessException)とシステム例外(SystemException)を分離
- @RestControllerAdvice で集中ハンドリング
- 業務例外は4xx応答、システム例外は5xx応答
- クロステナントアクセスは TenantAccessDeniedException として403を返却
- タスクの編集権限違反は TaskOwnershipException として403を返却
- タスクの参照権限違反は TaskNotViewableException として404を返却(リソース存在を漏らさない)

### 7.2 ログ設計

| 種類 | 出力先 | 保管 | 内容 |
|---|---|---|---|
| アプリログ | CloudWatch Logs | 30日 | INFO以上、JSON形式 |
| アクセスログ | ALBアクセスログ(S3) | 90日 | リクエスト・レスポンス情報 |
| 監査ログ | DB(audit_logs) | 1年 | 認証・データ変更の記録 |
| エラーログ | CloudWatch Logs | 180日 | WARN/ERROR/FATAL |

- 全ログにtenant_idと相関ID(traceId)を含める

## 8. バッチ設計

### 8.1 バッチ一覧

| No. | バッチ名 | 実行時刻 | 概要 |
|---|---|---|---|
| B-01 | 期限当日通知メール | 毎日 09:00 | 当日期限のタスクを所有者・担当者へメール通知(ADR-0005 §3.4 通知ルール、関係者は対象外)。要件 F-18(要件定義書 §3.7)に対応 |
| B-02 | 期限超過リマインド | 毎日 09:30 | 未完了かつ期限超過タスクを所有者・担当者に通知 |
| B-03 | 監査ログ削除 | 毎月 01日 02:00 | 1年以上前の監査ログを削除 |
| B-04 | DBバックアップ | 毎日 03:00 | RDS自動スナップショット |
| B-05 | 監査ログ整合性検証 | 毎日 02:00 | ハッシュチェーンの整合性を検証 |
| B-06 | テナント使用量集計 | 毎日 04:00 | 課金用にタスク数・ユーザー数を集計 |

### 8.2 実装方針

- Spring の @Scheduled アノテーションで実装、ECS Fargate のスケジュールタスクとは別運用も検討
- 複数インスタンス起動時はShedLockで排他制御(本書 §4.2.8 参照)
- バッチも tenant_id ループで処理し、テナント分離を遵守
- 失敗時はSlack/メールでアラート(将来対応)
