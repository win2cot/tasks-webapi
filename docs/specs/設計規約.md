## 0. 本書の位置付け

本書は **tasks-webapi の設計レベルで遵守すべき規約** を定める。具体的には、アーキテクチャ、パッケージ構成、API・DB・例外・ログ・セキュリティに関する**「やってよいこと」と「やってはいけないこと」**を明文化する。

実装言語レベルの規約(コーディングスタイル、null 安全、Lombok の使い方、テストの書き方など)は別冊 [`コーディング規約.md`](./コーディング規約.md) を参照のこと。

本規約と基本設計書(`基本設計書.md`)に齟齬がある場合、**最新版で齟齬を解消**する。実装と齟齬がある場合、原則として**設計書を正**としつつ、変更が必要であれば ADR を起票して合意してから実装に着手する(ADR の運用は [`../adr/`](../adr/) を参照)。

## 1. アーキテクチャ規約

### 1.1 採用するアーキテクチャスタイル

**Spring Modulith による feature-by-package + 各 feature 内部でクリーンアーキテクチャ4層** のハイブリッド方式を採用する。

| 層(外側) | 形態 | 強制手段 |
|---|---|---|
| モジュール境界 | Spring Modulith による feature 単位パッケージ(`xyz.dgz48.tasks.webapi.<feature>`) | `ApplicationModules.verify()` を `ModularityTests` で実行(CI 必須) |
| 内部構造 | 各 feature パッケージ配下に Domain / UseCase / Adapter / Infra の4層を切る | Code Review(ArchUnit 採用は ADR-0002 で当面保留) |

#### 理由

- 既存 scaffold が Spring Modulith を採用しており、`ModularityTests.verifyModularity()` で feature 間の不正参照を機械的に検知できる。
- 基本設計書 §2.1.1 のクリーンアーキ4層は**依存方向の規律**として有効であり、feature 内部の組織化に転用すると保守性が高い。
- フラットな4層パッケージ(`adapter.web` / `adapter.persistence` …) を最上位に置くと、機能横断のクラスが肥大化しがち。Spring Modulith ならテナント機能・タスク機能・ユーザー機能を**機能単位で独立にデプロイ可能なモジュール**として切り出せる。

> 新 feature を追加する際の具体的な手順・パッケージ構造・チェックリストは **[docs/dev/feature-template.md](../dev/feature-template.md)** を参照のこと。

### 1.2 パッケージ規約

#### 1.2.1 ルートパッケージ

```text
xyz.dgz48.tasks.webapi
```

> 基本設計書 §2.1.1 の `com.example.task.*` は例示値であるため、本規約のルートを正とする。設計書側の表記齟齬は別 Issue で追従修正する。

#### 1.2.2 feature パッケージ

```text
xyz.dgz48.tasks.webapi
├── security      ← 認証/認可(JWT 変換、Principal、SecurityConfig)
├── user          ← ユーザー、user_tenants
├── tenant        ← テナント、TenantContext
├── task          ← タスク本体(Task / Stakeholder / 認可)
├── notification  ← メール通知(SES + 通知設定)
├── audit         ← 監査ログ(ハッシュチェーン)
├── dashboard     ← ダッシュボード集計
├── shared        ← どの feature にも属さない共通(例外基底クラス等)
└── TasksWebapiApplication.java
```

- 1 feature = 1 トップレベルパッケージ。**他 feature の `internal` 配下を参照してはならない**(Spring Modulith が検知)。
- feature 間連携は **イベント(Spring Modulith の `@ApplicationModuleListener`)** または **`@NamedInterface` で公開した型** のみで行う。直接 Service を呼んで Cross-feature 依存を作らない。

#### 1.2.3 feature 内部の4層

各 feature パッケージは、内部で以下の4階層を切る。**外側から内側への依存のみ可、逆方向は禁止**。

```text
xyz.dgz48.tasks.webapi.task
├── domain        ← エンティティ、値オブジェクト、ドメインサービス、リポジトリ I/F (POJO、Spring 非依存)
├── usecase       ← ユースケース実装、入出力 Port、トランザクション境界 (@Transactional はここ)
├── adapter
│   ├── web       ← REST Controller、DTO、Bean Validation アノテーション
│   ├── persistence ← JPA Entity、Spring Data JPA Repository 実装、Hibernate Filter
│   └── external  ← 外部連携(SES、Keycloak Admin API 等)
└── infra         ← feature 固有の Spring 設定(必要時のみ)
```

- **依存方向**: `infra` → `adapter.*` → `usecase` → `domain`
- **JPA アノテーションは `adapter.persistence` の Entity のみ**(`domain` には書かない)。`domain` は POJO に保つ。
- **JPA Entity の単純クラス名は `~JpaEntity` で終える**(例:domain の `Task`(POJO) と adapter.persistence の `TaskJpaEntity` を区別する)。詳細はコーディング規約 §4.0 を参照。
- **DTO ↔ Domain の変換は `adapter` 層の Mapper** で行う(MapStruct 採否は ADR で決定)。
- **`@Transactional` は `usecase` 層のみ** に付与する(Controller やリポジトリ実装に書かない)。

#### 1.2.4 `package-info.java`

すべてのパッケージに `package-info.java` を置き、`@NullMarked`(JSpecify)を付与する(コーディング規約 §2.1 参照)。

### 1.3 Spring Modulith に関する追加規約

- `ApplicationModules.of(TasksWebapiApplication.class).verify()` は CI で必ず通す(既設定の `ModularityTests` を削除/無効化禁止)。
- 新規 feature を追加した場合、`ModularityTests` のみで検証が不足するなら、`@ApplicationModule(allowedDependencies = ...)` を `package-info.java` に明示する。
- 公開 API 型は `<feature>.api` パッケージ または `@NamedInterface` で明示。**`internal` パッケージ配下は他 feature から参照不可**。

#### 1.3.1 `shared` パッケージの扱い(Open module)

`shared` は「どの feature にも属さない共通型」(例外基底クラス、共通定数、汎用 Value Object 等)を置く特別な feature であり、**全 feature から自由に参照されるため Spring Modulith の Open module として宣言する**(決定根拠: [ADR-0003](../adr/0003-shared-package-as-open-module.md))。

```java
// xyz/dgz48/tasks/webapi/shared/package-info.java
@org.springframework.modulith.ApplicationModule(type = org.springframework.modulith.ApplicationModule.Type.OPEN)
@org.jspecify.annotations.NullMarked
package xyz.dgz48.tasks.webapi.shared;
```

- Open module 宣言により、`ApplicationModules.verify()` は他モジュールから `shared` の任意のクラスへの依存を許可する。
- `shared` には **Spring Bean / Service / Repository / Entity を置かない**(POJO・例外・定数のみ)。状態を持たせると依存方向が壊れる。
- 業務ロジックを `shared` に逃がさない。「どの feature にも属さないが業務的意味を持つ概念」が出現した場合、新規 feature の起票を検討し、規約改定 ADR を起票する。

## 2. API設計規約

### 2.1 OpenAPI ファースト

**`api/openapi.yaml` を API 仕様の唯一の正本(SSoT, Single Source of Truth)** とする。

- Controller / DTO / テストと OpenAPI の間に乖離が見つかった場合、**OpenAPI を先に修正**してから実装を追従する。
- スキーマ変更 PR では、OpenAPI / DDL / 基本設計書の 3 点を**同時に**更新する(レビュー報告書 G-2「3点一貫性」)。3 点のいずれかに変更が含まれない PR はマージ前に差し戻し。
- 自動整合性チェックスクリプト(レビュー報告書 §検証方法)は CI に組み込む(別 Issue 化)。

### 2.2 URL / メソッド規約

- 基本パスは `/api/`。バージョンは URL に含めず、後方互換を破る変更は新エンドポイントで提供する。
- リソース名は複数形・ケバブケース(`/api/tasks`、`/api/task-stakeholders`)。
- 部分更新は `PATCH`、完全置換は `PUT`、作成は `POST`、削除は `DELETE`。
- ID は path、フィルタ・ページングは query、本体は JSON request body。

### 2.3 認可違反時の HTTP ステータス(基本設計書 §6.2.3 より)

**強制ルール**。実装者の独自判断による逸脱は禁止。

| 違反種別 | HTTP ステータス | 例外クラス |
|---|---|---|
| 未認証(JWT 不正・期限切れ) | **401** Unauthorized | Spring Security 既定 |
| 参照系(GET)で参照不可 | **404** Not Found | `TaskNotViewableException` 等 |
| 更新系(PUT / PATCH / DELETE)で権限不足 | **403** Forbidden | `TaskOwnershipException` 等 |
| テナント越境アクセス | **403** Forbidden | `TenantBoundaryViolationException` |
| ロールベース不可(SaaS Admin 専用 API 等) | **403** Forbidden | Spring Security `AccessDeniedException` |

- 「参照不可なら 404」**の根拠は NIST AC-4(情報フロー保護)**。リソースの存在自体を漏らさない。
- 「編集権限なし」と「リソース未存在」を**実装上で取り違えないため**、上記の `*NotViewable` / `*Ownership` の例外クラスを分離する(コーディング規約 §6 参照)。
- 認可違反は**必ず `audit_logs` に `*_DENIED` で記録**する(設計書 §6.2.3 実装方針)。

### 2.4 エラー応答スキーマ(基本設計書 §5.4 より)

全 API 共通で以下の 6 フィールドを返す。

```json
{
  "timestamp": "2026-05-15T10:00:00+09:00",
  "status":    400,
  "error":     "Bad Request",
  "code":      "E_VALIDATION",
  "message":   "タイトルは必須です",
  "path":      "/api/tasks"
}
```

- **`timestamp` は RFC 3339 形式の JST(オフセット `+09:00`)で返す**。JVM・CI・DB・API レスポンス全層を `Asia/Tokyo` で揃え、UTC への正規化(`Z` サフィックス)は禁止する(日本圏前提、Issue #265 で確定)。OpenAPI 上は `format: date-time` で表現する(クライアントは必要に応じてパース時に各自のロケールへ変換する想定)。Java 側では `OffsetDateTime` を `ZoneOffset.of("+09:00")` または `ZoneId.of("Asia/Tokyo")` に揃えてシリアライズし、`Instant`(UTC 軸)を JSON へ直接シリアライズする用途は避ける。Jackson は `spring.jackson.time-zone: Asia/Tokyo` と `WRITE_DATES_AS_TIMESTAMPS=false` 前提(コーディング規約 §12.1 と整合)。
- `code` の定義値は以下のとおり(基本設計書 §5.4 と完全一致、本書はそのミラー)。新規 `code` の追加は**基本設計書 §5.4 を先に改訂**してから本書を追従する。

| HTTP ステータス | `code` | 意味 |
|---|---|---|
| 400 | `E_VALIDATION` | 入力検証エラー |
| 401 | `E_UNAUTHORIZED` | 未認証 / トークン無効 |
| 403 | `E_FORBIDDEN` | 権限なし / 他テナント参照 |
| 404 | `E_NOT_FOUND` | リソース未存在 |
| 409 | `E_CONFLICT` | 一意制約違反など(詳細は基本設計書 §5.4.1) |
| 422 | `E_UNPROCESSABLE` | 処理不能エンティティ(詳細は基本設計書 §5.4.2) |
| 500 | `E_INTERNAL` | サーバ内部エラー |

- 戻り型は **独自 `record ErrorResponse(...)`** を採用し、`code` は `ErrorCode` enum で型安全に表現する([ADR-0011](../adr/0011-error-response-record.md))。Spring Boot 4+ の `ProblemDetail` を継承する案、`ProblemDetail.properties` Map に `code` を載せる案は ADR-0011 §2 で検討の上、不採用。実 class は `xyz.dgz48.tasks.webapi.shared.web.ErrorResponse` / `xyz.dgz48.tasks.webapi.shared.web.ErrorCode`(`shared` Open module、[ADR-0003](../adr/0003-shared-package-as-open-module.md))に配置する。
- 各フィールドの生成元: `timestamp` は `OffsetDateTime.now(ZoneId.of("Asia/Tokyo"))`、`status` は HTTP ステータスコード(整数)、**`error` は `org.springframework.http.HttpStatus#getReasonPhrase()`**(例: `"Bad Request"` / `"Not Found"` — 独自文字列は使わない。クライアントは `code` で意味分岐、`error` は人間可読のヒント)、`code` は `ErrorCode` enum、`message` は日本語(コーディング規約 §13)、`path` は `HttpServletRequest#getRequestURI()`(query string を含めない)。

### 2.5 enum 値の 3 点一貫性

`Visibility` / `Priority` / `Role` / `TaskStatus` 等の enum は **設計書 / OpenAPI / DDL の 3 か所で文字列一致**させる(レビュー報告書 G-2)。

- 内部値(`NOT_STARTED` 等)と UI 表示値(`未着手` 等)の対応は、**フロントエンドの i18n リソース** で一元管理する。バックエンドは内部値のみを露出する(レビュー指摘 m-1 / m-2)。
- enum 追加 / 値変更時は、**設計書(基本設計書)を最初に改訂**してから、OpenAPI / DDL / 実装(Java enum)を追従させる。具体的には:
  - **同一 PR にまとめる**ことを**原則**とする(設計書 + OpenAPI + DDL + Java enum の 4 点同時更新)。差分のない 3 点一貫性を 1 つの PR で保証できるため、レビューが容易。
  - 4 点同時更新が PR サイズの都合で大きすぎる場合は**例外として分割可**だが、その場合も **設計書を改訂する PR が必ず先に main にマージされる**順序を守る(設計書が SSoT のため、後追い改訂は SSoT を壊す)。
  - **禁止**: 設計書未改訂のまま OpenAPI / DDL / Java enum を先行変更する PR(SSoT 違反)。レビュアーはこの順序違反を発見次第、設計書改訂 PR を先行させるよう差し戻す。

### 2.6 レスポンスコード列挙順序

**強制ルール**。`api/openapi.yaml` の各エンドポイントにおける `responses` キーの列挙順を以下のとおり統一する。

```text
2xx(成功) → 4xx(クライアントエラー昇順) → 5xx(サーバエラー)
```

具体的な昇順:

| 順位 | ステータスコード | 意味 |
|---|---|---|
| 1 | `2xx`(200 / 201 / 204) | 成功 |
| 2 | `400` | 入力検証エラー |
| 3 | `401` | 未認証 |
| 4 | `403` | 権限不足 |
| 5 | `404` | リソース未存在 / 参照権限不足 |
| 6 | `409` | 競合 |
| 7 | `422` | 処理不能エンティティ |
| 8 | `5xx`(500 等) | サーバエラー |

- エンドポイントが定義するコードのみを列挙すればよい(全コードを必ず網羅する必要はない)。
- 新規エンドポイントを追加する場合、および既存エンドポイントのレスポンス定義を変更する場合は、上記順序に準拠すること。
- PR レビューで「並び順」を理由とした指摘が発生した場合は、本ルールを根拠として修正する。

## 3. データベース設計規約

### 3.1 Flyway マイグレーション

- ファイル名: **`V{semver}_{seq}__{snake_case}.sql`**(例: `V1.3.0_02__add_task_stakeholders.sql`)。
- リリース版(`{semver}`)と 2 桁 Seq(`{seq}`)で**追記順を強制**する。
- 既存マイグレーションファイルの**事後編集は禁止**(チェックサム不整合)。修正は新規 `V*` ファイルで行う。
- `R__` リピータブルは使わない(必要になった時点で ADR)。
- ShedLock 用テーブル等の運用マイグレーションは `V99__*.sql` 等で分離してもよい(レビュー指摘 M-1)。

#### 初期版直接追記の例外運用(dev 初回デプロイ前、Sprint 1 完了まで)

dev 環境への tasks-webapi 初回デプロイ前(Flyway 履歴テーブル `flyway_schema_history` がどの稼働環境にも存在しない期間)の **Sprint 1 完了(実績: 2026-06-07)まで** は、上記「事後編集は禁止」の **期間限定例外** として、**初期版 `V1.0.0_NN__*.sql` への直接追記を許容**する。

- **対象期間**: 〜 2026-06-07(Sprint 1 完了日、実績)。Sprint 2 開始(2026-06-08)以降は本例外の対象外、必ず新規 `V1.0.x` ファイル。
- **根拠(dev タイムラインに基づく)**:
  - **Sprint 0(W3-W4)**: Terraform 基盤(VPC / SG / ALB / Parameter Store / Route53 PHZ)のみ構築、**dev RDS まだ無し** → `flyway_schema_history` は存在しない。
  - **Sprint 1(W5-W6、S1Infra-1)**: dev RDS MySQL 8.4 構築。ただし tasks-webapi はまだデプロイされず、**初回 Flyway migration 未実行 → `flyway_schema_history` 未生成**。
  - **Sprint 2(W2-W3、S2Infra-2)**: tasks-webapi 初回 ECS デプロイ → **初回 Flyway migration 実行 → `flyway_schema_history` が dev RDS に生成**。
  - Sprint 1 完了時点までは「どの稼働環境にも履歴テーブルが存在しない」状態が継続するため、初期版を直接編集してもチェックサム不整合が実害に至らない。新規 `V1.0.1`〜`V1.0.x` を量産するとファイル数が直線的に膨らむ。
- **Issue 起票時のチェック**: Sprint 1 完了までに schema 変更を伴う Issue を起票する場合、本ルールに従い「`V1.0.0_NN__*.sql` への追記」と明記する。「`V1.0.x__新規...sql` 追加」前提で起票しない(N1 #83 / N2 #84 が実装フェーズで V1.0.0 統合へ流れ Issue 本文と実装が乖離した事例 — 2026-05-24 N1〜N8 見直しで発見 — の再発防止)。
- **Sprint 2 開始以降**: Sprint 2 (W2-W3) で tasks-webapi 初回デプロイが走るため、以降のスキーマ変更は通常通り新規 `V1.0.x` ファイルで行う(本例外の対象外)。`flyway_schema_history` が dev で生成された後に V1.0.0_NN を編集するとチェックサム不整合で起動失敗する。
- **本例外セクションの撤去**: Sprint 2 開始時(2026-06-08)に本セクションを撤去し、通常の「事後編集禁止」ルールに一本化する。

#### Schema 変更と JPA Entity の同 PR ルール

スキーマ変更を伴う実装は、**Flyway migration の DDL と JPA Entity / Domain クラスの修正を 1 PR で同時 commit** する。

- **根拠**: `spring.jpa.hibernate.ddl-auto: validate` 前提のため、片方だけ merge された中間状態でアプリ起動・CI が壊れる(schema 先行 → Entity が古いまま validate 失敗、Entity 先行 → schema が無く validate 失敗)。
- **Issue 起票時もまとめる**: Schema 用 Issue / Entity 用 Issue の分離は禁止。1 Issue にまとめ、PR も 1 本。N1 (#83) と N3 (#85) を別 Issue として起票したことが構造的に同時マージを保証できなかった事例(2026-05-24 N1〜N8 見直しで発見)。
- **本ルールの恒久適用**: 上記「初期版直接追記の例外(Sprint 1 完了まで)」と異なり、本ルールは MVP リリース後も継続適用する。

### 3.2 文字コード / 照合順序

- `ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci` を**全テーブルで統一**。
- 例外を作る場合は ADR で根拠を残す。

### 3.3 マルチテナント

- **全業務テーブルに `tenant_id BIGINT NOT NULL` 列を必須化**(基本設計書 §6.3)。
- `tenant_id` を**第1キーに含む複合インデックス**を必ず作成する(例: `INDEX idx_tasks_tenant_status (tenant_id, status)`)。
- アプリ層では Hibernate Filter(`@Filter` / `@FilterDef`)で**全 SELECT** に `tenant_id` 条件を自動付与する(エンティティ取得時 / 関連ナビゲーション時)。本方式は [ADR-0010](../adr/0010-tenant-isolation-hibernate-filter.md) で確定(Spring AOP 併用案は GraalVM Native Image との相性および関連ナビゲーション非対応のため不採用)。
- **Hibernate Filter は `@Modifying` JPQL の UPDATE / DELETE には適用されない**ため、書き込み系のリポジトリメソッドは次のいずれかで明示的に `tenant_id` を絞り込む:
  - Spring Data JPA の派生クエリ(例: `deleteByTenantIdAndId(Long tenantId, Long id)`)
  - 明示的な `@Modifying @Query("UPDATE ... WHERE tenant_id = :tenantId AND id = :id ...")` JPQL
  - Repository を経由せず `EntityManager#find` → 取得した管理エンティティを変更し flush に任せる(この場合は Filter が効いた select 経路を必ず通る)
- **「テナント無視で全件検索」する Repository メソッドを書かない**。バッチや管理者用途で必要になったら、明示的に `@FilterDef` を一時無効化するヘルパー経由でのみ実施し、レビュー必須。
- `X-Tenant-Id` ヘッダ未指定かつ認証済みリクエスト(免除パス `/api/auth/**`, `/api/tenants/**`, `/actuator/**` を除く)は `UserTenantsResolverService` が `joined_at ASC` で最初の ACTIVE メンバーシップを自動選択して `TenantContext` を設定する。0 件の場合は 403 を返す([ADR-0016](../adr/0016-initial-tenant-selection-policy.md))。
- **クエリ実装の優先順位**: 次の順で選択し、上位で表現できる場合は下位を使わない:
  1. **メソッド名導出クエリ**(`findBy...` / `countBy...` / `existsBy...` 等)— Hibernate Filter が効く・起動時クエリ検証あり・`@Query` 文字列の保守不要
  2. **明示クエリ(`@Query`, JPQL)**— 導出で表現できない場合のみ。該当例: 単一列スカラ射影 / GROUP BY・複雑集計 / 単一文 bulk DML(`@Modifying`。派生 `deleteBy...` は select-then-delete のため bulk 等価でない)
  3. **native query**— JPQL でも無理・著しく不自然なときのみ。許容カテゴリは (1) Spring Modulith のモジュール境界を越えて他モジュールのテーブルを参照する集計、(2) DB 固有関数・JPQL の表現力を超える複雑クエリ ―― に限り、`// native 理由: …` を必ず付与する
- **native / bulk DML（`@Modifying`）は Hibernate Filter が効かない**ため、tenant スコープのテーブルに対するこれらのクエリは `tenant_id` を明示絞り込みし、`// tenant_id 明示: bulk DML は Filter 不適用(ADR-0010 §3)` を併記する。
- **Hibernate Filter に依存した派生クエリを Filter 無効文脈から呼ぶとクロステナント漏洩しうる**。`TenantFilterBypassService.runAsSaaSAdmin()` は Filter を無効化するため、そのブロック内からテナントスコープの派生クエリを呼んではならない。現状 `runAsSaaSAdmin` の全呼び出し元は `tenant` feature 内に閉じておりタスク系ユースケースに触れていないが、将来 SaaS Admin 文脈でタスク系ユースケースを呼ぶ場合は注意すること。
- **クエリ射影の既定方針**: 既定はエンティティ取得 + adapter でのマッピング。専用射影(scalar / DTO)は (a) エンティティが持たない他テーブル列が必要、(b) 高頻度・横長テーブルで over-fetch が実測で効く、のいずれかに限る。同一テーブルの列 subset 違いでメソッドを増やさない。

#### 3.3.x TenantFilterBypassService — Filter 一時無効化のガイドライン

テナントフィルタを一時無効化する唯一の公式経路は `xyz.dgz48.tasks.webapi.shared.usecase.TenantFilterBypassService` の `runAsSaaSAdmin(Supplier<T>)` メソッドである。

| 項目 | 内容 |
|---|---|
| **呼び出し API** | `TenantFilterBypassService#runAsSaaSAdmin(Supplier<T> action)` |
| **前提条件** | 呼び出し元でトランザクションが開始済みであること。`@Transactional(propagation = Propagation.MANDATORY)` により強制し、トランザクションなし呼び出しは `IllegalTransactionStateException` でフェイルファスト |
| **権限要件** | SaaS Admin ロール(`ROLE_APP_ADMIN`)必須。非 Admin が呼び出した場合は `SaasAdminRequiredException`(HTTP 403)を送出 |
| **スコープ** | `Supplier` の実行スコープ内のみフィルタを無効化。`try-finally` で構造的にフィルタを復元するため、スコープ外への漏洩はない |
| **禁止事項** | `session.disableFilter("tenantFilter")` の直接呼び出しは禁止。必ず本サービス経由でのみ実施すること |
| **レビュー** | 本サービスを利用する箇所はコードレビューで必ずレビュアーが確認する |

#### 3.3.1 Hibernate Filter 除外テーブル

**付与基準**: `TenantFilteredEntity`(共通 `@MappedSuperclass`)を継承するのは「`tenant_id BIGINT NOT NULL` 列を持ち、テナント境界で分離すべき業務エンティティ」のみとする。新規 JPA エンティティを追加する際は、下記除外一覧に該当しない限り必ず `TenantFilteredEntity` を継承すること。

以下のテーブルは Hibernate Filter の対象外とし、対応する JPA エンティティは `TenantFilteredEntity` を継承しない。

| テーブル | 除外理由 |
|---|---|
| `tenants` | マスタテーブル。`tenant_id` 列を持たない(このテーブル自体がテナント境界) |
| `users` | プラットフォーム横断ユーザー。`tenant_id` 列を持たない |
| `user_tenants` | テナント帰属解決テーブル。`TenantContext` 確立前(`TenantContextFilter` 内)にクロステナント参照が必要であり、Hibernate Filter を適用するとデッドロック的矛盾が生じる |
| `audit_logs` | `tenant_id` が nullable(システム横断イベントは `NULL`)。テナント範囲を超えた参照が必要 |
| `shedlock` | バッチ分散ロックテーブル。`tenant_id` 列を持たない。フレームワーク管理 |
| `app_admin_users` | SaaS Admin ユーザー管理。Keycloak が保持し、テナント境界の外側 |

将来追加される業務テーブル(`task_stakeholders` / `user_notification_settings` 等)は `tenant_id BIGINT NOT NULL` を持つため、対応 JPA エンティティは `TenantFilteredEntity` を継承する。

静的検証: `HibernateFilterEntityAuditTest`(test クラス)が `EntityManagerFactory` のメタモデルを走査し、全 JPA エンティティが上記付与基準を満たすことを CI で確認する([ADR-0010](../adr/0010-tenant-isolation-hibernate-filter.md) §6.1)。

### 3.4 主キー / 外部キー

- 業務テーブルの主キーは `id BIGINT AUTO_INCREMENT`。UUID は使わない(検索性能・index サイズの観点)。
- **すべての FK に `ON DELETE` を明示**(`CASCADE` / `RESTRICT` / `SET NULL` のいずれか)。MySQL デフォルト依存は禁止(レビュー指摘 n-1)。
- 親子関係が明確なもの(例: `task_stakeholders.task_id`)は `CASCADE`、ユーザー参照系(例: `tasks.owner_id`)は `RESTRICT` を原則とする。

### 3.5 監査列

- 業務テーブルには原則として以下を持つ:
  - `created_at DATETIME NOT NULL`
  - `updated_at DATETIME NOT NULL`
  - `created_by BIGINT NOT NULL`(`users.id` への FK)
  - `updated_by BIGINT NOT NULL`(同上)
- 例外(参照テーブル等)は ADR で記録する。
- **`tenants` テーブルは例外**として `created_by` / `updated_by` を**持たない**(2026-05-16 SaaS Admin スコープ整理ブレストの確定方針 / 根拠は [ADR-0004](../adr/0004-tenants-as-audit-column-exception.md))。理由:
  - SaaS Admin はマルチテナント不変の対象外である `tenants` テーブルの状態列(`status`)のみを操作対象とし、業務テーブルには触れない。
  - 「誰がいつテナントを作成・停止したか」は `audit_logs.actor_sub` で追跡可能であり、列として別途持つ必要がない。
  - 「テナントオーナー」を業務側で引きたいケースは `user_tenants` の現役 TENANT_ADMIN を参照することで代替できる(初代作成者を保持しても、所有権の実態とは乖離するため列を設けない)。

### 3.6 論理削除

- **物理削除を原則とする**。論理削除が必要な業務要件が出た場合は、対象テーブル単位に `deleted_at` 列追加 + ADR で承認。

## 4. 例外・ログ設計規約

### 4.1 例外階層

- アプリ内の業務例外は `xyz.dgz48.tasks.webapi.shared.exception.DomainException` を基底とする(`RuntimeException` のサブクラス)。
- 認可違反は §2.3 のとおり、**HTTP マッピング先ごとに別クラス**を作る(`TaskNotViewableException`/`TaskOwnershipException` 等)。
- `@RestControllerAdvice` で例外 → ErrorResponse へ集中マッピングする。Controller 内で `try-catch` して `ResponseEntity` を組み立てる実装は禁止。

### 4.2 ログ

出力スキーマ(JSON フィールド契約)・相関キー設計・event タクソノミ・環境別デフォルトの正本は [ログ設計](./ログ設計.md)(Issue #451 / ADR-0019)。

- 出力: SLF4J(Lombok `@Slf4j`)+ Spring Boot 標準構造化ログ(logstash 形式、ADR-0019)。`System.out.println` / `e.printStackTrace()` 禁止。
- レベル: `ERROR` は復旧操作が必要な状況、`WARN` は性能劣化や認可拒否、`INFO` はビジネスイベント、`DEBUG` は開発時の詳細。環境別デフォルトと一時変更手順はログ設計 §2 / §6。
- **MDC 項目**: `requestId`(全リクエスト)/ `userId`(認証後)/ `tenantId`(TenantContext 構築時のみ)/ `apiScope`(プラットフォーム API のみ)/ `batchId`(バッチのみ)。フィルタで自動セットする(実装は別 PR)。
- **PII(氏名・メール・電話番号等)はログに出力しない**(マスクして出すことも不可)。参照は内部 ID のみ。非意図的漏洩経路への対策を含む作法はコーディング規約 §7。
- 例外スタックは1回だけ出力する。再ラップ時は原例外を `cause` に保持。

### 4.3 監査ログ(基本設計書 §6.7)

- `audit_logs.hash_chain` は **前レコードの `id + detail + created_at` の SHA-256**。
- 認可違反、データ変更、ログイン/ログアウトは**必ず**記録する。
- ハッシュチェーンの日次整合性チェックバッチを停止しない。

## 5. セキュリティ設計規約

### 5.1 認証

- Keycloak(OIDC + PKCE)に**全面委譲**する。本システムにパスワード・秘密鍵を持たない。
- JWT 検証は Spring Security OAuth2 Resource Server。`TasksJwtAuthenticationConverter` が `Jwt.sub` を `users.oidc_sub` で突合し `TasksPrincipal` を作る。**この変換ロジックを Controller / Service で複製しない**。

### 5.2 認可

- ロール:
  - **SaaS Admin** … Keycloak realm role **`APP_ADMIN`**。本システム DB には保持しない(レビュー指摘 m-3)。
  - **Tenant Admin** / **Member** … 本システム DB の `user_tenants.role` に保持(`TENANT_ADMIN` / `MEMBER`)。
- 認可は**粒度に応じて層を分ける**:
  - **粗粒度の認可**(ロールの有無、認証済みかどうか、テナント所属の有無)は **`@PreAuthorize`** によるメソッドレベル宣言的認可で Controller の入口に付与する。
  - **業務ルールに踏み込んだ認可**(タスクの可視性 / 所有者・担当者・関係者の判定、Tenant Admin の強制編集等)は、feature の **`domain` 層に `<Feature>AuthorizationDomainService`(例: `TaskAuthorizationDomainService`)として SSOT に集約**する(`.claude/CLAUDE.md` の Authorization Model 節と整合)。
- 各層の役割分担:
  - `domain`: `AuthorizationDomainService` が「この主体はこのリソースに対して X 操作が可能か」を真偽で返す純粋関数。Spring 非依存。
  - `usecase`: 業務操作の入口で `AuthorizationDomainService` を呼び出し、不可なら例外を投げる(§2.3 のマッピングに従う `TaskNotViewableException` / `TaskOwnershipException` 等)。
  - `adapter.web`(Controller): `@PreAuthorize` で粗粒度の前提条件を確認し、本処理は `usecase` に委譲。**Controller での if 分岐による業務認可は禁止**。

### 5.3 入力検証

- すべての REST 入力に **Jakarta Bean Validation**(`@NotNull` / `@Size` / `@Email` 等)。
- Controller メソッドの引数に `@Valid` を**必ず**付与。
- 入力サイズ上限(VARCHAR(255) 等)は DTO と DDL で一致させる。

### 5.4 通信・データ保護

- HTTPS 必須(TLS 1.3 を優先、最低 1.2)、HSTS 有効。
- DB 接続は SSL/TLS を有効化。RDS は KMS で保存時暗号化。
- 機密情報の管理方針は [infrastructure-plan §3.4 / §3.5](../architecture/infrastructure-plan.md) に従う:
  - **tasks-webapi の RDS 接続**: IAM 認証(K-A 案、infrastructure-plan §3.5)。DB password は使用せず、`application.yml` の `password` は空文字注入(`${DATASOURCE_PASSWORD:}`)。
  - **Keycloak の SMTP password 等、IAM 認証で代替できない機密情報**: AWS Systems Manager **Parameter Store SecureString** で管理(infrastructure-plan §3.4、S0Infra-7、ADR-0006 派生)。
  - **AWS Secrets Manager は本プロジェクトでは未採用**(Parameter Store SecureString で同等のセキュリティを実現 + 運用コスト圧縮、2026-05-23 議論確定事項 #9、#250)。
  - いずれも `application.yml` への直書きは禁止。ECS Task Definition の `environment` / `secrets` で注入。

### 5.5 CSRF / CORS

- ステートレス REST のため CSRF は無効化(JWT 検証で代替)。
- CORS 許可 Origin はホワイトリスト管理。`*` を使わない。

## 6. ダッシュボード集計の認可スコープ(基本設計書 §6.2.2)

ダッシュボード(`GET /api/dashboard/summary`)は、§6.2.1 のタスク参照認可フィルタを**適用した後**のタスク集合に対して件数集計する。

- PRIVATE タスクは所有者のダッシュボードでのみ加算される。
- STAKEHOLDERS タスクは「所有者・担当者・関係者リスト登録ユーザー」にのみ加算される。
- TENANT タスクはテナント全員に同一件数で加算される。
- Tenant Admin にはテナント内全タスクが加算される。

**規約**: ダッシュボード集計を実装する PR では、「他者の PRIVATE が件数で漏れていないこと」を検証する**結合テスト**を必ず追加する。

## 7. バッチ設計規約

- スケジューラは Spring の `@Scheduled` + **ShedLock** によりインスタンス間排他制御する。
- ShedLock テーブル(`shedlock`)は運用マイグレーションとして `V99__shedlock.sql` 等で分離してよい(基本設計書 §4.2.8、レビュー指摘 M-1)。
- バッチも MDC に `batchId` を持たせ、ログから追跡可能にする。

## 8. ADR(Architecture Decision Record)

- アーキテクチャに関わる意思決定は ADR を起票する。配置は [`../adr/`](../adr/)、テンプレートは [`../adr/0000-template.md`](../adr/0000-template.md)。
- 起票が必要な変更の例:
  - 新ライブラリ・新フレームワークの導入(例: MapStruct、ArchUnit)
  - 既存設計の方針転換(例: Hibernate Filter → Spring AOP への切替)
  - 規約本文の変更(本書 / コーディング規約)
- 起票せず実装に進んでよいのは、**本書 / コーディング規約 / 基本設計書 のいずれにも反していない実装の細部**のみ。

## 10. ドキュメント管理規約

本セクションは `docs/specs/` 配下の設計ドキュメント(要件定義書・基本設計書・開発計画書・設計規約・コーディング規約)の管理ルールを定める。

### 10.1 ドキュメントヘッダー形式

各ドキュメントの冒頭は以下の形式で統一する。

- 日付は **`YYYY-MM-DD` 形式** に統一する(`YYYY年M月D日` 形式は使用しない)
- ヘッダーのバージョンと日付は、**常に改訂履歴の最新エントリと一致させる**

### 10.2 バージョン番号体系

| ドキュメント | 形式 | インクリメント基準 |
|---|---|---|
| 要件定義書 | `X.Y` | 要件・スコープの実質的変更で Y を上げる |
| 基本設計書 | `X.Y.Z` | 設計内容の変更で Z を上げる。広範な改訂は Y を上げ Z をリセット |
| 開発計画書 | `X.Y` | 計画内容の実質的変更で Y を上げる |
| 設計規約・コーディング規約 | `X.Y` | 規約の追加・変更・削除で Y を上げる |

- 誤字修正・体裁のみの変更はバージョンインクリメント不要
- 参照整合(他ドキュメントのバージョン表記を更新するだけ)は **最小インクリメント**(Z または Y の +1)とする

### 10.3 改訂日のルール

- 改訂日は **実際にコミットした日付(`git log` のコミット日)** を記載する
- 計画日・着手日・レビュー日は記載しない
- 複数コミットにまたがる場合は、**マージコミット日** を採用する

### 10.4 改訂履歴の記載義務

- 内容を変更した場合は、**同一 PR 内でバージョンをインクリメントし改訂履歴にエントリを追加する**
- 改訂内容欄には「何を・なぜ変えたか」を簡潔に記載する(例: `§1.2: 要件定義書参照を v1.0 から v1.3 へ更新(Issue #93)`)
- 改訂履歴テーブルの形式:

| 版数 | 改訂日 | 改訂内容 | 改訂者 |
|---|---|---|---|
| X.Y(.Z) | YYYY-MM-DD | 変更内容(Issue #N または PR #N を付記) | 開発チーム |
