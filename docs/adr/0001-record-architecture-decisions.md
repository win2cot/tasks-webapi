## 1. コンテキスト

tasks-webapi はマルチテナント SaaS として、Sprint 0(2026-07-14 予定)以降に本格実装に入る段階である。要件定義書 v1.2 / 基本設計書 v1.3.2 / 開発計画書 v1.2 / OpenAPI v1.3.1 / 詳細設計レビュー(2026-05-06) が揃い、これから多くの**アーキテクチャ的意思決定**(MapStruct 採否、Hibernate Filter vs Spring AOP、ProblemDetail vs 独自 ErrorResponse、ArchUnit 導入有無 など)を Sprint 0〜1 で連続して下す必要がある。

これらの意思決定を Slack や PR コメントで散発的に行うと、**「いつ・誰が・なぜ」が失われ**、新規参画メンバーや将来の自分が同じ議論を繰り返すリスクが高い。チームとして決定経緯を**1か所に追記蓄積**できる仕組みが必要である。

## 2. 検討した選択肢

### 選択肢 A: ADR(Architecture Decision Record)を導入する

- 概要: `docs/adr/` 配下に MADR ライクな軽量 Markdown(連番 + タイトル)で意思決定を記録する。
- 利点:
  - git 管理されレビューできる。
  - 過去の意思決定の文脈を1か所で追える。
  - PR レビュー時に「これは ADR が必要か?」という判断を機械的に下しやすい。
  - ツール不要(Markdown のみ)。
- 欠点:
  - 起票のオーバーヘッド。
  - 形骸化のリスク(全 PR で起票するわけではないため、運用ルールを設計規約で明文化する必要がある)。

### 選択肢 B: Confluence / Notion 等の外部ツールに残す

- 概要: 別途 wiki ツールで意思決定ログを管理する。
- 利点: 表・図など柔軟なレイアウト。
- 欠点:
  - **GitHub 一本化方針(基本設計書 / 開発計画書 v1.2)に反する**。Confluence/Jira は使わないと既に決定済み。
  - リポジトリと履歴が分離する。

### 選択肢 C: 何もしない(PR 本文・Slack で都度議論)

- 概要: 既存運用を継続。
- 利点: 何の追加運用も要らない。
- 欠点:
  - 過去の意思決定の根拠が霧散する。
  - 同じ議論を繰り返すコスト。
  - レビュー時に「これは新たな決定か、既存方針か」が判定不能。

## 3. 決定

**採用**: 選択肢 A(ADR を導入する)

具体的には:

- 配置: `docs/adr/`
- ファイル名: `NNNN-<kebab-case-title>.md`(`NNNN` は 4 桁連番、`0001` から)
- テンプレート: [`./0000-template.md`](./0000-template.md)
- 体裁: MADR(Markdown Architectural Decision Records)互換の軽量フォーマット。日本語で記述する。
- Status は `Proposed` → `Accepted` → 必要に応じて `Deprecated` / `Superseded by ADR-NNNN`。

## 4. 理由

- **GitHub 一本化方針との整合**: Markdown を git 上で管理することで、開発計画書 v1.2 が定める「Confluence / Jira 不使用」と矛盾しない。
- **設計規約と1対1で連動**: 設計規約 §8 で「ADR が必要な変更」を列挙してあり、規約改定そのものも ADR で行う運用にできる。
- **PR との親和性**: 大きめの意思決定を伴う PR では、ADR を**同 PR に含める**ことで、決定と実装が同時にレビューされる。
- **形骸化対策**: 起票が必要なケースを規約で明文化(設計規約 §8)し、レビュアーが PR で「ADR は?」と問える状態にする。
- **軽量**: 1 ADR は数百〜千字。Sprint 1 つあたり数本までを目安に運用する。

## 5. 影響

### 良い影響

- 「なぜそうしたのか」が**残る**(過去 PR コメントを掘り返さなくてよい)。
- 新規参画メンバーがオンボーディング時に `docs/adr/` を通読することで設計判断の体系を獲得できる。
- PR レビュー時の「これは決定済みか、新規論点か」が判別しやすい。

### 悪い影響・制約

- 起票・レビューの工数が増える(年に 10〜20 本程度を想定)。
- ADR 形式が陳腐化しないよう、運用を継続的に見直す必要がある(必要なら ADR-NNNN で再決定)。

### 既存ドキュメント・規約への波及

- 設計規約 §8 に ADR 導入を反映済み(本 ADR と同 PR で更新)。
- コーディング規約 §16 で「新規ライブラリ追加は ADR 必須」を明文化済み(本 ADR と同 PR で更新)。
