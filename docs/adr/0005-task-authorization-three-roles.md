## 1. コンテキスト

2026-05-16 の SaaS Admin スコープ整理ブレストを受けて、基本設計書 v1.4(PR #165)で認可モデルを「**プラットフォーム軸(`APP_ADMIN`)とテナント軸(`TENANT_ADMIN` / `MEMBER`)の 2 軸直交**」として再整理した(基本設計書 §6.2)。同日 ADR-0004 がその文脈で `tenants` テーブルを監査列例外として定義した。

しかし基本設計書 §6.2.1「タスクごとの認可ロジック」内には、Tenant Admin の業務タスク特権が残存していた:

- 参照: 「Tenant Admin は常に参照可」
- 編集: 「所有者のみ。ただし Tenant Admin は強制編集可」
- 削除: 「所有者または Tenant Admin」
- ステータス変更: 「所有者・担当者・Tenant Admin」
- 物理削除: 「Tenant Admin のみ」

この記述は次の点で 2 軸直交モデルと矛盾する:

1. **テナント軸の中で更にロール差をタスク認可に持ち込んでいる** — タスクは本来「所有者・担当者・関係者」というタスクスコープの自然な 3 役割で認可されるべきだが、そこに「Tenant Admin かどうか」が混入していた
2. **直交性の精神に反する** — テナント運営権限(招待・ロール管理・監査ログ参照)とタスク認可は別レイヤーであるべき
3. **プライバシー懸念** — Tenant Admin が他人の PRIVATE / STAKEHOLDERS タスクをいつでも覗ける状態は、NIST AC-4(情報フロー保護)の精神と緊張する

加えて PR #179(2026-05-17)のレビューサイクルで、§6.2.1 の「公開範囲変更・関係者編集: 所有者のみ」のみ Tenant Admin 例外が**明記されていない**ことが指摘された(#181)。これは:
- 設計書の記載漏れ(他の編集系には例外がある)
- 意図的な厳格化(公開範囲・関係者リストは所有者の専権事項)

のどちらか曖昧で、設計判断が必要だった。

本 ADR はこれらを根本から整理し、**タスク認可をタスクスコープの 3 役割のみで評価する**(Tenant Admin の業務タスク特権を撤廃する)決定を記録する。

## 2. 検討した選択肢

### 選択肢 A: タスク認可は所有者・担当者・関係者の 3 役割のみで評価、Tenant Admin の業務タスク特権を廃止

- 概要:
  - Tenant Admin は「テナント運営権限」(ユーザー招待・ロール管理・監査ログ参照)のみを持つ
  - タスクの参照・編集・削除・ステータス変更・公開範囲変更・関係者編集に対する Tenant Admin の特権を撤廃
  - Tenant Admin が自分が所有/担当/関係するタスクは、通常の 3 役割ロジックで操作可能
- 利点:
  - **2 軸直交モデルが厳密に成立**(ADR-0004 と整合)
  - タスク認可ロジックがシンプル(3 役割の関係のみで決定)、`TaskAuthorizationDomainService` の実装が容易
  - プライバシーが強化される(Tenant Admin であっても他人の PRIVATE / STAKEHOLDERS タスクは見られない)
- 欠点:
  - 退職ユーザーが所有していたタスクが「所有者不在」状態になる(所有者引継ぎ機構が別途必要)
  - Tenant Admin による「強制介入」の手段がない(運用上の柔軟性が下がる)

### 選択肢 B: 現状維持(Tenant Admin はタスクに対し強制編集・常時参照・削除権限を持つ)

- 概要: 既存の §6.2.1 ロジック(Tenant Admin 特権あり)をそのまま維持
- 利点:
  - 運営上の柔軟性(所有者不在タスクの救済・監査調査でタスク内容まで踏み込める)
- 欠点:
  - 2 軸直交モデル(ADR-0004)と矛盾し続ける
  - 認可ロジックが「3 役割 × 2 テナント軸ロール」で複雑化
  - プライバシー懸念
  - §6.2.1 の「公開範囲変更・関係者編集: 所有者のみ」記述の解釈問題(#181)が残る

### 選択肢 C: Tenant Admin は閲覧のみ可、編集系は不可(中間案)

- 概要: 全タスクの参照は可、編集系は所有者のみ
- 利点: プライバシーは妥協、運用柔軟性の一部は維持
- 欠点: 中途半端、結局 2 軸直交モデルに反する

## 3. 決定

**採用**: 選択肢 A(タスク認可は所有者・担当者・関係者の 3 役割のみで評価、Tenant Admin の業務タスク特権を廃止)

### 3.1 タスク認可ルール(基本設計書 §6.2.1 の新仕様)

| 操作 | 認可主体 |
|---|---|
| **参照(GET /api/tasks/{id})** | visibility に応じて分岐(下記) |
|   visibility = TENANT | テナント内全員(`TENANT_ADMIN` / `MEMBER` 区別なし) |
|   visibility = STAKEHOLDERS | 所有者・担当者・`task_stakeholders` に登録された者 |
|   visibility = PRIVATE | 所有者・担当者 |
| **編集(PUT /api/tasks/{id})** | 所有者のみ |
| **論理削除(DELETE /api/tasks/{id})** | 所有者のみ |
| **物理削除** | **MVP では API として提供しない**(後述 3.2) |
| **ステータス変更(PATCH /api/tasks/{id}/status)** | 所有者・担当者 |
| **公開範囲変更(PATCH /api/tasks/{id}/visibility)** | 所有者のみ |
| **関係者追加(POST /api/tasks/{id}/stakeholders)** | 所有者・担当者 |
| **関係者削除(DELETE /api/tasks/{id}/stakeholders/{userId})** | 所有者・担当者 |
| **関係者一覧参照(GET /api/tasks/{id}/stakeholders)** | タスク参照と同一の認可ルールを適用(タスクが参照可能なら、そのタスクの関係者リストも参照可能) |

### 3.2 物理削除(F-10)の扱い

- 現状の「物理削除は Tenant Admin のみ」は、Tenant Admin の業務タスク特権撤廃により再定義が必要
- **MVP では物理削除 API を提供しない**(論理削除のみで運用)
- 完全削除が必要なケース(GDPR / 顧客解約)は Phase 2 で **テナント解約(#167)に統合**して扱う

### 3.3 関係者指名と visibility の独立

- 関係者リスト(`task_stakeholders`)への登録は **visibility に関係なく可能**
- ただし「関係者であること」を根拠とする参照権限が発動するのは visibility = STAKEHOLDERS のときのみ:
  - **visibility = TENANT のとき**: 関係者であるか否かに関わらず、テナントメンバー全員として参照可
  - **visibility = STAKEHOLDERS のとき**: 所有者・担当者・関係者リスト登録ユーザーが参照可
  - **visibility = PRIVATE のとき**: 関係者リストに登録されていても、所有者・担当者でなければ参照不可(404 で拒否)
- PRIVATE / TENANT 時の関係者登録は「状態の事前準備」として機能する(将来 STAKEHOLDERS に変更した時点で、関係者として参照可になる)

### 3.4 通知ルール(F-18 期限当日通知のみが対象)

本 ADR の通知ルールは **F-18(期限当日通知)のみを対象**とする。アサイン通知・関係者追加通知・ステータス変更通知等の他通知イベントは MVP では未定義であり、必要に応じて後続 PR / Phase 2 で整理する。

F-18 期限当日通知の対象:

- **所有者・担当者**(通知対象)
- 関係者は通知対象**外**(常に通知しない)

### 3.5 ダッシュボード集計(基本設計書 §6.2.2 の新仕様)

- **S-03 個人視点ダッシュボード(`/api/dashboard/summary`)**: §6.2.1 のタスク参照認可フィルタを適用した後の集合で集計(Tenant Admin であっても通常ルールに従う)
  - PRIVATE → 所有者・担当者にのみ加算
  - STAKEHOLDERS → 所有者・担当者・関係者にのみ加算
  - TENANT → テナント全員に加算
- **新規: S-15 テナント運営者向けダッシュボード**(本 ADR では **MVP に含める方針** のみ決定、詳細仕様は本 ADR マージ後の要件定義書・基本設計書改訂 PR で確定する)
  - 認可: `hasRole('TENANT_ADMIN')`
  - スコープ: テナント内の **visibility ∈ {TENANT, STAKEHOLDERS} のタスクのみ**を集計対象とし、**PRIVATE タスクは件数も含めて集計対象外**(NIST AC-4 情報フロー保護の精神を S-03 から S-15 まで一貫させる)
  - 表示項目候補(詳細は後続 PR で確定): 集計可能なタスクの総数 / 当日期限・期限超過・本日完了件数 / ステータス別・優先度別件数 / 所属メンバー数
  - 「メンバー別稼働状況」等の個人特定可能な指標は MVP で含めない(過剰)

> **PRIVATE 除外の意図**: Tenant Admin であっても PRIVATE タスクの**存在**を集計値経由で推定できないようにする。S-03 の認可フィルタが「PRIVATE は所有者・担当者にのみ加算」としているのと同じ哲学を S-15 にも適用し、ダッシュボード集計レイヤー全体で NIST AC-4 を厳守する。Tenant Admin が把握できるのは「テナント全体で共有または関係者限定で動いている業務量」に限定される(運営者視点としては許容範囲との判断)。

### 3.6 Tenant Admin の責務(タスク無関係の権限は維持)

- ユーザー招待(F-05)
- ロール管理(F-06)
- テナント内監査ログ参照(F-19)
- **テナント運営者向けダッシュボード参照**(本 ADR で新設、3.5 参照)

これらは「テナント運営」のスコープであり、タスク認可とは独立。

## 4. 理由

- ADR-0004 で確立した「プラットフォーム軸 × テナント軸の 2 軸直交」モデルを、テナント内のロール(`TENANT_ADMIN` / `MEMBER`)とタスク認可の関係にも適用して**完全な直交化**を達成する
- タスク認可ロジックが「3 役割の関係のみ」に簡素化され、`TaskAuthorizationDomainService` の実装・テストが容易になる
- プライバシー懸念(Tenant Admin が他人の PRIVATE / STAKEHOLDERS タスクを覗ける)を解消
- 「業務タスクは所有者の専権事項、Tenant Admin はテナント運営権限のみ」という権限分離が明確になり、設計の予測可能性が上がる
- #181(§6.2.1 公開範囲・関係者編集の Tenant Admin 例外有無)の論点が、本 ADR により**根本から解消**(Tenant Admin 例外そのものを廃止するので、例外有無の議論が不要に)

## 5. 影響

### 良い影響(Positive)

- 認可ロジックが大幅に簡素化される(`TaskAuthorizationDomainService` のコードが小さく、テストパターンが減る)
- 2 軸直交モデル(ADR-0004)が厳密に適用され、認可モデルに一貫性が出る
- プライバシーが強化される(Tenant Admin であっても他人のタスク内容に触れない)
- §6.2.1 の「Tenant Admin 例外」の有無に関する曖昧さ(#181)が解消する

### 悪い影響・制約(Negative)

- **退職ユーザーが所有していたタスクが「所有者不在」状態になる運用リスク**を本 ADR の決定として明示的に受け入れる:
  - MVP では「所有者不在タスク」を運用リスクとして許容
  - 救済策(所有者引継ぎ API 等)の検討は **Phase 2 サブ Issue #187(#162 配下)で管理**する
  - 救済策の実現可能性は低い見込み(組織的には人事プロセス側で退職時のタスク移管・所有者付け替えを行う前提)。MVP リリース後の運用実態を見て #187 で要否を再判定する
- Tenant Admin が「問題タスク」を強制的に修正・削除できない(運用負担が所有者にコンタクトする方向にシフト)
- Tenant Admin であっても PRIVATE タスクの**件数**を S-15 で把握できない(NIST AC-4 一貫性のための意図的な妥協、3.5 参照)
- 物理削除 API が MVP に存在しない(論理削除のみ。完全削除は Phase 2 テナント解約 #167 に統合)

### Phase 2 検討項目(本 ADR 派生、#162 配下で管理)

- **所有者引継ぎ API(所有者不在タスクの救済)**: Issue **#187** で管理。退職ユーザーが所有していたタスクを Tenant Admin が引き継ぐ機構。実現可能性は低い(人事プロセス側で移管想定)が、運用リスクの管理として記録は残す。MVP リリース後の運用実態を踏まえて Phase 2 で要否を再判定
- **物理削除 API**: 既に #167(テナント解約)で扱う方針。本 ADR で「タスク単独の物理削除 API は MVP に含まない」を明示

### 実装上の注意: 関係者テーブルと参照権限の分離

`§3.3` で示したとおり、関係者リスト(`task_stakeholders`)への登録は visibility に関係なく可能だが、**関係者であることを根拠とした参照権限が発動するのは visibility = STAKEHOLDERS のときのみ**である。実装者が `TaskAuthorizationDomainService` を書く際の注意点:

- `task_stakeholders` への登録は権限付与ではなく「リストへの追加」に過ぎない
- 参照権限の評価は必ず `visibility` を先に確認し、`STAKEHOLDERS` のときに限り関係者リストを参照する
- `PRIVATE` の参照権限チェックでは `task_stakeholders` を **JOIN しない** よう注意(登録されていても権限は発動しない)
- `TENANT` の参照権限はテナントメンバーであれば全員可。関係者リストへの登録は無関係

### 既存テスト・実装への影響

- 現時点でタスク認可は実装未着手(Sprint 0 着手前)のため、コード変更の影響なし
- Sprint 0 で `TaskAuthorizationDomainService` を実装する際、本 ADR のロジックに従う
